# Validated lr-dev File-Quality Workflow Through Two PoC Iterations

The session began as a feature exploration of Claude Code's dynamic Workflow tool — what it does, how it visualizes, and whether the lr-dev quality concept brainstormed in earlier sessions was a good fit. After a small demo workflow confirmed the primitive, the user pulled the conversation toward the substantive work: implementing the §2 file-by-file quality analysis pipeline from the existing `draft-lr-dev-quality.md` as a runnable workflow script.

A first prototype landed quickly — five phases (Decompose / Analyze / Aggregate / Verify / Review) with §4 schemas enforced as JSON-Schema constraints on each `agent()` call. The user agreed to a real test run on a small target file. The first run produced 19 scenarios and 4 bugs from a single method, all schema-conformant, with a sensible cross-cutting note about shared root causes. Wall-clock ~7 minutes, ~770k tokens. Useful output, but several things smelled wrong on inspection: per-scenario intent-citation verify dropped zero scenarios across 19 fan-out agents (scope creep); bugs received no adversarial verification at all; and the analysts had no access to the codebase beyond the file under analysis — they verified citations against the scenario's own self-description.

The user corrected the design through a series of short, sharp messages: bugs need per-bug adversarial verification, not scenarios; gap analysis against existing tests needs to be added as an optional step alongside bug verification; each analyst, verifier, and gap analyzer needs to operate from a per-repo context agent's perspective. The first instinct was to use `/lr:attach` from inside the workflow subagents — the user immediately corrected that to `/lr:boot`, which is the right primitive for fresh-session subagents that have no host to attach into. The script was rewritten end-to-end on that pattern, with two parallel verify lanes (per-bug adversarial fan-out + single gap analyzer running concurrently) replacing the v1 per-scenario gate.

The v2 run was the standout moment of the session. A lore-aware verifier — booted as the per-repo context agent, given the actual caller graph — flipped four of nine bugs to false-positive after reading code the v1 verifier had no access to. One reversal in particular: a literal that v1 confidently flagged as a typo turned out to be a non-standard supplier-emitted code, with the lore-aware analyst tracing it through the supplier's own enum file and concluding the literal was correct as written; the bug was the *missing* canonical arm, not the existing one. This was the operational case that proved the lr-dev reframe — a no-context analyzer would have generated a "fix" that broke real integration silently. The 27-gap output from the gap analyzer cited prior known-issue topics by filename in its "why" field — the bidirectional IR working as designed.

Then the conversation widened. The user raised the storage problem (no run produced any persisted artifact) and walked through the right architecture for a whole-repo sweep system: a separate quality repo holding manifests, run snapshots, catalog files, and AI-generated tests; a per-repo context agent staying in the existing agent-repo conventions; and the source repo untouched — three repos, three distinct jobs. The host's first proposal mixed the context agent and quality repo into one; the user corrected that, and a clearer separation emerged. The user also clarified that generated tests should be a permanent safety net, not a stepping stone migrated into the source repo — drift between the AI suite and the human suite is itself information to preserve. Two reflection topics captured the architecture: `quality-repo-architecture.md` and the workflow-primitive notes alongside the v2 PoC reflection.

Reflect/merge/summarize ran with two guests attached late in the session. The user's explicit instruction at finalization time scoped the host's reflections to general framework lessons — workflow primitive, three-repo architecture, lr-dev shape — with all source-domain specifics routed to the guests' own lore. The guest merges integrated the bug cluster into existing known-issue topics, created a new generalized supplier-quirk pattern, and tied the prototype back to existing OKR candidates and onboarding material. Three lore topics created across active agents (two on the host, one on a guest), six topics updated, two workflow scripts and three drafts modified, all uncommitted pending phase 4.

Open threads: the workflow's persistence stage (markdown serialization to a quality repo) is not yet built; the manifest-driven whole-repo sweep skill is the next infrastructure piece; a triage skill to flip bugs from `unreviewed` to `confirmed` is missing. The repo carrying the guest agents has a pre-existing dirty version stamp from a deferred earlier upgrade — surfaced cleanly by the v15 write-aware boot gate, but still requires the user to commit or revert before auto-upgrade resumes. None of these block what was learned this session; they are queue items for the next one.
