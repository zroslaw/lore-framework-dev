# Built the ULA bug-verification track and shipped v16

The session opened with the user returning from a separate testbed session where they had run ULA's first working pass and asked Claude to triage the bugs it found in `CheckUpdatesHelper.m` (the My-Turbo-Boost-Switcher app). That triage had concluded every finding was a non-critical code smell or an intentional simplification. The user brought the result back and asked, in effect, what it meant for the ULA skill. The architect reframed it: an "all non-critical" verdict is not a finder failure but the signature of a *missing verification track* — ULA's finder is deliberately high-recall and defers confirmation. A side-correction landed early: Claude had described the bug-finding step as "clean-room / context-blind," which a read of the actual prompts disproved (clean-room is a Step B property about not peeking at tests; the bug step is explicitly context-aware). That reading-before-asserting reflex became its own lesson.

The bulk of the work designed and built that verification layer. The ULA unit pass grew from A→B→C to **A→B→C→D→E**: Step D re-investigates each Step A bug for real-ness and *real system impact* (past the clean-room constraint, free to read the whole repo), revising severity or moving non-bugs aside; Step E is a completeness guardrail. On top of the per-unit self-verification, an **independent aggregation-level re-verification** runs after persist — a fresh subagent re-applies the D/E checks across all units with whole-file context, since a unit agent grading its own work lacks independence. Bug findings gained a multi-axis schema along the way: `severity` (with a `negligible` tier for "real but harmless"), `confidence`, optional `category`, and — added later in the session — `impact-summary` (plain-language, any-engineer-readable) and `nature` (`product` vs `technical`). Cross-unit findings (`crossUnit[]`, external vs interaction) are captured rather than dropped, and rejected findings move to a preserved `dismissed[]` carrying their original fields plus `dismissal-reason` and a two-stage `dismissed-by` (`unit` / `aggregator`) so false-positive history stays analysable.

Several user corrections steered the design. The user rejected, sharply, a required structured `context` field that Claude had added to "enforce" the agent's context-tracing ("drop this shit"); it was reverted to prose-only enforcement, and the lesson — *"enforce X" does not mean "add a required schema field"; prefer prose unless the field is itself the deliverable* — was captured. The user also reframed an early request for a "category" field: what they actually wanted was *importance*, which became `severity`, and the multi-axis split followed from disentangling type from importance from confidence. A live workflow failure (a `JSON Parse error` from a hand-serialized `args` string with a missing colon) produced a durable guard: pass workflow `args` as a structured object, never a hand-built JSON string.

Shipping happened in two stages. First the accumulated work plus the previously-staged dev→df rename were committed and pushed as BETA with manifests untouched (`005f18a`), keeping marketplace users on 1.15.0. Then, at the user's request, the architect cut **v16**: VERSION 15→16, both plugin manifests → `1.16.0`, and a new `release-notes/16.md` (with the Clear Plugin Cache footer), pushed as `3ff811f`. That manifest bump is the cache-detection lever, so v16 is now visible to the marketplace — the first lr-dev / Dark Factory module to ship with real version visibility rather than `--plugin-dir`-only discoverability. The push to the public repo was held for explicit confirmation first, given it releases to all installs.

A plot twist arrived during finalization itself: the merge subagent hit a monthly spend limit mid-run. Verification confirmed it had persisted nothing to lore before failing (all seven reflections intact, no partial edits), so the merge was completed **inline** in the host session instead — three new lore topics (`ula-bug-verification-track`, `ula-finding-schema`, `feedback-schemas-as-enforcement-overreach`), updates to `aiqa-ula-feature` (now A→B→C→D→E, shipped status), `df-module-conventions` (args-as-object checklist item), `verify-before-acting-on-suspected-bugs` (the read-before-asserting instance), and `versioning-release-types` (v16 backfill), plus the `lore-context.md` index and a v16 restamp of `lore-repo.md`.

The session landed with v16 live on `origin/main` and the lore merged and about to be committed by this finalize. Open threads: the new verification path still wants a live ULA re-run on a *meatier* file that actually has tests, to exercise both the verification layer and the so-far-unstressed gap Direction B; a `both` value for `nature` remains a deferred option if dual-nature bugs recur in practice; and the `lore-agents` repo, still stamped at 15, will auto-reconcile to 16 on its next boot.
