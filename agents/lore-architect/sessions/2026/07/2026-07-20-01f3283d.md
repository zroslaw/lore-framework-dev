# Found and fixed a real Being Keeper kill-tree bug

The session opened with a request to boot as lore-architect and run another deep review pass on
the Lore Beings / Being Keeper (`lrb`) feature, then surface the three most valuable improvements.
Three parallel reviewers were forked — one over the design lore and its own self-flagged gaps, one
doing a line-by-line read of `scripts/lrb.py`, and one auditing the test suite — and their findings
were synthesized into a ranked top three: budget-enforcement leaks in the daily-USD spawn gate
(a midnight-boundary accounting bug plus two race conditions), an unattended full-permission
task-execution trust gap for team-shared beings, and a multi-engine coverage gap where claude had
six real-engine lifecycle scenarios to codex/cursor's one each, compounded by a confirmed
concurrency-slot-leak bug that specifically fires when PID identity can't be verified — precisely
the codex/cursor blind spot.

The user judged the first two findings acceptable to defer at the current proof-of-concept phase and
asked only for backlog entries, but wanted the third fixed or at least tested, plus two extra asks:
resolve the cursor cost-blindness finding and check a suspected-stale test file. The session
implemented all of it directly: two new backlog entries recording the deferred budget/trust
findings; `cmd_engines_add` now requires `--session-cost-usd` for `cursor` (mirroring codex, since
real cursor-agent responses omit `total_cost_usd`); a time-boxed reap for entries whose PID identity
stays permanently unverifiable, closing the concurrency-slot leak; deletion of the superseded
`test_lrb_cursor_real_e2e.py` with its references retargeted to the A3 lifecycle scenario; new
mocked tests for the previously-untested `lrb install --launchd` path; and new B2/B3 lifecycle
scenarios mirroring claude's B1 process-tree-kill scenario for codex and cursor.

Running those new scenarios for real against live engines (with the user's explicit go-ahead, since
real money and real background processes were involved) immediately turned up a genuine production
bug rather than just closing a coverage gap: cursor-agent's sandboxed shell tool runs commands
inside a freshly `setsid`'d session, so the Keeper's `killpg`-based kill never reached a
timed-out session's real grandchild process — confirmed live when the orphaned process chain was
still running on the machine and had to be killed by hand. The fix (`_kill` now also walks the full
ppid-descendant tree and signals every descendant directly) surfaced a second, subtler bug while its
own regression test was being built: descendants must be enumerated *before* the parent is signaled,
not after, or a fast-dying parent gets reaped and the OS reparents the survivor to PID 1, erasing the
very ppid link the walk depends on. The added latency from that fix's `ps` call then broke two
pre-existing unit tests that had implicitly relied on `_kill`'s prior near-instantaneous speed as a
timing guarantee — both were fixed by relaxing an over-specific same-tick assertion and adding an
explicit synchronization point where one was genuinely needed.

By the end, all three engines were re-verified passing for real on their kill-tree scenarios —
claude, the now-fixed cursor, and codex (whose first attempt had failed on an early, non-blocking
session completion, retried clean on the second attempt and was attributed to model behavior rather
than a Keeper defect, echoing the already-documented cursor backend flakiness). The full unit suite
(91 tests) ran green and stable across repeated runs, and no orphaned processes, tempdirs, or real
launchd jobs were left on the machine at any point. Three reflection topics were written and merged
into lore, generalizing the session's debugging lessons: the enumerate-before-signal ordering rule
for any future ancestry-based kill-tree code, a macOS-portable technique for testing a
`setsid`-escape scenario without the (Linux-only) `setsid` binary, and a diagnostic pattern for
telling apart "the new code is wrong" from "a hot-path speedup exposed a pre-existing test race."

The code and test changes remain uncommitted, since they sit in `lore-framework/scripts/`,
`lore-framework/docs/`, and `lore-framework-dev/tests/` — outside finalize's `agents/`-scoped commit.
The two deliberately deferred findings (budget-enforcement edge cases and the unattended
full-permission trust gap) stay open in `framework-improvements-backlog.md` for a future session.
