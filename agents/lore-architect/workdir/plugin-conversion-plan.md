# Lore Framework Plugin Conversion Plan

Convert `lore-framework/` from a standalone repo with manual setup into a Claude Code plugin distributed as `lr`.

## Context and Decisions

**Plugin name/prefix**: `lr` — commands become `/lr:boot`, `/lr:reflect`, etc.

**All operations are user-triggered commands** — no model-invoked skills. The framework deliberately requires explicit user action for all operations.

**Two ways to boot an agent**:
1. Generic: `/lr:boot <agent-name>` — always available, discovers agents by scanning the domain
2. Per-agent: `/lr-<agent-name>-agent` — optional, generated by `/lr:register-repo` into `.claude/commands/` for convenience and discoverability

**No ambient context for now** — framework context is loaded on-demand when skills are invoked. Can revisit later (SessionStart hook is the preferred mechanism if needed).

**`lore-framework/` no longer needs to exist in the domain directory** — the plugin is installed via the plugin system. The source repo remains on GitHub for development.

### 1.3 Decide on instruction placement

Each skill needs its operational instructions. Two approaches per skill:

**A) Self-contained** — the SKILL.md contains all instructions inline. Best for short skills (list-agents, list-repos, finalize).

**B) Reference-based** — the SKILL.md tells Claude to read a doc from `${CLAUDE_PLUGIN_ROOT}/docs/`. Best for complex skills with long instructions (check, reflect, merge, register-repo).

Decision per skill:

| Skill | Approach | Rationale |
|---|---|---|
| boot | Self-contained | Short: discover agent, read 3 files |
| reflect | Reference | Long instructions in process-reflection.md |
| merge | Reference | Long instructions in process-merge.md |
| finalize | Self-contained | Short: run reflect then merge |
| register-repo | Reference | Multi-step process in register-repo.md |
| unregister-repo | Self-contained | Short: scan and delete |
| create-repo | Reference | Multi-step scaffolding in create-repo.md |
| create-agent | Reference | Multi-step process in create-agent.md |
| list-agents | Self-contained | Short: scan and report |
| list-repos | Self-contained | Short: scan and report |
| check | Reference | Very long: 16-point audit |

## Phase 2: Write All Skills

### 2.1 `/lr:boot` — The new generic boot command

This is the most important new skill. It replaces per-agent commands with a single discoverable entry point.

```markdown
---
description: Load a lore agent by name. Usage: /lr:boot <agent-name>
---

Boot the agent named "$ARGUMENTS".

Steps:
1. Search all directories in the current working directory for agent repos
   (directories containing an `agents/` subdirectory).
2. Within each agent repo, look for `agents/$ARGUMENTS/` containing `role.md`.
3. If not found, list all available agents and report the error.
4. If found, read these three files in order:
   a. `${CLAUDE_PLUGIN_ROOT}/docs/agent-boot.md` — how to operate as a lore agent
   b. `<repo>/agents/<agent-name>/role.md` — the agent's role and identity
   c. `<repo>/agents/<agent-name>/lore-context.md` — the agent's working knowledge
5. Confirm you are loaded as the agent and briefly state your role and what you know.
```

Key change from current system: the boot command discovers agents dynamically instead of relying on pre-generated command files with hardcoded paths.

### 2.2 `/lr:reflect`

```markdown
---
description: Extract session knowledge into reflection topics. Run at end of session.
---

Read `${CLAUDE_PLUGIN_ROOT}/docs/process-reflection.md` for detailed instructions.

Review this session and extract knowledge worth preserving into reflection
topics in your agent's `reflections/` directory.

When done, confirm that reflection is complete and list the topics you created.
```

### 2.3 `/lr:merge`

```markdown
---
description: Integrate reflection topics into the agent's lore. Run after /lr:reflect.
---

Read `${CLAUDE_PLUGIN_ROOT}/docs/process-merge.md` for detailed instructions.

Integrate all reflection topics from your agent's `reflections/` directory
into your `lore/`, `lore-context.md`, and `role.md` as appropriate.

When done, confirm what was merged.
```

### 2.4 `/lr:finalize`

```markdown
---
description: Full session finalization — reflect then merge. Run at end of session.
---

Run the full finalization process for the current session.

**Step 1 — Reflect:**
Read `${CLAUDE_PLUGIN_ROOT}/docs/process-reflection.md` for instructions.
Review this session and extract knowledge worth preserving into reflection
topics in your agent's `reflections/` directory.

**Step 2 — Merge:**
Read `${CLAUDE_PLUGIN_ROOT}/docs/process-merge.md` for instructions.
Integrate all reflection topics into your `lore/`, `lore-context.md`,
and `role.md` as appropriate. Clean up `reflections/` and commit.

When done, confirm what was reflected and merged.
```

### 2.9 `/lr:list-agents`

```markdown
---
description: List all agents in this domain with their repo and purpose.
---

List all lore agents available in this domain.

**Method 1 — From registered commands (if any exist):**
Check `.claude/commands/` for `lr-*-agent.md` files. Each encodes an agent name
and contains the path to its role.md.

**Method 2 — From directory scan (always):**
Scan all directories in the working directory for agent repos (containing `agents/`).
Within each, find agent directories (containing `role.md`).

Combine both methods. For each agent found:
- Read `role.md` to extract a one-line purpose (first paragraph after heading).
- Note which repo it belongs to.
- Note whether it has a registered boot command.

Output a table: **Agent**, **Repo**, **Registered**, **Purpose**.
```

Note: This skill is updated from the current version. It now discovers agents both from registered commands AND from directory scanning, since registration is optional.

### 2.10 `/lr:list-repos`

```markdown
---
description: List all agent repos in this domain with their purpose.
---

List all lore agent repos in this domain.

Scan all directories in the working directory for agent repos — directories
containing an `agents/` subdirectory with at least one agent (subdirectory
containing `role.md`).

For each repo found:
- Count the agents it contains.
- Read `README.md` (if present) for a description.
- Check whether it's registered (has boot commands in `.claude/commands/`).

Output a table: **Repo**, **Agents**, **Registered**, **Purpose**.
```

### 2.11 `/lr:check`

```markdown
---

Work through all checks in order. Report each issue found.
At the end, print a summary: total issues found, or "All checks passed" if none.
```

This requires extracting the check logic from the current `lr-check.md` command file into a new `docs/check.md`. The checks themselves need updating for the plugin context (see Phase 2.12).

### 2.12 Update docs for plugin context

## Phase 4: Handle the Boot Command Path Problem

This is the trickiest part of the conversion. When `/lr:register-repo` generates per-agent boot commands in `.claude/commands/`, those commands need to reference `agent-boot.md` (the lore agent operating instructions). Currently they reference `lore-framework/docs/agent-boot.md`. In the plugin world, `lore-framework/` doesn't exist in the domain.

### Options:

**Option A — Embed the instructions**
The generated boot command embeds the essential operating instructions directly, instead of referencing a file. The boot command becomes self-contained but longer.

Pro: No path dependency. Works even if the plugin is uninstalled.
Con: Content duplication. If agent-boot.md is updated (plugin update), existing boot commands have stale instructions.

**Option B — Reference the plugin skill**
The generated boot command says: "First run /lr:boot to understand how lore agents operate, then load this specific agent by reading these files: ..."

Pro: Always uses current plugin instructions.
Con: Slightly awkward indirection. Depends on the plugin being installed.

**Option C — Reference a well-known path**
Establish a convention: the plugin places (or the user is told to place) a symlink or copy of agent-boot.md at a known location in the domain (e.g., `.lore/agent-boot.md`).

Pro: Stable path, works without plugin.
Con: Adds back a domain-level artifact. Defeats part of the plugin benefit.

**Option D — Inline a minimal version, reference plugin for full**
The generated boot command contains a short summary of agent operating principles (5-10 lines) plus "For full operating instructions, see the lr plugin's agent-boot.md." The essential behavior (read role.md, lore-context.md, search lore/) is embedded; the full details are in the plugin.

Pro: Works standalone with degraded but functional behavior. Best of both worlds.
Con: Still some duplication.

**Recommended: Option D.** The generated boot commands embed the essential operating rules (read the 3 boot files, search lore before assuming, finalize via /lr:finalize) and reference the plugin for the full version. This makes registered commands functional even without the plugin while keeping them concise.

## Open Questions

1. **`${CLAUDE_PLUGIN_ROOT}` in skill bodies** — Does this variable resolve correctly when used in skill instruction text (for paths the model should read), or only in MCP/hook configs? Need to verify. If not, skills may need to say "read the file `docs/agent-boot.md` from within the lr plugin directory" and rely on the model's ability to locate the plugin installation.

2. **Backward compatibility period** — Should we maintain dual support (old command-based + new plugin-based) for a transition period? Or clean cut? I lean toward clean cut since the user base is small.

3. **Per-agent commands namespace** — Currently generated as `/lr-<name>-agent` (flat in `.claude/commands/`). Should they stay as-is, or should registration generate them differently? The current format works — they're outside the plugin namespace and that's fine.

4. **The lore-architect self-hosting story** — Once converted, the lore-architect works on the plugin source repo (`lore-framework/`) using `claude --plugin-dir .` for testing. The development workflow needs documenting.

5. **create-agent interaction with registration** — Currently `create-agent` always generates a boot command. In the plugin world, should it still do that (maintaining the "registration is default" behavior), or should it only create the agent directory and let the user register separately? I lean toward: still generate the boot command by default, since that's the more helpful behavior.
