# Built the paste-link onboarding set and closed a Codex fallback bug

The session opened with a housekeeping task — registering direct boot shortcuts (`/lr-<agent>-agent`) for every lore agent across the workspace's two agent repos, `lore-agents` and `lore-framework-dev` — which doubled as a live boot of `lore-architect` itself once the shortcuts existed. From there the user raised the actual goal: Lore Framework had recently shipped Cursor and Codex support plus an enhanced Claude Code plugin, but had no onboarding documentation to match — nothing that would let a stranger install the framework and start using it well, and specifically nothing shaped for the idea of pasting a link straight into an AI coding agent and asking it to do the install.

The bulk of the session designed and wrote that missing documentation. After surveying the existing `README.md`, `INSTALL-CODEX.md`, and `INSTALL-CURSOR.md`, the work settled on five root-level files: `QUICKSTART.md`, a new genre of onboarding doc written *to the AI agent* as a literal installer (detect engine → fetch the matching per-engine page → execute, narrating and pausing before side effects → verify → hand off); `INSTALL-CLAUDE.md`, closing a parity gap since Claude Code's install had only ever lived inline in the README; matching agent-operator preambles added to the existing `INSTALL-CODEX.md`/`INSTALL-CURSOR.md`; and `FIRST-STEPS.md`, a guided walkthrough from creating an agent repo through boot, work, and finalize. `README.md` was substantially reworked into an actual landing page — a hook naming the "every session starts from zero" pain, a value-prop pull-quote, and a use-cases section spanning work, cross-team, and personal usage (grounded in the user's own personal-agent roster — tax, health, an RC-glider hobby, license admin — generalized and depersonalized), plus a self-referential "the framework is maintained by one of its own agents" example pointing at `lore-architect` in `lore-framework-dev`.

The user then asked for the whole set to be reviewed from three independent angles using parallel sonnet subagents, iterated until clean. Three lenses ran — a newcomer/adopter reading for hook and followability, an "AI-as-literal-installer" reading for execution fidelity, and an editorial pass for concision and terminology hygiene, the last enforced against the user's explicit "don't let this get bloated" concern. Round one surfaced four real issues: `FIRST-STEPS.md`'s walkthrough ended in an unexplained `git push` failure because `create-repo` never sets up a remote; the README's front matter had grown heavier than the maintainer wanted, with a term collision ("skills it picked up" vs. the framework's own `/lr:*` skills) and a meta example crowding the primary call to action; and `INSTALL-CURSOR.md` leaked internal-only jargon (version-gated probe references) at a first-time reader. All were fixed — the git-remote gap got an explicit "sharing is optional" step, the README front matter was trimmed and the meta example moved below "Getting started" as a secondary "go deeper" beat, and the Cursor jargon was stripped.

Round two, after fixes were applied, turned up one further high-severity finding — and only from the installer lens: `FIRST-STEPS.md` and `INSTALL-CODEX.md` had inherited an instruction telling a Codex agent to find a skill's procedure by guessing `docs/<skill>.md` from the skill name. That guess is wrong wherever the two names diverge — `boot` is actually `docs/agent-boot.md`, `merge` is `docs/process-merge.md`, `reflect` is `docs/process-reflection.md`, `register-agent` is `docs/register-repo.md` — and it would have broken a Codex agent at the walkthrough's central step, `/lr:boot`. Verifying against the actual files confirmed the bug, and traced it to its real source: `docs/engines/codex.md` itself carried the same wrong guess, even though the sibling Cursor profile had already named and guarded against exactly this class of error ("Name mapping is not mechanical"). The fix went in at the source and both downstream docs, correcting the instruction to read `skills/<skill>/SKILL.md` instead of guessing.

The session landed with the onboarding set finished and reviewed to convergence, plus a genuine framework bug fixed rather than just a documentation issue. `lore-framework`'s changes were committed locally (not yet pushed, per the user's request to keep working across one more session first) — deliberately excluding an unrelated pre-existing modification to `MARKETPLACE.md` (a runtime-release-commit-hash correction, verified as a legitimate but separate change) that was left untouched and flagged rather than swept in. `docs/engines/codex.md` is lifecycle-harness-covered, so the Codex lifecycle scenarios still need to run before this change is eventually pushed. On the lore-architect side, reflection produced four topics — naming the new paste-link installer doc genre, the skill-doc-filename divergence bug class, a fourth "AI-as-installer" review lens for the pre-ship toolkit, and a landing-page-specific refinement to meta-example placement — merged into `lore/` (three new topics, four existing ones extended with cross-references) and folded into `lore-context.md`.
