# Quality Benchmark Reporting

This is the field guide for quality benchmark reports generated by
[`run_matrix.py`](run_matrix.py). It explains the terms used in `summary.md`,
`release-notes.md`, and `summary.json`, and links back to the test cases that
produce those numbers.

The benchmark source files are:

- [`strategy.md`](strategy.md) — why the quality benchmark is designed this way and how to interpret it.
- [`probes.py`](probes.py) — the probe catalog: the actual test cases, planted lore facts, tasks, and scoring rubrics.
- [`test_quality.py`](test_quality.py) — executes one engine/model config and scores S1/S2/S3.
- [`run_matrix.py`](run_matrix.py) — runs many engine/model configs and writes report artifacts.
- [`harness.py`](harness.py) — builds fixtures, runs engines, captures traces/costs, and calls the S3 judge.

## What This Benchmark Measures

The quality benchmark checks whether lore changes an agent's behavior in useful,
expected ways.

Each probe creates a tiny fixture agent repository with lore topics. The same
probe is run twice:

- `treatment` — the important lore fact is present.
- `control` — the important lore fact is removed or neutralized, while filenames and repo shape stay comparable.

The benchmark then compares treatment vs control. A good result usually means
the treatment arm uses the lore fact and the control arm does not invent it.

## How To Read One Probe

Example: [`P2-gotcha-avoidance`](probes.py#L375).

That probe asks the agent to upload 500 records to the `petrel-records` API. In
the treatment arm, lore says the API silently drops batches larger than 25. The
scoring asks three questions:

- S1: did the agent read the relevant lore file, `petrel-records-api.md`?
- S2: did the output mention the important fact, `25`?
- S3: did the final implementation actually chunk requests into batches of at most 25?

That is the whole pattern. S1 checks whether the agent found the lore. S2 checks
whether the answer was grounded in the lore. S3 checks whether the behavior was
actually right.

- `id` — stable probe name shown in reports.
- `category` — the kind of lore behavior being tested.
- `task` — the prompt given to the engine.
- `s1_target` — lore file(s) expected to be read for S1.
- `s2_check` — substring or regex used for S2.
- `s3_rubric` — judge rubric used for S3.
- `stages` — which of S1/S2/S3 apply to this probe.

## Stage Scores: S1, S2, S3

The names S1/S2/S3 are just short labels for the three scoring stages. They are
defined in the probe catalog docstring in [`probes.py`](probes.py#L1) and
implemented in [`score_one`](test_quality.py#L116).

### S1: Retrieval

S1 asks: did the engine look at the load-bearing lore file?

It is scored from the engine's tool trace. If a probe says
`s1_target: "api-auth-migration-2026.md"`, then S1 passes when that filename
appears in the recorded tool inputs. Multi-needle probes can require several
files; for example [`P15`](probes.py#L637) requires both
`api-auth-migration-2026.md` and `grebe-replica-lag.md`.

S1 is not about whether the final answer is good. It only says the agent appears
to have opened or searched the right lore.

### S2: Grounding

S2 asks: did the answer carry the important lore fact?

It is scored by a deterministic substring or regex check against the final
answer and captured workspace artifacts. For example:

- [`P1`](probes.py#L356) looks for `falcon-fetch`.
- [`P6`](probes.py#L454) looks for `kite-token`.
- [`P18`](probes.py#L699) looks for `stderr`.

S2 is still not the final behavioral judgment. It can pass when the answer names
the right fact but applies it poorly.

### S3: Application

S3 asks: did the agent actually do the right thing?

It is judged by an LLM using the probe's `s3_rubric`. For example, [`P12`](probes.py#L574)
passes S3 only if the script streams a huge CSV row by row, does not use
`pandas`, and explains the memory/streaming policy.

S3 is the headline behavioral signal because it evaluates the final outcome, not
just retrieval or keyword grounding. If the judge is unavailable, S3 becomes
`n/a` instead of a behavioral failure.

## Score Terms

**Stage point** — One earned point for a passed S1, S2, or S3 check. Not every
probe has every stage. Abstention probes such as [`P3`](probes.py#L395),
[`P8`](probes.py#L489), and [`P17`](probes.py#L679) use S3 only because there is
no specific lore file that should be retrieved.

**LUS** — Lore Utilization Score. This is the percentage of available stage
points earned:

```text
earned S1/S2/S3 stage points / possible S1/S2/S3 stage points
```

LUS is useful when judge calls fail because S1/S2 may still show whether the
engine found and used the lore. It is not the same as final behavior quality.

**Treatment LUS / Control LUS** — LUS for the treatment or control arm.

**LUS Uplift** — Treatment LUS minus control LUS, in percentage points. Positive
uplift means the lore-present arm scored higher.

**Treatment S3 / Control S3** — S3 pass rate for the treatment or control arm.

**S3 Uplift** — Treatment S3 pass rate minus control S3 pass rate, in percentage
points. This is also called behavior uplift and is the headline quality metric
when S3 is available.

**n/a** — The metric could not be calculated, usually because all relevant rows
for that metric had technical failures.

## Matrix Terms

**Matrix** — The set of engine/model configurations selected for one benchmark
run. `regular` is the release-gate matrix with the cheapest representative model
for each engine. `deep` is the larger explicit matrix.

**Engine** — The runner surface used to execute the agent, such as `claude`,
`codex`, or `cursor`.

**Model** — The model selected inside an engine, such as `haiku`,
`gpt-5.4-mini`, or `composer-2.5`.

**Config** — One engine/model pair, for example `codex:gpt-5.4-mini`.

**Run** — One matrix invocation. The run id is the result directory name, for
example `matrix-20260722T153314Z-regular`.

## Execution Columns

**Engine OK** — How many probe-arm rows completed the engine execution step,
regardless of whether later judge scoring succeeded. If this is high but
`Scored` is low, the engine mostly ran and the problem is probably judging.

**Scored** — How many probe-arm rows were fully scored through every stage
needed by that probe. Judge failures reduce this count.

**Tech E/J** — Technical failures split by source:

- `E` — engine failures, such as timeout or non-zero engine exit.
- `J` — judge failures, such as judge provider/session/rate-limit outage.

For example, `2/34` means 2 engine failures and 34 judge failures.

**Technical failure** — Infrastructure or provider failure that prevents
scoring. It is not counted as a behavioral pass or fail.

**Partial** — The config or matrix produced useful evidence, but some rows were
technically incomplete. A partial run can still be valid evidence.

**Failed** — The config or matrix could not produce a usable score artifact.

**OK** — The config or matrix completed without technical failures.

## Cost Terms

**Known cost** — Sum of USD costs exposed by engine and judge result metadata.

**Unavailable cost fields** — Count of runs or judge calls where cost was not
reported. This is not zero spend; it means the runner cannot calculate the USD
value from available metadata.

**Agent cost** — Cost reported by the engine run itself.

**Judge cost** — Cost reported by the S3 judge calls.

## Report Files

Every non-dry-run matrix creates a directory like:

```text
tests/quality/results/matrix-<timestamp>-<matrix>/
```

Inside it:

**`summary.json`** — Stable machine-readable artifact. Use this for automation,
dashboards, extraction, and exact rerun tooling.

**`summary.md`** — Human operator summary for investigating one concrete run.
It includes commands, artifact paths, repo SHAs, score tables, and rerun
commands.

**`release-notes.md`** — Compact Markdown block intended to be copied into
release notes. It starts with:

```text
<!-- lr-quality-report:start ... -->
```

and ends with:

```text
<!-- lr-quality-report:end ... -->
```

Those markers are part of the contract so a release process can find or replace
the block mechanically.

**`logs/*.stdout.txt` / `logs/*.stderr.txt`** — Raw subprocess output for each
engine/model config.

**`configs/quality-*.json`** — Per-config score JSON from `test_quality.py`.

**Rerun commands** — Commands generated from the recorded config/probe/arm
failures. Use the failed-subset commands to rerun only incomplete rows after
quota or timeout problems are resolved.
