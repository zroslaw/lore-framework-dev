# Made the lore plugin marketplace-ready across engines; wrote v25 ship checklist

The session opened with the user asking whether the `lore-framework` repo was well-formed for the
Cursor plugin marketplace, pointing at the official `cursor/plugin-template` repository as the
reference. The framework already carried substantial Cursor packaging (the v21 dual skill tree,
`.cursor-plugin/plugin.json`, `.cursor-skills/` wrappers), so the task was an audit against best
practice rather than a build from scratch.

Grounding the check against the template's actual files rather than a summary paid off. The template
confirmed that single-plugin repos put `.cursor-plugin/plugin.json` at the root and **omit** a Cursor
`marketplace.json` (that file is a multi-plugin-monorepo construct), which validated our existing
layout. The audit surfaced three quality gaps: no logo asset, and missing `displayName` and
`keywords` in the Cursor manifest. All 30 `.cursor-skills` carried the validator-required
`name`+`description` frontmatter. With the user's approval, a self-contained placeholder
`assets/logo.svg` (a knowledge-graph glyph on an indigo badge) was committed and the Cursor manifest
enriched.

The conversation then moved to update mechanics and cross-engine marketplaces. The user asked how
Cursor plugin updates propagate, then how to get onto the Codex and Claude Code marketplaces with
good visibility. These were researched against official docs (cursor.com, code.claude.com,
learn.chatgpt.com/codex) and captured as two new lore topics —
`cursor-plugin-distribution-update-model.md` and `engine-marketplace-readiness.md` — plus corrections
to the stale `plugin-distribution.md`. The Claude manifest gained `$schema`, `displayName`,
`homepage`, and `keywords`, and `claude plugin validate --strict` was run: it flagged one issue (the
marketplace manifest lacked a `description`), which was added, after which both the marketplace and
plugin manifests passed strict validation clean.

The most consequential finding was a **Codex packaging discrepancy**: the framework's port-era lore
says Codex consumes the Claude `marketplace.json` via `codex plugin marketplace add`, but the current
official Codex build-plugins spec describes a distinct `.codex-plugin/plugin.json` (with an
`interface` visibility block) plus `.agents/plugins/marketplace.json` — and the repo ships neither.
Rather than assert either way, this was recorded as a verify-first open risk, since recalled lore
reflects what was true at port time and the doc read came from a fast summarizer.

Late in the session the user flagged that v25 felt unfinished and asked for an internal-review ship
checklist, with particular care for the multi-engine plugin angle. A direct git check established the
ground truth: branch `v25-cursor-ops-parity` at `4f3bfcf` has Cursor ops parity committed but unpushed
(last pushed is v24, `da473b6`); the workspace slice (workspace-pull/init renames) is designed but not
implemented; `migrations/25.md` is missing; and this session's manifest work is uncommitted. A
comprehensive **v25 SHIP CHECKLIST** was written into `framework-improvements-backlog.md`, replacing
the thin "Next-Session Active Threads" section — organized as scope decision, workspace-slice
implementation, per-engine packaging (Claude / Cursor / Codex handled separately), version-ship
disciplines, verification, and ship.

The session landed with all knowledge captured and finalized, but v25 itself deliberately not
advanced: the plugin-manifest and logo changes were left uncommitted in the `lore-framework/` working
tree so they can join a single coherent v25 commit next session. Nothing was pushed to `lore-framework`.

Next steps (next session, per the checklist): decide v25 scope (recommendation — ship Cursor
parity + workspace slice + Claude/Cursor manifest polish as v25, split Codex formal packaging to v26);
implement the workspace-slice renames; author `migrations/25.md`; verify the Codex packaging question
on a real Codex build; then run the full pre-ship gate (multi-lens review + lifecycle harness +
`/lr:check` + strict validate) before pushing v25.
