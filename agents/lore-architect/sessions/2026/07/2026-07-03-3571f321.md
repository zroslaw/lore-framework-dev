# Assessed lore-framework externally, then scoped Codex and Cursor ports

The session opened before any agent was booted, with the user asking for a plain outside
read on the `lore-framework` repo: was the architecture good, how did it compare to similar
projects, and what would improve it. That review covered the README, conventions, the
boot/reflect/merge process docs, the DF module, and a peek at the live `lore-agents` repo's
real usage, paired with web research into comparable systems (claude-mem,
claude-memory-compiler, the two unrelated `agentmemory` projects, memsearch,
claude-supermemory, and Claude Code's own built-in auto-memory). The verdict was that the
core bet — team-shared markdown-in-git with an explicit reflect/merge lifecycle — was sound
and more mature than most public alternatives, with the biggest risks being unverified merge
quality, no staleness signal at recall, and scope creep from the DF/AIQA module.

The user then booted the `lore-architect` agent properly via its registered shortcut. Boot
ran the version-check upgrade path for real for the first time — the repo was stamped at 17
against a framework `VERSION` of 18, and since v18 is release-notes-only (the `lr-wait`
MCP primitive) with no colliding dirty files, the upgrade stamped `lore-repo.md` to 18
cleanly. Asked to weigh in on the earlier outside review using its own accumulated design
history, the architect cross-checked it against `architecture-review-dispositions.md` and
`framework-improvements-backlog.md`: two of the review's suggestions (splitting the DF module
out, adding a schema field to enforce token budgets) turned out to contradict settled
decisions, most of the rest was already tracked, and two items were genuinely new — post-merge
diff verification and staleness surfaced at recall time — with a third (a lint script for
dangling references) folded into an already-known gap rather than proposed as new machinery.

The user then pivoted to a larger question: how portable is the framework to other agent
engines, specifically Cursor and Codex, with Claude Code intended to remain the major version
but broader adoption wanting equivalent experiences elsewhere. Fresh documentation research on
both platforms turned up a favorable, somewhat surprising fact — the industry had converged on
the framework's own primitives since it was designed. `SKILL.md` is now an open, cross-vendor
standard that Codex and Cursor both implement (Codex having deprecated its older custom-prompt
mechanism in its favor), both read `AGENTS.md` natively, and both support MCP, meaning the
framework's knowledge substrate (plain markdown in git) was already portable and the
procedural layer (skills, subagents, memory-file injection) needed packaging more than
redesign. The architect produced a dependency-by-dependency portability assessment covering
what ports free, what needs mechanical rework, and what stays Claude-only for now.

Asked for detailed, separate migration plans to be worked in their own dedicated future
sessions, the architect wrote two full drafts into `workdir/`: `draft-port-codex.md` and
`draft-port-cursor.md`, each with a verified platform-capability table, a dependency-to-target
mapping, a phased plan (a shared engine-neutral Phase 0 covering an `docs/engines/` adapter
convention and an `AGENTS.md`-first init target, then per-engine phases), open questions, and
risks. Both converge on the same dominant risk — the framework's procedures are prose executed
by the model, and fidelity on non-Claude models is unverified — and both end with an empirical
fidelity report intended to feed the framework's already-parked simplification theme rather
than just papering over per-engine quirks.

The session closed with a full finalization. Four reflection topics were written and merged: a
competitive-landscape survey (with the new positioning insight that multi-engine support is a
differentiator no surveyed competitor can match, since all of them are bound to one engine or
one vendor), the two new backlog items from the cross-check, and an anchor topic for the
portability direction itself, added to `lore-context.md`'s Active Design Explorations
alongside the existing `lr-dev` and autonomous-agents entries. The merge subagent also caught
and fixed a stale topic count in `lore-context.md` left over from an earlier session. One
thing worth flagging plainly: this finalize commits and pushes only `lore-framework-dev` (this
agent's lore) — the `lore-framework` plugin repo itself still carries the entire, fully-built
`lr-wait` v18 feature completely uncommitted, deliberately held from the prior session pending
the user's separate go-ahead to ship it. That decision was untouched by this session and
remains open. Next steps: commit and push the plugin repo when the user is ready to ship v18,
and pick up either port draft in its own dedicated session.
