# Agents are executors first, advisors second

When framing Lore Agents to newcomers, **lead with execution as the primary value, conversational use as the secondary affordance**. Don't reverse the order — readers stuck on the advisor framing miss what makes Lore Agents categorically different from "a wiki you can chat with."

- **Primary value:** *getting things done* — quickly, efficiently, with less tedium. Agents work *with* people on real tasks: coding, reviewing/merging MRs, arranging/analyzing JIRA tickets, navigating unfamiliar code, automating repetitive day-to-day work.
- **Secondary affordance:** because the agent has the accumulated context (recent tasks, decisions, history, rationale), it can *also* answer questions about its area. Useful for onboarding and recovery. But not the motivating use case.

The framing matters because it shapes:

1. **Why people adopt them** — practical productivity, not knowledge-curation duty.
2. **Why the feedback loop works** — daily execution drives learning; daily learning makes execution better. Advisor-only framing breaks the loop. See `positive-feedback-loop-framing.md`.
3. **The trajectory** — current Activities lore agents skew advisor-y today, but the framework is designed for execution. Hide the present-state lean in onboarding prose; the trajectory is real.

Companion to `team-shared-knowledge-principle.md` and `framework-as-engine-not-kb.md`. Together these three name the foundational identity claims of the framework: shared (not personal), engine (not KB), executor (not advisor).
