# Built and validated a multi-engine lifecycle testing harness, fixing two doc bugs it found

The session opened with a follow-up on prior work: the user wanted to move the multi-engine portability direction (Codex and Cursor ports, previously only sketched in two workdir drafts) toward something testable, and asked how to arrange automated testing across platforms — including Claude Code itself. After early back-and-forth about verbosity (the user twice asked for plainer, shorter language, which reshaped the rest of the session's communication style), the design converged on a three-layer pipeline: fast deterministic script tests, static lint checks, and a Layer-3 lifecycle harness that runs a real engine headless against a throwaway fixture workspace and asserts on the resulting files and git state. The user asked for one shared, engine-neutral scenario catalog — fixture, prompt, assertions per scenario — with only a thin per-engine driver differing. That catalog and design were written to `workdir/draft-testing-pipeline.md` and linked from `multi-engine-portability-direction.md` as its Phase 0.5.

The user then asked to actually build a few scenarios and smoke-test them, which the session did: `tests/lifecycle/harness.py` (a fixture builder using a local bare git remote for zero-network testing, plus a `claude -p` headless driver) and an initial three boot scenarios, first validated on sonnet. Asked to compare against haiku, the session found real failures — not flaky ones, but two genuine ambiguities in `agent-boot.md` that a weaker model resolved incorrectly: it confused "current working directory" with the plugin directory it had just read from (escalating to a filesystem-wide `find` that was also the source of macOS permission prompts the user had separately noticed), and it read "best-effort — never blocks boot" as license to skip the auto-pull step outright rather than just tolerate its failure. Both were traced precisely by re-running with `--output-format stream-json` to inspect the actual tool calls, then fixed with targeted wording changes in `agent-boot.md`. Haiku's boot pass rate went from roughly 50% to 6/6 clean across two full trials after the fix.

From there the session scaled up substantially. Three more scenarios were added and validated on haiku (recall, reflect, and — most importantly — merge, the framework's own highest-risk procedure per its design docs, which passed cleanly twice). Then, at the user's request to implement "the rest" of the catalog, thirteen more scenarios were built and validated on sonnet in a single pass with no further debugging needed: boot-unknown-agent, boot-version-mismatch, the upgrade-gate-on-dirty-tree collision check, consult, attach, summarize, a full finalize end-to-end run, and the repo/workspace group (create-repo, create-agent, init, workspace-sync, check, update --dry-run). Nineteen of the catalog's twenty-one Tier-1 scenarios now pass on Claude Code. Two were deliberately deferred (concurrent-finalize collision and finalize-with-guest, both needing more fixture scaffolding than the rest), along with the four Tier-2 Claude-only items, with reasons recorded rather than silently dropped.

The clearest surprise was that the harness had value the design hadn't fully anticipated: it wasn't just a port-readiness gate, it was already a live fidelity check on Claude Code's own procedure docs, and it found real bugs on its very first use. That observation was promoted into its own principle during finalization — prose ambiguity a strong model resolves by inference is invisible to prose review and only surfaces under execution by a different or weaker model — which also justified a role update: the architect now owns running the harness as a second, empirical leg of pre-ship verification, alongside the existing multi-lens prose review.

The session ended with reflect and merge run for the host agent. Merge created three new lore topics (the harness implementation, the fidelity-fix case study, and the new execution-testing principle), updated `multi-engine-portability-direction.md` and `system-design-principles.md`, and integrated a role-update into `role.md`. One thing intentionally not yet done: the actual test code under `tests/lifecycle/` and the `agent-boot.md` fix in the `lore-framework` plugin repo sit outside `agents/lore-architect/`, so `/lr:finalize`'s Phase 4 — scoped to `agents/` by design — will not sweep them into this commit. Those need their own explicit commits, in their own repos, before this session's actual code work is preserved anywhere but the local disk.
