# Opened lr-dev: an SDLC extension toward a dark-factory SDLC

The session opened with the user articulating a north star: turn the software-development lifecycle into a "dark factory" — software built internally and autonomously by lore agents, with only inputs and occasional follow-ups. As a deliberately safer first step, the user wanted a framework and workflows to automatically find bugs and inconsistencies in production repos and raise unit-test coverage. The architect recognized the north star as nearly identical to the already-parked `autonomous-agents-vision.md`, and grounded the rest of the conversation in existing lore rather than starting cold.

Early on, the architect argued that QA is the *correct* beachhead, not merely an easy one: a dark factory needs trustable autonomy, trustable autonomy needs cheap automatic verification, and QA is the one SDLC activity that carries its own oracle (tests pass/fail, coverage moves, bugs reproduce) while having near-zero blast radius. From there the design grew through several user braindumps. The architect proposed the load-bearing spine — a **three-tier knowledge model** (framework / repo / agent), where the new middle tier, *repo lore*, holds what is objectively true about a codebase (product + technical + per-artifact), is agent-agnostic, and survives onboarding a fresh agent, versus *agent lore*, which is the worker's subjective experience.

The user steered the architecture in concrete directions: don't create a dedicated QA agent yet; treat lr-dev as a module/mode inside the existing `lr` plugin rather than a separate plugin; gate repo-lore writes behind two conditions (repo is lr-dev-enabled via a soft `CLAUDE.md` signal pointing to the repo-lore remote, and the agent is in lr-dev mode, permanently via `role.md` or temporarily via a session switch). The user also specified that product and technical lore must be stored separately with no cross-duplication, each with its own `*-lore-context.md` entry point, and that per-file `.lore.md` artifact lore must stay sparse and lean, maintained on a finalization-on-touch basis. The user named `dev-repo-lore.md` as the new descriptor and asked it to carry a framework `version` and a `source-repo` link.

Two strong reuse points surfaced. The user's proposed three-subagent independent code review turned out to be the architect's existing `parallel-reviewer-fanout-pattern.md`, so it was promoted to a reusable `lr:dev-review` skill. And the user's "FileAgent/UnitSubAgent" terminology collided with the reserved LRF "Agent" term, so the roles were renamed to lowercase **file quality analyst** and **unit quality analyst**. The architect also proposed treating test scenarios as a bidirectional intermediate representation (framework-agnostic, forward to code and reverse for gap analysis) and flagged the test-gen trap: generated tests can canonize current buggy behavior unless every expected assertion cites its source of intent.

All design detail was captured in `workdir/draft-lr-dev.md`, which the user then asked to split — so the general concept (knowledge model, mode mechanism, repo lore, review skill) stayed there and a companion `workdir/draft-lr-dev-quality.md` was created for the first feature (the file-by-file workflow plus the standardized bug/scenario/file-report schemas and format rules). A backlog section for lr-dev was added, including the cross-cutting consequence that `/lr:update`, boot auto-upgrade, and version-check/auto-pull must eventually walk the new `dev-repo-lore.md` repo kind.

The session landed with both drafts stabilized, a backlog entry, and — through finalization — a new anchor lore topic `lr-dev-direction.md` cross-linked to the autonomous-agents direction. The architect designed mode-switch (mode = loaded context + acknowledged capability, like boot; enforcement by convention) and the three deliverable schemas, but these are proposals pending the user's reactions. Two decisions remain explicitly open: how `role.md` declares permanent lr-dev mode (frontmatter `extensions:` field versus a body marker), and where File reports physically live (with prune-versus-retain). The user paused the thread to preserve context, flagging a second concurrent idea to explore in a separate session that should carry these essential points forward; the natural next concrete move on this thread is grounding the test-scenario schema on the `My-Turbo-Boost-Switcher` mouse repo.
