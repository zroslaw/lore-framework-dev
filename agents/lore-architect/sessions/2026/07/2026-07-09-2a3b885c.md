# Expanded the quality benchmark's probe catalog and fixed a judge blind spot

The session opened with the user booting lore-architect and asking for a plain explanation of how the lore-agent quality benchmark works — what it measures, how the probes and scoring stages fit together, and whether it was worth improving. The host walked through the mechanics: planted-knowledge probes run in treatment/control arms against an identical fixture repo, scored across three stages (S1 retrieval, S2 grounding, S3 behavior via LLM judge), with Behavior Uplift as the headline cross-engine metric. The user then asked pointedly whether the benchmark's fidelity was good enough, specifically pushing on test coverage and category breadth rather than the operational punch-list the host had offered.

The host's assessment was that the original 8-probe, 5-category catalog was adequate for the coarse question it was built to answer (does lore produce any uplift at all) but too thin for anything finer — one flipped probe moved a category's score by 50 points, and whole classes of lore payload (negative knowledge, cross-topic synthesis, workdir-tool reuse) were untested. The user agreed and asked for the catalog to be expanded before the queued regular/deep tier restructure locks it in. The host designed and implemented a v2 catalog: 18 probes across 8 categories, adding negative-knowledge (a rolled-back Redis cache, a banned pandas policy), synthesis (two probes requiring two needles combined), and workdir-tool (an audited script the agent should reuse rather than rewrite) as new categories, plus a third variant each for decision-recall, gotcha-avoidance, abstention, knowledge-update, and implicit-adaptation. Supporting harness changes included list-valued `s1_target` for multi-needle synthesis probes and a `WORKDIR_FILES` fixture mechanism for the workdir-tool probe.

Separately, the user asked the host to also land improvement #1 from its own punch-list: giving the S3 judge visibility into workspace artifacts, not just the agent's final message — a known blind spot that had been deflating scores for file-writing engines. The host implemented `collect_workspace_artifacts()` (git diff plus new files, capped and sanitized) and wired it into both the judge prompt and the S2 grounding check. Validating the change surfaced a real bug: passing captured cursor-engine output through to the judge's `subprocess.run` call crashed with an embedded-NUL-byte error, since UTF-8 decoding with `errors="ignore"` does not strip `\x00` and `execve` requires NUL-terminated strings. The host added a `_strip_nulls()` guard at every capture boundary rather than patching only the crash site.

The plot twist came from a costly execution mistake: the host ran the first real shakeout of the new catalog on claude+sonnet by default, without accounting for the fact that the S3 judge is always claude-billed regardless of engine — doubling up on the same subscription quota and exhausting the session limit mid-batch. The user corrected this sharply ("always run tests with cursor! cursor + composer 2.5!!!"), and the host saved both a cross-session Claude Code memory and a lore reflection capturing the rule: ad-hoc quality-benchmark shakeout runs default to `LR_ENGINE=cursor`, since only the judge then touches the shared claude quota. The retried cursor+composer-2.5 run also hit the embedded-NUL-byte bug for the first time (cursor writes files directly, exercising the artifact-capture path that claude/codex's message-based capture never did), confirming the fix was necessary, not speculative.

The rerun after the null-byte fix completed cleanly: 0 errors, 0 invalid judge cells, treatment 100% S3 pass rate (18/18), control 33.3% (6/18), behavior uplift +66.7 points. Reviewing the per-probe scorecard surfaced a second finding, unprompted by the user: two of the new probes — P12 (banned-library-avoidance) and P16 (workdir-tool-reuse) — passed in *both* arms, meaning the model's default no-lore behavior already matched the lore-mandated behavior, so those probes currently measure nothing. The host flagged this rather than treating the strong aggregate uplift as the whole story. A final small fix, requested by the user, corrected the scorecard's fixed 24-character probe-name column, which had been corrupting alignment for the longer v2 probe IDs; the fix computes the column width from the data instead of a hardcoded constant.

The session landed with the v2 catalog, the judge-artifact fix, and the null-byte fix all implemented and validated by one clean real-engine run, but **not yet committed** — `tests/quality/` sits outside finalize's `agents/` commit scope by existing convention and needs its own manual commit, same as the lifecycle harness. P12 and P16 remain open and explicitly un-fixed; the host judged that reworking their task framing needed its own deliberate pass rather than a rushed patch at session end, and the user did not ask for a rerun. Finalization (reflect → merge → summarize) then ran to close the session: reflection produced four topics, all merged into lore-architect's existing knowledge graph — `quality-benchmark-feature.md` and `benchmark-harness-operational-lessons.md` updated in place (the cursor-smoke-test rule folded into the existing quota-batching lesson rather than duplicated), two new topics created (`quality-benchmark-v2-catalog.md`, `quality-benchmark-v2-known-probe-gaps.md`), and `lore-context.md`'s benchmark pointer refreshed.

Next steps: rework P12 and P16 with harder task framing so the lore's actual contribution is what's being tested, not general model competence; re-run the full catalog once fixed; then proceed with the already-queued regular/deep tier restructure (`quality-benchmark-tiers-proposal.md`) now that it has a sounder catalog to lock in. The `tests/quality/` code changes remain uncommitted and need a manual commit outside this finalize's scope.
