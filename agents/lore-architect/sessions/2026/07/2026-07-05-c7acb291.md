# Cursor dual skill tree for prefixed picker names

The session opened with the user confirming the Lore plugin loaded in Cursor but finding unprefixed
skill names (`boot`, `recall`) in the dropdown confusing — they expected the `lr` namespace visible
on Claude Code as `/lr:boot`.

The agent verified Cursor official docs and empirically confirmed the root cause: Cursor uses the
skill **folder name** as picker identity; plugin manifest `name` does not namespace skills, and
frontmatter `name:` cannot override a mismatched folder. Category subfolders do not help either.

The user asked whether one canonical repo could serve all engines gracefully. The agent proposed
and implemented a **dual skill tree** in local `lore-framework` source (v21, not pushed to remote):
canonical `skills/<skill>/` for Claude Code (`/lr:<skill>`) plus `skills/cursor/lr-<skill>/`
wrappers loaded via `.cursor-plugin/plugin.json` for Cursor (`/lr-<skill>`). A
`scripts/sync-cursor-skills` generator and `/lr:check` step 21 maintain parity. All 27 wrappers
were generated and verified with `cursor-agent --plugin-dir`.

A brief detour booted lore-architect mid-session (`/lr-lore-architect-agent`) to continue design
work in role. The user then directed keeping framework changes local only while finalizing the
lore-architect session and pushing `lore-framework-dev`.

Reflection and merge captured the pattern in `cursor-dual-skill-tree-one-repo.md`, updated
`slash-command-system.md` and `cursor-port-validated-end-to-end.md`, and noted the superseded
`lore-framework-cursor/` sibling is deletable.
