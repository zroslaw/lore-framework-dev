# Designed Codex-native installation and shortcuts

The session opened by revisiting the remaining Codex migration gap: Lore finalization could write
lore files under the default sandbox but could not update `.git`. The user chose the stronger
product contract: supported Codex finalization requires `.git` write access. The Codex engine
profile and README now say that the plugin cannot grant this permission itself; trusted Codex
launch or configuration must provide it, while a commit-blocked run remains a degraded fallback
rather than the intended workflow.

The discussion then shifted from user installation instructions to a genuinely Codex-native
experience. The desired flow is that a user pastes the Lore Framework repository link into Codex
and asks it to install. Codex should discover the short installation contract in the README, run
the marketplace and plugin commands itself, request only necessary approvals, and tell the user
to restart Codex so the plugin and its update path become active.

The session also explored how named Lore agents could appear naturally in Codex prompt
autocomplete. The design settled on engine-native personal skills under
`~/.codex/skills/lr-<agent-name>-agent/SKILL.md`, invoked as `$lr-<agent-name>-agent`. The
registration, unregistration, listing, consistency-check, and surrounding documentation were
updated to describe Claude commands and Codex personal skills as two forms of the same thin
absolute-path boot delegation.

A key boundary emerged during the work: the documentation and design are complete, but
Codex-aware generation and removal of those personal skills are not yet implemented or
lifecycle-tested. The lore records this gap explicitly. A possible future split of growing
Codex-specific operational guidance into dedicated files was added to the framework-improvements
backlog rather than expanding the current scope.

The session landed with the repository-driven installation contract, `.git` permission decision,
and per-agent shortcut design reflected consistently across public docs and lore. Next work is
implementation and lifecycle validation of Codex-aware `/lr:register-repo`,
`/lr:unregister-repo`, and related list/check behavior; automatic sandbox widening remains outside
the plugin's authority.
