# Designed and shipped session summaries feature for lore framework v7

The session opened with the user proposing a new capability for the lore framework: preserve and store the history of each session at finalization, without consuming the active session's context. The initial sketch was to parse Claude Code's internal JSONL session files, filter out technical noise, and reuse the result both as input to reflection and as a long-lived archive for later search and meta-analysis of the framework itself.

The architect pushed back gently on two fronts and then dug into the evidence. An investigation located the current session's JSONL at `~/.claude/projects/-Users-yaroslav-Documents-git-repos/02cf720a-6923-4988-af8e-4b2450852dd6.jsonl` and inspected its schema — line types of `user`, `assistant`, `system`, `attachment`, `file-history-snapshot`, and `last-prompt`, each carrying timestamps and a `sessionId`. The investigation surfaced concrete obstacles: the `~/.claude/sessions/<pid>.json` registry goes stale after `/clear` (seen live this session), cwd encoding into the directory name is lossy, the format has no public stability contract, and the filtering decisions are not obvious — tool calls are simultaneously the richest signal and the noisiest content.

The plot twist came when the user absorbed the findings and pivoted the design. Instead of parsing the JSONL, the framework would write a short, model-composed markdown summary at finalization — a human-readable record with YAML frontmatter, committed to the agent repo and shareable. Correlation back to the private JSONL would be established through a UUID: generated by summarize, stored in the summary's frontmatter, and echoed in the agent's user-visible output so it lands in the JSONL naturally. Anyone who wants to later replay a session can `grep` for the UUID in their own `~/.claude/projects/`.

Design work continued through a review cycle. The architect drafted `workdir/framework-improvements.md` and `workdir/draft-session-summaries.md`. The user reviewed and locked the key decisions: session-wide scope with a `participants` field (not per-agent iteration), host agent's repo as the storage location, year/month nested directories to avoid single-directory bloat, UUIDv4 in frontmatter with short 8-char prefix in the filename, free-form kebab-case topic tags to support later analysis, and a mandatory user-review gate before any file is written. Three known limitations were captured explicitly in the backlog rather than papered over: compaction-aware narrative quality, cross-repo guest participation, and reliable start-time capture.

Implementation was a clean pass across the framework. A new `skills/summarize/SKILL.md` followed the thin-skill pattern, pointing to the new authoritative `docs/summarize.md` which defined the full process, frontmatter schema, narrative prompt, privacy conventions, and failure modes. The existing `skills/finalize/SKILL.md` gained a third phase so finalize now orchestrates reflect → merge → summarize, with summarize explicitly additive and non-blocking. `docs/conventions.md` grew a `sessions/` entry in the agent-repo directory tree. The framework `VERSION` bumped from 6 to 7. A new `release-notes/7.md` described the feature end-to-end. No migration was needed since `sessions/` directories are created on demand — the user accepted the simpler path over a formal migration step.

The session landed with v7 fully implemented as unstaged local changes in `lore-framework/`. All files are internally consistent. The user triggered `/lr:summarize` immediately after plugin reload as the dogfood test of the new skill — this file is that test. Next steps on the user's side: review the generated summary, approve or decline the write, commit the framework v7 changes, and on the next boot of any v6 agent repo the auto-upgrade will stamp `version: "7"` after displaying the release notes. The three backlog items wait in the architect's workdir for a future session to pick up.
