# Built three communication "style skills" as regular `/lr:` commands

The session opened as a status check: the user asked where the effort to run the framework on
OpenAI Codex had stopped. The architect recalled the concrete resume point — the framework-root and
defer-clarity change sets validated on Claude/haiku but staged in an isolated copy, with the
subagent-adapter pass queued as the next dedicated port session. The user then pivoted to a new,
practical goal: deliver skills that reduce the day-to-day communication friction between them and the
agent, starting with one asking the agent to write in plain, simple English (the user reads English
as a second language and finds compressed prose costly to parse).

Work began on that first skill. The architect helped name it (`plain-language`) and word the rule,
then — on the user's instruction to "make it a skill in the lr framework" — built it as a
**framework-level soft skill**: a new `docs/soft-skills.md` registry plus a new `agent-boot.md` step
that would load it for every agent at boot. This was effectively an attempt to resolve the
long-open "soft skills as a formal entity" backlog item on the fly.

That was the session's main plot twist: the user rejected the mechanism. They wanted a **regular,
invokable `/lr:` skill** — a slash command — not a boot-loaded subsystem, and asked to keep the
soft-skill concept separate. The architect reverted the whole soft-skills subsystem cleanly
(`agent-boot.md` back to its committed state, `docs/soft-skills.md` deleted) and rebuilt
`plain-language` as a standard thin-pointer skill (`skills/plain-language/SKILL.md` →
`docs/plain-language.md`). The user then asked to extract the existing `follow-me` behavior out of
lore up to the framework the same way, yielding `/lr:follow-me`; the old lore topic
`soft-skill-follow-me-mode.md` was reduced to a pointer holding design history. Finally the third
skill, `/lr:dialogue`, was designed and built — short conversational turns, essential-piece-first,
keeping the user's context in sync rather than dumping long articles. The three were framed as a new
**style-skills category** composing on three levels: sentence (`plain-language`), turn (`dialogue`),
and thinking-direction (`follow-me`).

Where it landed: all three skills are built and left **uncommitted** in `lore-framework/`, held
deliberately to ship together with the codex-adoption release rather than with v18 (the finished but
still-uncommitted `lr-wait` release). The style-skill mentions the architect had temporarily written
into `release-notes/18.md` were pulled back out, restoring v18 to lr-wait-only. The decision on
timing was delegated to the architect, who chose to hold the skills for the port release to keep the
version history clean. Reflection and merge then recorded the new category and the over-build lesson
into lore: new topics `style-skills.md` and `skill-request-defaults-to-regular-skill.md`, updates to
`slash-command-system.md` (operations-vs-style distinction), `framework-improvements-backlog.md`
(soft-skills disposition), and a reconciled `lore-context.md` that replaced the stale "Soft Skills
(surface at boot)" section with a "Style Skills" one.

Next steps: the skills need a release-notes entry when the codex-adoption version is cut — tracked in
`port-landing-next-steps.md`, which remains the resume anchor for the port (subagent-adapter pass,
then apply the staged change sets to the real framework and run the full suite). Nothing was pushed;
the three style skills stay local by the user's choice.
