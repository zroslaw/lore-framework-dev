# Built and validated the lore-agent quality benchmark

The session opened with the user asking for a new kind of test: the lifecycle
harness verifies that agents follow framework procedures, but nothing measured
whether a lore agent actually serves its purpose — recalling the right
knowledge when a task needs it and adjusting its behavior accordingly. The
user wanted a measurable number for comparing lore agents' output quality,
with cost and time kept as separate metrics.

A research pass over industry evals (needle-in-a-haystack, RULER, NoLiMa,
LongMemEval-V2, BEAM, ImplicitMemBench, RAGAS) produced the design, captured
in `workdir/draft-lore-quality-benchmark.md`: many simple planted-knowledge
probes, each scored in three stages (retrieval, grounding, application), run
in treatment and control arms so the score delta — Behavior Uplift — isolates
what the lore itself buys. The user approved a PoC, which landed as a third
test track in `tests/quality/` (probes, engine-driving harness, unittest
runner gated behind LR_QUALITY=1), reusing the lifecycle harness's fixture
and canary discipline. The catalog grew in two rounds: an easy trio proved
the pipeline (every configuration scored 100% with lore, ~33% without), and
when the user asked for tests where differences would be visible, five hard
probes were added — NoLiMa-style paraphrase, a two-hop link chain, a
knowledge update over a stale topic, an implicit dry-run norm, and a tempted
abstention — plus engine support so one env var switches between claude,
cursor, and codex.

The plot twists were operational. A batch of 80 concurrent claude-billed runs
hit the subscription's session limit twice, which exposed a genuine harness
bug: the judge's limit-message replies were being scored as FAIL, silently
corrupting two configs' scores. The fix — judge failures invalidate cells and
fail the suite loudly, engine timeouts fail one probe rather than the suite —
was itself validated by the second limit hit, which produced a clean partial
scorecard instead of garbage. A second finding: the S3 judge sees only the
final message, so engines that write files and summarize (cursor+sonnet
chronically) get undercredited; this remains the benchmark's main measurement
debt.

The final five-configuration table spread exactly as designed: treatment
scores 62.5–100%, uplift +37.5 to +75. Composer-2.5 on cursor and sonnet on
claude both aced the hard catalog; haiku and gpt-5.4-mini each dropped one
probe; sonnet-on-cursor trailed — establishing that model–engine fit, not
model tier, predicts how reliably lore becomes behavior. Controls failed
identically across vendors (all proposed 02:00 cron jobs inside the planted
forbidden window), and all twenty abstention cells passed. The user
considered a LongMemEval comparative track for benchmarking against memory
systems like Mem0 but deferred it, citing its drawbacks — its verbatim-recall
bias fits poorly with lore's deliberately curated knowledge.

Finalization ran without phase 4 at the user's request: four new lore topics
(the quality-benchmark cluster), three cross-referenced updates, and this
summary are left uncommitted, as is the entire `tests/quality/` tree — which
in any case needs its own commit outside finalize's `agents/` scope. Open
threads: committing both trees, the judge-sees-workspace-diff improvement,
k>1 repetitions for score stability, and folding the extended-catalog results
back into the design draft.
