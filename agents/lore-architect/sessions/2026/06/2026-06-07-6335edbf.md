# Closed the DF/ULA design thread and landed the DF rename

The session resumed the Dark Factory / ULA design thread that had been parked on 2026-06-05. The user booted lore-architect, confirmed the held state, and worked through the five open items in **follow-me mode** — driving each naming and structure decision while the agent tracked the direction and offered only small suggestions. By the end of the discussion every open item was settled: per-file artifact granularity (the unit as a *field*, not a folder), a grouped per-file artifact shape (`{ source-sha, config, units: [...] }`), the Provenance Header field set, and a full naming scheme. The naming work resolved several collisions deliberately rather than by qualification — the top-level directory became `repo-lore/` (rejecting `sources/`, which collides with a source repo's `src/`), the per-file directory became "Source File Lore," the landing file became `file-lore.md` (the file-level analogue of an agent's `lore-context.md`, with "context" reserved for its context-window meaning), and "artifact" was narrowed to mean a generated output only, with the input always called the "source file." A running glossary was captured at `workdir/draft-df-glossary.md`.

With the design closed, the user asked to align the plugin to the Dark Factory north star, and the agent executed a full rename across `lore-framework/`: the module directory `dev/` became `df/`, the skill prefix `dev-` became `df-` (so `/lr:df-repo-init` and `/lr:df-ula-file`), the output repo `<repo>-aiqa` became `<repo>-df`, `aiqa.config.yaml` became `df.config.yaml`, and `repo-init` graduated out of the AIQA module up to `df/repo-init.md` because it is aspect-agnostic. AIQA was kept as the umbrella concept and the `df/aiqa/` aspect subdirectory. The agent re-pointed every reference, added `provenance.schema.json`, reshaped the artifact docs to the per-file/grouped layout, and verified the sweep was clean.

A requested consistency pass over the whole ULA process — reading the prompts, schemas, workflow, and skill rather than only grepping — surfaced real gaps. The most important: `source-sha` was being computed with `git rev-parse HEAD:<path>` (the committed blob) while the pass analyzes the working-tree file, so the provenance would lie on any dirty file; this was corrected to `git hash-object`. The read also caught stale prose in `split.md` ("becomes the unit's directory") that the token grep had missed. The agent then implemented "B" — dropping the whole-file `fileContents` from the workflow args and having the split agent read the file itself — which unified the split-vs-unit read approach and made the provenance honest across all agents.

The plot twists came from running the workflow for real. The user executed `/lr:df-ula-file` in a separate session and it failed at the args guard. The agent had offered two candidate causes — payload size versus args string-coercion — and the user chose the size fix (B); but B did not fix it, because the actual cause was that the dynamic-Workflow runtime delivers `args` as a JSON *string*, proven by a three-line repro. The real fix was a `JSON.parse` guard. A second runtime bug then appeared: `agent({schema})` rejected the schemas' draft-2020-12 `$schema` meta-URL (the runtime's validator is draft-07-era), fixed by stripping `$schema` at the `agent()` boundary while keeping it in the files for editor tooling. With both fixes the pass ran end-to-end on `My-Turbo-Boost-Switcher/CheckUpdatesHelper.m`, producing five units and roughly twenty substantive Objective-C findings. When the agent went to canonicalize the fixes, it found the testbed's `$schema`-strip had never been persisted to the file — the green run had used an in-session edit — so the strip had to be properly added (a `noMeta` helper) rather than assumed present.

The agent verified the produced artifacts in `My-Turbo-Boost-Switcher-df/` matched the design on every axis — the `repo-lore/<file>/ula/` layout with no unit directory, the Provenance Header whose `source-sha` equalled a live `git hash-object`, the grouped `units: []` shape, and per-element gap invariants — confirming the restructured ULA works. The session then finalized: eight reflection topics were merged into lore (two new operational topics, `dev-module-conventions.md` renamed to `df-module-conventions.md` with a new workflow-authoring checklist, the now-closed `df-ula-design-in-progress.md` deleted, and thirteen topics updated), and `lore-context.md` was refreshed to record the closed thread and the landed rename.

The plugin changes (the DF rename, the structural align, and the two workflow fixes) are staged but not yet committed — the framework-code commit is being kept separate from the lore commit, per the user's show-before-push preference. Open follow-ups: commit and push the plugin work; re-validate ULA on a file that has existing tests, since `CheckUpdatesHelper.m` had none and the clean-room comparison path (gap Direction B) went unexercised; design the `file-lore.md` "context" aspect, which ULA does not write; and eventually ship v16 (VERSION bump, plugin-manifest bump, release notes) once the BETA stabilizes.
