# Fixed a macOS PID bug, added Codex support, and shipped Lore Beings as v28

The session opened with lore-architect booting from a prior handoff: the Lore Beings module (autonomous scheduled agents, driven by a deterministic "Being Keeper" daemon) had been built, hardened across two Claude Code review rounds, then taken over by a Codex session for a third review — but that third review ran inside a sandbox that blocked `ps`, and the result was left as local, unpushed worktree checkpoints in both `lore-framework` and `lore-framework-dev`. The user asked lore-architect to review the current implementation and fix or improve it as needed.

Running the Codex checkpoint's own test suite on real, unsandboxed macOS immediately failed two PID-reuse tests. The root cause was a genuine Keeper bug invisible to every prior review: `ps -p <pid> -o lstart= -o command=` prints both fields on one line on macOS, not one per line as the code assumed, so the recorded process-identity "start time" secretly embedded the (mutable) command line — and a genuinely alive re-adopted session could read as a confirmed PID-reuse mismatch and get reaped while still running. The fix splits the query into two separate `ps` calls. A companion test bug (dereferencing an already-reaped list entry) and a minor outbox-normalization gap were fixed alongside it, and the framework and dev worktrees were checkpointed and merged into their repos' local mains.

A stop-hook then flagged that the original ask — "provide a ready to ship version" and "e2e test it for Claude Code and Codex" — was only half met: only Claude Code had ever been e2e-tested, and the Keeper's `spawn_session` hardcoded Claude's CLI shape entirely, with no path for Codex at all. Closing that gap meant real design work: probing the actual `codex exec --json` contract directly (JSONL events, a terminal `turn.completed`/`turn.failed`, token usage but critically **no USD cost field anywhere**), then adding an engine-`kind` concept (`claude`/`codex`) so each configured engine declares its own invocation argv and result-parsing contract. Because Codex reports no cost, `lrb engines add` now requires a mandatory flat `--session-cost-usd` for codex-kind engines, charged per finished session regardless of outcome — otherwise the daily-usd budget gate would silently never trip, the same "unenforceable cap" failure shape the MVP design had already named for a different case. Six new tests were added against an argv-verifying codex stub fixture (76 focused Keeper tests total, 135 across the dev suite), and the feature was proven against real engines: a live `claude`/haiku session and a live `codex`/gpt-5.4-mini session, both driven through the actual Keeper tick loop end to end, both finishing `ok` with correctly-charged cost.

With both engines verified, lore-architect bumped the framework to v28, updated all four version-bearing plugin manifests to `1.28.0`, wrote `release-notes/28.md` with the required cache-clear footer, backfilled `versioning-release-types.md`, and committed the release on `lore-framework` main (`44bc57d`) on top of the two fix commits. The `lore-framework-dev` main also picked up the test changes via a merge commit, after manually resolving a three-hunk conflict in `workdir/draft-lore-beings.md` between the branch's build notes and a later hand-edit already on main.

The release is committed locally but **not pushed or tagged** — that remains the honest state, since this session's real-engine e2e was a targeted smoke test through the Keeper's own tick loop, not a run of the framework's full lifecycle test harness, which is this agent's own stated pre-push gate for anything touching procedure-adjacent code. The user then ran `/lr:finalize`. Reflection produced five topics: the empirical Codex JSONL contract, the macOS `ps -o` multi-field gotcha, a role-update candidate naming the broader "a review environment with a blocked capability can green-light an unexercised code path" principle (extending the existing empirical-verification discipline), the engine-`kind` design decision, and the v28 release-state note. Merge — run in a single subagent since no guests were attached — folded these into `lore/` (three new topics, two existing topics extended), added the new curation-discipline bullet to `role.md`, and refreshed `lore-context.md`'s Lore Beings and Versioning sections, incidentally also fixing a pre-existing v26/v27 inconsistency it found in the same paragraph.

What remains open: the full lifecycle suite and the push/tag for v28, `lrb install --launchd` on a real machine, and the module's actual MVP success criterion — the Chronicler completing a full week unattended — none of which happened this session and all of which stay explicit, user-triggered next steps.
