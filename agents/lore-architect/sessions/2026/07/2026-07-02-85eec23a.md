# Built lr-wait — an on-demand wait/event primitive (v18)

The session opened with the user wanting a "wait primitive" for Claude Code agents: a way for an
agent to stop and wait for an event from the outside world, then continue when that event arrives
carrying information and instructions. The intended setting was a background or headless (`claude -p`)
run — a terminal the user doesn't watch continuously but can check. The architect framed this as the
missing *inbound* half of an agent's signal layer: existing lore covered the outbound direction
(agent → world via `Stop`/`Notification` hooks and the switchboard), and this was its mirror,
world → agent resume-on-event. It slotted directly into the parked autonomous-agents direction.

Design work was grounded in current Claude Code behavior rather than memory — the hooks and headless
docs were read, confirming that a `Stop` hook can block-and-continue but fires on *every* stop, and
that a local stdio MCP tool call can block for ~28h (`MCP_TOOL_TIMEOUT`) versus the ~10-minute
Bash-tool cap. Those facts drove two decisions: expose an explicit **tool the agent calls only when
told to wait** (not a Stop hook), backed by an **MCP stdio server** (not a blocking Bash command).
The transport became a **file inbox** — events are files, content plain text by default. The user
shaped the surface interactively: multiple consumption modes (`one` / `all` / by-name prefix), a
`sleep` tool, and wait-with-timeout as just a bounded `wait_for_event`.

The build produced `scripts/wait-server.py` (a pure-stdlib Python MCP server — the framework's first
bundled MCP server and first `python3` dependency), the bash `scripts/lr-emit` emitter, `.mcp.json`
registration, a thin `/lr:wait` skill and `docs/wait.md`, and the full v18 ship ceremony (VERSION 18,
manifests `1.18.0`, release notes with the cache-clear footer, three new `conventions.md` sections).
It was verified two ways: a stdlib smoke test driving the real server over JSON-RPC, and a live
headless `claude -p` run in which the agent parked inside `wait_for_event` and woke on an emitted
event — confirming the one open assumption that `-p` holds a foreground MCP call open.

At the user's request the change was then reviewed by three parallel read-only agents (correctness,
framework-fit, product/UX). They earned their keep: the correctness reviewer empirically reproduced a
**must-fix `lr-emit` infinite-loop hang** on a missing option value, and the set surfaced several
worthwhile robustness and clarity fixes — claim-first `consume` to prevent double-delivery across
sessions, a busy-error reply for a stray mid-wait request, a tool description that tells the agent to
*act on* event content and treat it as untrusted, the inbox cwd-mismatch footgun, and README/doctor/
trigger-list gaps. All were applied and the test suite grew to 23 (unit + integration), re-run green.
Two softer lessons also landed: the user's repeated, emphatic push for plain, example-first
explanations, and an architectural steer that **tests belong in the dev repo, not the plugin** —
which became the new Dev-Only Artifacts convention.

The session finalized with reflect + merge + summarize. Six reflection topics were integrated: two
new lore topics (`wait-primitive-feature`, `plugin-mcp-server-convention`), folds into the feedback,
reviewer-fanout, and plugin/agent-separation topics, and the critical `versioning-release-types.md`
backfill — the v18 history entry plus an extension of the cache-affecting *trigger definition* to
name bundled MCP servers. Crucially, per the user's instruction, **nothing was committed or pushed**:
the framework feature is built, reviewed, and tested but deliberately held. `lore-repo.md` was left
stamped at `17` (version-check restamps on the next boot after the ship lands).

Next steps: commit and push both repos when the user is ready — the framework ship is the gate, after
which the plugin cache should be cleared per the release notes. A handful of reviewer suggestions were
consciously deferred as low-value or scope-creep (a total-deadline stdin reader, protocolVersion echo,
a `.mcp.json` check in `/lr:check`, and peek/list tools).
