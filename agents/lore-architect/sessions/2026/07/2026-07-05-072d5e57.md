# Validated local Cursor port end to end

The session opened with a direct execution goal: stop treating the Cursor port as a design draft
and carry it through autonomously to a near-landing local build. The starting state was useful but
unfinished. The framework already had the Codex port shipped in canonical v19, a parked
`draft-port-cursor.md`, and a blocked first Cursor probe that had never gotten past account quota.
The user wanted the same treatment Codex had received: a separate local build, real engine
validation, tests, and a clear report of what was left before landing into canonical
`lore-framework`.

The work quickly became concrete. A sibling build, `lore-framework-cursor/`, was created from the
canonical framework and used as the staging ground. Real `cursor-agent` behavior was probed first:
`--plugin-dir` worked, slash skill invocation worked, `${CLAUDE_PLUGIN_ROOT}` was empty, and the
parent `cursor-agent` process was visible through `ps -o args= -p $PPID`, giving boot a strong
engine-detection signal. Those facts led to a deliberate design choice: instead of claiming an
unverified Cursor-native subagent mechanism, the local build got a conservative `docs/engines/cursor.md`
profile with a serial host-side override for the fan-out procedures. That kept the framework diff
small: one new engine profile plus targeted edits to `agent-boot.md`, `attach.md`, `init.md`, and
`resolve-conflicts.md`.

The interesting plot twist was that the risky part was not the framework docs themselves but the
operational edges of the real local engine. Cursor CLI auth inspection (`status`, `--list-models`)
needed to run outside the sandbox because of macOS keychain access, and the first repo/workspace
suite exposed a test mismatch rather than a framework bug: `init` correctly wrote `AGENTS.md` on
Cursor, while the lifecycle assertion still hardcoded `CLAUDE.md`. Fixing that in the harness was
the only red test in the whole run. The other important nuance was finalize scope: the harness code
lives under `tests/`, outside Lore finalize's `git add agents/` boundary, so the session could
validate and summarize the harness work but not sweep those test changes into the final lore-agent
commit automatically.

The empirical leg was the heart of the session. Cursor was wired into
`lore-framework-dev/tests/lifecycle/harness.py` via `cursor-agent -p`, JSON output parsing, engine
selection, and `/dev/null` stdin handling. The individual scenario groups passed first, then the
full implemented lifecycle catalog ran cleanly against the real local engine using
`composer-2.5-fast`: 19 tests, 19 passed, across boot, recall, consult, attach, reflect, merge,
summarize, finalize, create-repo, create-agent, init, workspace-sync, check, and update
`--dry-run`. That upgraded Cursor from "quota-blocked and unvalidated" to "validated locally as a
near-landing build." A validation report and execution-plan artifact were written into the
`lore-architect` workdir, and the agent's lore was updated with the new Cursor state.

Where it landed is strong but intentionally not overstated. The local `lore-framework-cursor/`
build is now a credible near-landing artifact, and the lore records that the conservative serial
profile is sufficient for the currently implemented Tier-1 lifecycle surface. What remains is
landing discipline rather than foundational uncertainty: copy the validated framework diff into
canonical `lore-framework/`, carry the harness changes in `lore-framework-dev/`, add release notes,
and rerun the Cursor suite against canonical after landing. Still deliberately deferred are
`lr-wait` / `.mcp.json`, DF/AIQA, migration-surface Cursor work, and any stronger claim about
Cursor-native parallel subagents or worktree interplay.
