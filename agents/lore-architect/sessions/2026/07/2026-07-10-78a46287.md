# Designed v25 Cursor ops parity and validated mid-session fallback

The session opened with the user asking how to install and automate Lore Framework plugins on
Cursor, and whether installation could be scoped to a workspace. The lore-architect agent (booted
via the file-driven Cursor workaround — no native `/lr-*` plugin loaded) walked through the
distinction between VS Code extensions, Cursor marketplace plugins, and Lore's local-checkout model.

The substantive work centered on the `zroslaw/lore-framework` repo and the user's existing symlink
at `~/.cursor/plugins/local/lore-framework`. The agent demonstrated booting `lore-architect` from
disk, then drafted a thorough **v25 Cursor ops parity** design in
`workdir/draft-v25-cursor-ops-parity.md`. Four independent lore-architect subagent review rounds
refined scope: two-step install model, canonical fallback via `.cursor-skills/` wrappers, atomic
doctor ailment, three-manifest `/lr:check` #19 extension, mandatory lifecycle harness as last gate,
and dropping unverified `workspaceOpen` hook templates.

A key plot twist was Round 3 catching that a single-script install claim was circular (users must
clone first), that `${CLAUDE_PLUGIN_ROOT}` is a false diagnostic on Cursor, and that check #19
never covered `.cursor-plugin/plugin.json` (explaining manifest lag at 1.23.0). Round 4 approved
the design with zero blocking issues.

The session landed with v25 **design-approved, implementation deferred** to a separate session.
No changes were made to the `lore-framework/` plugin repo. Lore was updated with design summary
topics and backlog pointers. Finalize committed locally; push withheld per user request.
