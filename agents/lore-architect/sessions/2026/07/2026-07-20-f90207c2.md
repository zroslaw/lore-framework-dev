# Built real-engine lifecycle scenarios for the Being Keeper

The session opened with a question about the recently-drafted Lore Beings / Being Keeper feature
(`lrb`, shipped locally as v28 BETA but not yet pushed): did it have any lifecycle test coverage,
and should it run across all engines? A quick audit found the answer was effectively no — the
framework's 27-scenario `tests/lifecycle/` catalog had zero entries for the Keeper. What existed
instead was a deterministic 83-test stub-engine suite (`test_lrb.py`, zero API cost, no real LLM
calls) and one standalone Cursor-only real e2e script, with no claude or codex equivalent.

The user asked to have a Fable-model subagent carefully design Keeper-specific lifecycle scenarios
before any implementation, then build and run them for all three engine kinds with the cheapest
viable models. The design subagent was briefed with hard constraints — sandbox the Keeper's real
machine home directory, guarantee teardown of any spawned process even on failure, skip cleanly on
missing engine auth — and, working from the actual `lrb.py` source rather than assumptions, found
that the required sandboxing already existed (`$LRB_HOME`/`$LRB_LAUNCHAGENTS_DIR` env overrides,
built in during the original Keeper build for exactly this purpose). It returned a 13-scenario
catalog with a 9-scenario recommended minimum, saved to `workdir/draft-lrb-lifecycle-tests.md`. The
user chose the recommended-minimum scope and explicitly opted to keep the cheapest model even for
the self-scheduling scenarios the design had flagged as higher-risk on a cheap model.

Implementation followed as `tests/lifecycle/keeper_harness.py` (fixture builder, teardown guard,
tick-loop driver) and `tests/lifecycle/test_lrb_lifecycle.py` (8 scenarios: core loop per engine
kind, real process-tree kill, self-scheduling round trip, self-scheduling denial under default
permissions, real PID-identity confirmed-match, and a real `lrb daemon` subprocess), gated behind a
new flag `LR_LIFECYCLE_KEEPER=1` kept deliberately separate from the framework's existing
`LR_LIFECYCLE=1` because Keeper scenarios can spawn a real background process, a strictly higher
blast-radius class. The plumbing was smoke-tested against the existing zero-cost stub engine before
any real money was spent.

Running all three engines for real surfaced several things. Two bugs turned out to be in the new
test code itself, not the Keeper: an E1 polling check matched the ledger's `"spawned"` marker
instead of waiting for an actually-finished outcome, and C1's self-scheduled one-shot task prompt
was ambiguous enough that a cheap model once mistook the Keeper's always-present self-scheduling
boilerplate for its actual instruction and rescheduled instead of answering — both were fixed and
retried successfully. Cursor was noisier: A3 hit real backend instability (`cursor-agent` losing
its connection to Cursor's own API relay twice, plus one empty-output crash) across four attempts
before landing a clean pass, with no evidence of a local or configuration cause (CLI already
current, no relevant flags, claude/codex ran fine concurrently). That one clean pass also surfaced
a genuine, more durable finding: real `cursor-agent` currently reports no `total_cost_usd` at all,
meaning a cursor-kind being's daily budget cap silently never trips unless `session_cost_usd` is
explicitly configured as a flat fallback — an asymmetry with codex, where that same fallback is
already mandatory. A second real finding, unrelated to flakiness, was that the claude engine kind
has no `--plugin-dir` configuration field at all in `lrb.py`, so a claude-kind being can only load
the framework's own skills via a wrapper script around the `command` — not something `docs/beings.md`
currently documents.

The session landed with all 8 recommended-minimum scenarios real-engine-verified (claude 6/6,
codex 1/1, cursor 1/1), the new test files and README documentation in place, and both `lrb.py`
findings filed as backlog items in `framework-improvements-backlog.md` rather than silently patched
into shipped code — they're user-facing schema decisions. The reflect/merge phases integrated five
reflection topics into existing lore (`cursor-agent-real-invocation-contract.md`,
`engine-kinds-design-decision.md`, `framework-improvements-backlog.md`, `lifecycle-testing-harness.md`,
`lore-beings-design.md`) plus one new topic on the spawn-prompt-distraction lesson, and refreshed
`lore-context.md`'s Lore Beings entry accordingly.

Open threads for next time: whether to make `--session-cost-usd` mandatory for the cursor engine
kind (or first re-verify the cost-blindness across other cursor models), whether to add a real
`--plugin-dir` field for the claude kind versus just documenting the wrapper workaround, and
whether to add a capped retry for cursor-backend connection flakiness in the test harness itself —
none of these were actioned this session, only proposed. The pre-existing, unrelated gate — the
full framework `LR_LIFECYCLE=1` suite still owed before the v28 push — remains untouched and open.
