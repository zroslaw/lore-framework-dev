## Soft Skills vs Hard Skills (concept — elaborate later)

- **Make "soft skills" an explicit framework entity.** User-raised 2026-06-03. The distinction: **hard skills** are shipped in separate repos and installed deliberately — "hard" because they're hard to change and control. **Soft skills** travel *with the lore agent itself* — behavioral/working-style guidance the agent carries and continuously updates when the user is unhappy or tells it explicitly. First concrete instance (the seed for this concept): "follow the user's thinking direction, don't over-think ahead or overtake the direction; suggest small improvements only." ## Preserve Claude Code Session JSONLs Alongside Session Metadata (parked 2026-06-05)

- **Archive raw Claude Code session JSONL files together with our session summaries; make them searchable and attachable.** User-raised 2026-06-05. Today's `sessions/YYYY/MM/<uuid>.md` is a *composed* summary — what the agent thought was worth preserving. The raw `~/.claude/projects/<encoded-cwd>/*.jsonl` files (full transcript: every user message, every tool call, every result, every reasoning trace) are far richer and live outside the agent repo, where they: (a) get cleaned up or rotated by Claude Code without our control, (b) aren't bound to our session UUID — only to Claude's own session id, (c) can't be searched alongside lore content. Proposal shape: at finalize (and at the new "call it a day" suspend, when that lands), **copy the raw JSONL(s) for the active Claude Code session into the agent repo** under a structured location, link them to the session UUID, and expose search/restore. Roughly: (1) **storage location** — `sessions/YYYY/MM/<uuid>.jsonl` sibling of the summary, or `sessions/YYYY/MM/<uuid>/transcript.jsonl` if there'll be more attachments later (logs, generated artifacts, reflection notes); (2) **resolution** — finalize already knows the session UUID it's writing; needs to also resolve "which JSONL files belong to *this* Claude Code session" — likely via the `--parent-session-id` known at boot for teammates, or via the canonical session-id file Claude Code writes (re-confirm by reading `jsonl-session-files-investigation.md`, which we already have); (3) **multi-agent capture** — `/lr:spawn-teammate` sessions: each teammate has its own JSONL; finalize on the lead-side captures the lead's, finalize on the teammate-side captures the teammate's, summary cross-links so a researcher can reconstruct the team conversation; (4) **search surface** — a `/lr:sessions-search <query>` skill that grep/JSONL-parses across archived transcripts, optionally scoped to a date range, an agent, or a session uuid; results return matching messages with their session UUID + summary pointer + line/turn anchor; (5) **archive policy** — JSONLs are large; consider compression (`.jsonl.gz`) for older sessions, possibly an age-out / cold-storage tier; (6) **attach to session feature** — a `/lr:session-attach <session-uuid>` that loads a past session's transcript into context for analysis ("what was I thinking when I designed X?"), composes with the call-it-a-day resume (suspend doc may itself be a thin pointer at the JSONL + a few prose highlights, not a hand-composed seed); (7) **privacy / size considerations** — transcripts may contain secrets, paths, internal links; consider a redaction pass option, repo-size impact (lots of large committed binary-ish files), and gitignore-vs-commit policy (probably commit by default for team-shared knowledge, with an opt-out); (8) **schema evolution** — Claude Code's JSONL schema can change between versions; archived transcripts must record the Claude Code version that produced them so search/parse code can handle multiple schemas. Composes with: `/lr:summarize` (the summary becomes the *index* for the transcript, pointing into specific turns), suspend/resume (the JSONL is the high-fidelity snapshot the suspend doc references), Agent Teams (per-teammate transcripts cross-linked by parent-session-id), code-review/research workflows (search past sessions for "have I solved this before?"). Already-relevant lore: **`jsonl-session-files-investigation.md`** documents what we currently know about the JSONL location/format — start there. Open seams the proposal doesn't yet decide: which finalize phase owns the copy (Phase 4 commit alongside the summary, or a new phase 3.5?), whether to also capture related artifacts (workflow scripts run, agent transcripts from spawned subagents), and whether transcript archival is mandatory for finalize or a per-session opt-in. Status: parked, captured as a deferred lifecycle/observability feature; pairs strongly with the call-it-a-day and boot-context-caching entries above (all three are "session as a first-class durable artifact" ideas). See `jsonl-session-files-investigation.md`, `session-summaries-feature.md`, `finalization-process.md`, `framework-improvements-backlog.md` § Call-It-a-Day, § Autonomous Agents.

## Call-It-a-Day — Suspend & Resume Unfinished Sessions (parked 2026-06-05)

- **"Call it a day" feature: preliminary finalization that lets you shut everything down without losing in-flight work.** User-raised 2026-06-05. The pain: at end-of-day there's often work not at a finalize-able boundary — a design exploration mid-iteration, an experiment partially run, an open backlog item being scoped. Today the choice is binary: either run `/lr:finalize` (which assumes the work is *done* — it commits, pushes, and writes a canonical session summary) or close Claude Code with everything in volatile context, losing the session shape on next boot. Proposal shape: a new lifecycle phase — **suspend** — that snapshots enough state to resume the *thread of thought* later without forcing closure. Roughly: (1) **mark the session as suspended-not-finalized** with a lightweight artifact (a `sessions/YYYY/MM/<uuid>.suspended.md` sibling of canonical session summaries — different schema, explicitly mid-flight); (2) **capture what's open** — current task list, draft files in `workdir/` that aren't committed, open design exploration topic, recent context highlights, "next time, do this" handoff; (3) **leave the working tree as-is** (no auto-commit of half-done work; the suspend doc records *pointers* to dirty files, not the files themselves) OR optionally commit-on-a-suspend-branch as an opt-in for users who want the snapshot durable; (4) **at next boot, surface unfinished sessions** as a one-screen dashboard — list of suspended sessions across active agents with one-line summaries + age + agent + "resume?" affordance; (5) **resume reads the suspend doc into context** — re-seeds the agent with the captured state so the conversation picks up where it left off, then deletes the suspend doc (or marks it consumed). Composes with: existing finalization (suspend → eventually `/lr:finalize` when the work *is* done; or suspend → discard if abandoned), auto-pull/push (suspend doc is per-agent-repo so it travels with the team — though may want a private/gitignored mode for solo work), Agent Teams (each teammate suspends independently; lead surfaces all teammates' open sessions on next team session). Open seams: (1) **suspend granularity** — one session at a time vs multiple parallel threads per agent (a long-running design exploration can span many sessions; suspend should support naming/tagging, not just one slot); (2) **resume into same agent vs cross-agent** — usually same-agent, but a suspended exploration may have been touched by attached guests whose state should also restore; (3) **storage location and gitignore policy** — `sessions/` is currently committed, but in-flight thoughts may want a separate `sessions-suspended/` opt-in policy; (4) **age-out / pruning** — stale suspend docs accumulate; rule for how long they stick around (manual prune? auto-archive after N days?); (5) **what survives, what doesn't** — Claude's actual context window is opaque; the suspend doc is best-effort reconstruction, must be authored knowing it's a *seed* not a *replay*; (6) **CLI surface** — `/lr:call-it-a-day` (or `/lr:suspend`) + `/lr:resume <session>` + `/lr:list-suspended`; (7) **interaction with `/lr:reflect`** — should suspend pre-run reflect to capture lore-worthy bits even though the session isn't done? Probably no — reflect-on-incomplete-work is a different shape than reflect-on-finished-work. Status: parked vision, no design doc yet — capture so it surfaces when end-of-day workflow friction becomes acute. The surrounding direction is the same family as Autonomous Agents (always-on background agents) — both are "what happens between sessions" features, but suspend/resume is for *interactive* sessions deliberately ending, where autonomy is for sessions that never end. See `finalization-process.md`, `session-summaries-feature.md`, `autonomous-agents-vision.md`, `framework-improvements-backlog.md` § Autonomous Agents.

## Agent Boot-Context Caching (parked 2026-06-05)

- **Pre-cache agent initialization context to make boot near-instant.** User-raised 2026-06-05. Today every boot reloads from scratch: `agent-boot.md` + `role.md` + `lore-context.md` + (now) auto-pull + version-check, all paid as input tokens on every session start. With multi-agent teams (`/lr:spawn-teammate`) and frequent re-boots during design iteration, this is one of the largest steady-state costs. Proposal shape: precompute a per-agent **boot-context cache** — the assembled prompt + loaded context — keyed by the inputs that would invalidate it (framework `VERSION`, agent's `role.md` content hash, `lore-context.md` content hash, possibly `agent-boot.md` + `teammate-conventions.md` content hashes for the universal pieces). Rebuild the cache on every framework update (`/lr:update` succeeds, plugin install) and on every agent edit (reflect/merge that touches `lore-context.md` or `role.md`). At boot, if cache key matches → use the cached context directly, skip the per-file Read calls. Open design seams: (1) **storage location** — per-agent (`agents/<name>/.boot-cache/`?), per-workspace, or in a framework-level cache dir; gitignored vs committed (committing keeps the team aligned; gitignoring keeps repos clean — likely gitignored, regenerated on first boot post-update); (2) **cache-key composition** — what counts as an input (just the read-aloud files? or also the universal docs they pull in like `teammate-conventions.md` on teammate boots?); (3) **invalidation triggers** — auto on framework version change, auto on lore merge, manual `/lr:cache-rebuild` for forced refresh; (4) **format** — raw text concatenation vs prebuilt prompt-cache-friendly structure (prefix-cacheable to leverage Anthropic's prompt caching, which would compound the savings — the *system* layer caches across requests, the boot-context cache keeps the *content* assembly cheap); (5) **interaction with `/lr:attach`** — guests' boot-contexts cacheable independently; (6) **detection of cache staleness vs runtime drift** — if `lore-context.md` was hand-edited without going through reflect/merge, the cache must invalidate cleanly; content-hash keying handles this. Likely a v17+ candidate; pairs naturally with prompt-caching support if/when the framework wires it in. Status: parked vision, no design doc yet; capture so it surfaces when boot performance becomes the felt bottleneck. See `freshness-contracts-at-session-boundaries.md` (the boundary the cache must respect), `agent-boot.md` (the procedure being cached), `auto-pull-mechanism.md` (one of the inputs the cache key must reflect).

## Boot-Time Auto-Commit + Auto-Push of Upgrades (parked 2026-06-05)

- **Auto-commit + auto-push the boot-time version upgrade.** User-raised 2026-06-05; designed to draft, then parked as too complicated to ship right now. The motivating failure: a lore agent repo can be locally upgraded (migrations applied, version stamped, committed) but never pushed — leaving the team unaware. Real instance: `activities-lore-agents` carrying unpushed `stamp version 13` + `stamp version 15`. Today's `version-check.md` Step 4 explicitly forbids commits ("user reviews and commits themselves"), which makes this orphan-commit state structurally available at every unattended boot. Proposal shape: after Step 3 (Stamp), add Step 4 (commit, scope = write-set ∪ `lore-repo.md`, never `git add -A`) and Step 5 (`git push --ff-only`, with content-equality reconciliation on push-rejection — fetch, compare remote write-set bytes against ours, on match `reset --soft HEAD~1` + `checkout origin/<branch> -- <write-set>`, on divergence keep local commit + warn). Same scoping applies to the reconciliation checkout (never full `git pull`). Boot must never fail — every commit/push/fetch failure path lands in degraded-mode boot with a warning. Asymmetry with `/lr:update` is preserved (boot is unattended → carries gate + auto-push; `/lr:update` is interactive → user commits/pushes themselves). Open seams: branch detection (`@{u}` resolution + no-upstream fallback), write-set grammar for migration-time deletions/renames so `git add -- <path>` picks them up, identity/network failure paths, ship as a procedural-doc release (cache-clear footer + manifest bump + `versioning-release-types.md` backfill + multi-round multi-lens review). Composes cleanly with the v13 auto-pull (which already defuses the common race). Full draft preserved in `workdir/draft-auto-push-after-upgrade.md` — resume from there. See `freshness-contracts-at-session-boundaries.md` (the principle, mirror of auto-pull on the *push* side), `dirty-tree-gates-write-vs-read-distinction.md` (the write-set discipline this extends), `auto-pull-mechanism.md` (model for graduated network-failure handling).

## Lore Housekeeping / Consolidation Job — the "sleep" pass (2026-06-13)

- **A periodic, agent-run deep-restructuring pass over an agent's own knowledge — distinct from per-session reflect/merge.** User-raised 2026-06-13. Per-session merge is *incremental and local* (integrate this session's reflections, preserve-don't-lose) and deliberately does not restructure — like waking working memory. Over many sessions the base drifts: `lore-context.md` bloats toward an index, topics over-granulate, the summary-topic mid-tier weakens, stale topics linger, cross-references rot. This job is the **consolidation force** — the analogue of what the brain does during sleep: stop encoding and reorganize long-term memory structurally. It is the *active mechanism* for the "no garbage-collector" gap (§ Architecture-Review Follow-Ups, subitem 3) and the periodic deep counterpart to the per-session `lore-context.md` shape discipline added to `process-merge.md` Step 4 this session.

  **Operations (one agent's lore at a time):** reshape `lore-context.md` to its intended shape (working-knowledge + summary-topic refs, no index, no history); consolidate over-granular / sibling topics and split oversized ones; summary-topic maintenance (create missing mid-tier summaries, prune redundant ones); staleness pass (delete superseded topics — delete-don't-mark); fix dangling cross-references (e.g. the known `contributions-feature.md` dead link); rename / rebalance for clarity.

  **Design seams (for the eventual design doc):**
  - **Distinct from `/lr:check`** — check *reports* content inconsistency; this *improves* structural quality (different axis). Check's warnings can seed a groom.
  - **Distinct from merge** — merge stays fast and incremental; don't fold this in. Merge may *suggest* "due for a groom"; it shouldn't run one inline. Separate skill.
  - **Execution** — booted-as-target subagent (the merge / consult boot-as-lens pattern), scoped per agent.
  - **Trigger / cadence** — manual skill first (`/lr:groom` / `/lr:consolidate` / `/lr:housekeeping` — name TBD); later signal- or schedule-driven (topic count, `lore-context.md` near budget, time-since-last-groom, accumulated `/lr:check` warnings). Natural fit for a per-agent background "night job" — ties to the autonomous-agents direction.
  - **Safety / blast radius** — high: it restructures and *deletes* team-shared knowledge. Wants a **preview / dry-run** that reports the proposed restructuring before applying (compose with the dry-run-summary discipline), a **grooming report** (what changed + why, before/after size), user review by default until trusted, and the merge "be conservative with deletions" guard applied more strongly. A big reorg is more push-conflict-prone than an incremental merge — consider a "groom when quiet" convention.

  Status: parked vision, captured the day the per-session shape discipline landed — they're the incremental / consolidation pair. Promote to its own lore topic + `workdir/` draft when active. See § Architecture-Review Follow-Ups (subitem 3), `process-merge.md` Step 4, `autonomous-agents-vision.md`, `consistency-checks.md`, `naming-foundational-principles.md`.
