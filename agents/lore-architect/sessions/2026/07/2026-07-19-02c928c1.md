# Designed Lore Beings: naming, Being Keeper, and Chronicler MVP

The session continued directly from the previous day's Agent Beings brainstorm — the
user re-booted lore-architect with the full prior-session transcript as context. The
agent took that as the go signal for the deferred design session and produced a first
draft (`workdir/draft-agent-beings.md`) promoting the backlog capture to a working
design: being descriptor schema, supervisor contract, outbox channel, autonomy levels,
and an MVP definition. The user then switched the session into dialogue and
plain-language modes, and the rest of the work happened as an extended design
conversation rather than document-first drafting.

The dialogue settled the architecture piece by piece. Python (stdlib-only, floor 3.9,
single file) was confirmed for the supervisor after weighing Node/TS, Go, and bash —
the decisive argument being that Python is present on every machine that can already
run the framework, since macOS ships it with the same Xcode Command Line Tools that
provide git. The supervisor's internals were fixed as a deliberately boring
single-threaded tick loop with sessions as OS subprocesses and an atomically-written
`state.json` holding only what cannot be recomputed. A key structural correction came
from the user's questions about multi-engine installs: the supervisor graduated from
per-workspace to machine-level — one daemon under launchd, installed to a stable home
(`~/.lore-beings/`) by an install command, serving a registry of workspaces — because
per-engine plugin directories are unstable homes for a daemon.

Naming took a long, productive detour. "Lore Beings" stuck early, but the supervisor's
name went through keeper/steward/warden (rejected: too personified for what must read
as unconscious substrate), body-metaphor names like Soma, and vegetative-nervous-system
names (Vagus, Medulla) before the user settled pragmatically on **Being Keeper** with
the **`lrb`** CLI and `lrb-*` skill prefix, following the DF module's name-prefix
precedent. Two further user decisions reshaped the original draft: existential-task
content moved entirely to the agent level (no framework-canonical morning/night
procedure docs — the framework injects only the runtime contract via the spawn
prompt), and the engine set became explicit user configuration (`lrb engines
add/remove`) rather than auto-detection, on the principle that a safety layer must not
change behavior because something unrelated got installed.

The main working-style plot twist: the agent twice moved to fold agreed decisions into
the draft mid-dialogue, and the user pushed back — "when we will be ready, I will tell
you." The correction was captured as a feedback lore topic: in dialogue-mode design
sessions, decisions live in conversation until the user explicitly triggers the draft.
The MVP also changed shape on the user's direction: instead of dogfooding on
lore-architect, the first being will be the **Chronicler**, a purpose-built
zero-blast-radius test being that keeps a workspace diary — with a single existential
task (morning) that self-schedules the midday and night sessions, deliberately making
the outbox path the main thing under test, on a light-tier model at ~$1/day.

The session landed with the user triggering the draft: `workdir/draft-lore-beings.md`
now holds the complete agreed design (superseding the deleted same-day
`draft-agent-beings.md`), with user decisions marked as such. Finalization merged two
reflections into lore — a new `lore-beings-design.md` anchor topic plus the
draft-gating feedback topic — and swept the autonomous-agents cluster (backlog,
substrate-split topic, vision topic, lore-context) to the settled naming and status.
Remaining open seams are listed in the draft: notification channel, failure policy,
collision hardening, canonical task templates, Linux/Windows, price tables. Next
step when the user is ready: build the Being Keeper MVP and let the Chronicler live
for a week.
