### Autonomous Agents / Lore Beings

- **Autonomous background agents** ÔÇö re-frame lore agents from session-bound interactive workers to always-on collaborators with persistent task state, asking for input only when needed. Implications across process model (daemonization), state model (task state, not just lore), communication model (agent Ôćĺ user channels), reflection model (autonomous reflect triggering), and multi-agent surface (switchboard for N parallel agents). Status: parked vision, no design doc yet. Concrete first step: `/lr:spawn-teammate` via Agent Teams. See `autonomous-agents-vision.md`.
- **Autonomous-agents substrate findings** ÔÇö concrete macOS building blocks identified: tmux for the process layer, iTerm2 Python API for GUI control, Claude Code hooks (`Stop`, `Notification`, etc.) as the signal source (strictly superior to regex triggers), escape sequences for status surface (tab color/title), `terminal-notifier` for click-actionable notifications, and a Python switchboard daemon under `launchd` for orchestration. Security: never bind `0.0.0.0` without auth ÔÇö RCE risk; prefer narrow HTTP API in front of the iTerm2 socket over raw socket exposure. Parked while Agent Teams experiment runs. See `autonomous-agents-substrate.md`.
- **Autonomy ladder ÔÇö six-step dependency order (2026-07-19 brainstorm).** The micromanagement pain decomposes into autonomy in *depth* (carry a task to completion), autonomy in *time* (exist without invocation), and autonomy in *organization* (teams/hierarchy, feedback-not-instructions). Build order:
  1. **Definition-of-done / quality-gate convention** ÔÇö a lore/`role.md` convention where each agent declares its definition of done, self-run quality gates, and decisions it makes without asking. Sharp framing: *every micromanagement instruction is lore that failed to be captured.* Cheapest, buildable now.
  2. **Feedback-to-lore tightening** ÔÇö make "user gave a correction" a first-class reflection trigger so micromanagement monotonically decreases; enables the manage-by-feedback model.
  3. **Autonomous finalization policy** ÔÇö gated self-reflection: agent writes reflections unattended; merge into team-shared lore on a reviewed cadence. Policy change from "user-triggered only"; prerequisite for any unattended existence.
  4. **Persistent task state** ÔÇö per-agent durable backlog (e.g. `workdir/tasks/`) surviving sessions, so any boot (scheduled or interactive) resumes standing work. The vision doc's flagged state-model gap.
  5. **The living loop** ÔÇö scheduled boots + `lr-wait` + task backlog = an agent that "lives." Budget caps and kill switch are part of this step, not an afterthought.
  6. **Teams & hierarchy** ÔÇö the org layer (delegation, escalation contracts, high-level tasking). Build last; multiplying agents that still need micromanaging multiplies the pain.
- **Lore Beings ÔÇö existential tasks + Being Keeper (formerly "Agent Beings"; DESIGN SETTLED 2026-07-19).** User-articulated elaboration of autonomy-in-time: an agent is a **being** with a declared lifecycle of **existential tasks** ÔÇö e.g. *morning wakeup* (boot, pull, review backlog, plan the day, schedule intraday work sessions), *night sleep* (reflect, finalize, consolidate ÔÇö converges with the "sleep pass" ┬ž Knowledge Quality), *weekly self-reflection* (long-range self-improvement). The **Being Keeper** ÔÇö deterministic daemon, not LLM ÔÇö is the "unconscious substrate / operating system": reads per-agent **`being.md` descriptors** (existential task list, schedules, budgets, autonomy level), spawns headless engine sessions on schedule, monitors status/logs/token spend, enforces budget caps, and accepts **schedule requests from beings** (file-outbox channel, symmetric to `lr-emit`) so a being can schedule its own future sessions. `lr-wait` remains the *synchronous, in-session* wait primitive; the Keeper owns *between-session* existence. Status: **design settled in a 2026-07-19 design dialogue** ÔÇö full agreed design in `workdir/draft-lore-beings.md` (superseded and replaced `draft-agent-beings.md`); anchor topic `lore-beings-design.md`; nothing shipped. Next step: MVP build ("the Chronicler lives for a week"). ### Call-It-a-Day ÔÇö Suspend & Resume Unfinished Sessions (parked 2026-06-05)

- **"Call it a day" feature: preliminary finalization that lets you shut everything down without losing in-flight work.** User-raised 2026-06-05. The pain: at end-of-day there's often work not at a finalize-able boundary ÔÇö a design exploration mid-iteration, an experiment partially run, an open backlog item being scoped. Today the choice is binary: either run `/lr:finalize` (which assumes the work is *done* ÔÇö it commits, pushes, and writes a canonical session summary) or close Claude Code with everything in volatile context, losing the session shape on next boot. Proposal shape: a new lifecycle phase ÔÇö **suspend** ÔÇö that snapshots enough state to resume the *thread of thought* later without forcing closure. Roughly: (1) **mark the session as suspended-not-finalized** with a lightweight artifact (a `sessions/YYYY/MM/<uuid>.suspended.md` sibling of canonical session summaries ÔÇö different schema, explicitly mid-flight); (2) **capture what's open** ÔÇö current task list, draft files in `workdir/` that aren't committed, open design exploration topic, recent context highlights, "next time, do this" handoff; (3) **leave the working tree as-is** (no auto-commit of half-done work; the suspend doc records *pointers* to dirty files, not the files themselves) OR optionally commit-on-a-suspend-branch as an opt-in for users who want the snapshot durable; (4) **at next boot, surface unfinished sessions** as a one-screen dashboard ÔÇö list of suspended sessions across active agents with one-line summaries + age + agent + "resume?" affordance; (5) **resume reads the suspend doc into context** ÔÇö re-seeds the agent with the captured state so the conversation picks up where it left off, then deletes the suspend doc (or marks it consumed). Composes with: existing finalization (suspend Ôćĺ eventually `/lr:finalize` when the work *is* done; or suspend Ôćĺ discard if abandoned), auto-pull/push (suspend doc is per-agent-repo so it travels with the team ÔÇö though may want a private/gitignored mode for solo work), Agent Teams (each teammate suspends independently; lead surfaces all teammates' open sessions on next team session). Open seams: (1) **suspend granularity** ÔÇö one session at a time vs multiple parallel threads per agent (a long-running design exploration can span many sessions; suspend should support naming/tagging, not just one slot); (2) **resume into same agent vs cross-agent** ÔÇö usually same-agent, but a suspended exploration may have been touched by attached guests whose state should also restore; (3) **storage location and gitignore policy** ÔÇö `sessions/` is currently committed, but in-flight thoughts may want a separate `sessions-suspended/` opt-in policy; (4) **age-out / pruning** ÔÇö stale suspend docs accumulate; rule for how long they stick around (manual prune? auto-archive after N days?); (5) **what survives, what doesn't** ÔÇö Claude's actual context window is opaque; the suspend doc is best-effort reconstruction, must be authored knowing it's a *seed* not a *replay*; (6) **CLI surface** ÔÇö `/lr:call-it-a-day` (or `/lr:suspend`) + `/lr:resume <session>` + `/lr:list-suspended`; (7) **interaction with `/lr:reflect`** ÔÇö should suspend pre-run reflect to capture lore-worthy bits even though the session isn't done? Probably no ÔÇö reflect-on-incomplete-work is a different shape than reflect-on-finished-work. Status: parked vision, no design doc yet ÔÇö capture so it surfaces when end-of-day workflow friction becomes acute. The surrounding direction is the same family as Autonomous Agents (always-on background agents) ÔÇö both are "what happens between sessions" features, but suspend/resume is for *interactive* sessions deliberately ending, where autonomy is for sessions that never end. See `finalization-process.md`, `session-summaries-feature.md`, `autonomous-agents-vision.md`, `framework-improvements-backlog.md` ┬ž Autonomous Agents.

### Preserve Claude Code Session JSONLs Alongside Session Metadata (parked 2026-06-05)

- **Archive raw Claude Code session JSONL files together with our session summaries; make them searchable and attachable.** User-raised 2026-06-05. Today's `sessions/YYYY/MM/<uuid>.md` is a *composed* summary ÔÇö what the agent thought was worth preserving. The raw `~/.claude/projects/<encoded-cwd>/*.jsonl` files (full transcript: every user message, every tool call, every result, every reasoning trace) are far richer and live outside the agent repo, where they: (a) get cleaned up or rotated by Claude Code without our control, (b) aren't bound to our session UUID ÔÇö only to Claude's own session id, (c) can't be searched alongside lore content. Proposal shape: at finalize (and at the new "call it a day" suspend, when that lands), **copy the raw JSONL(s) for the active Claude Code session into the agent repo** under a structured location, link them to the session UUID, and expose search/restore. Roughly: (1) **storage location** ÔÇö `sessions/YYYY/MM/<uuid>.jsonl` sibling of the summary, or `sessions/YYYY/MM/<uuid>/transcript.jsonl` if there'll be more attachments later (logs, generated artifacts, reflection notes); (2) **resolution** ÔÇö finalize already knows the session UUID it's writing; needs to also resolve "which JSONL files belong to *this* Claude Code session" ÔÇö likely via the `--parent-session-id` known at boot for teammates, or via the canonical session-id file Claude Code writes (re-confirm by reading `jsonl-session-files-investigation.md`, which we already have); (3) **multi-agent capture** ÔÇö `/lr:spawn-teammate` sessions: each teammate has its own JSONL; finalize on the lead-side captures the lead's, finalize on the teammate-side captures the teammate's, summary cross-links so a researcher can reconstruct the team conversation; (4) **search surface** ÔÇö a `/lr:sessions-search <query>` skill that grep/JSONL-parses across archived transcripts, optionally scoped to a date range, an agent, or a session uuid; results return matching messages with their session UUID + summary pointer + line/turn anchor; (5) **archive policy** ÔÇö JSONLs are large; consider compression (`.jsonl.gz`) for older sessions, possibly an age-out / cold-storage tier; (6) **attach to session feature** ÔÇö a `/lr:session-attach <session-uuid>` that loads a past session's transcript into context for analysis ("what was I thinking when I designed X?"), composes with the call-it-a-day resume (suspend doc may itself be a thin pointer at the JSONL + a few prose highlights, not a hand-composed seed); (7) **privacy / size considerations** ÔÇö transcripts may contain secrets, paths, internal links; consider a redaction pass option, repo-size impact (lots of large committed binary-ish files), and gitignore-vs-commit policy (probably commit by default for team-shared knowledge, with an opt-out); (8) **schema evolution** ÔÇö Claude Code's JSONL schema can change between versions; archived transcripts must record the Claude Code version that produced them so search/parse code can handle multiple schemas. Composes with: `/lr:summarize` (the summary becomes the *index* for the transcript, pointing into specific turns), suspend/resume (the JSONL is the high-fidelity snapshot the suspend doc references), Agent Teams (per-teammate transcripts cross-linked by parent-session-id), code-review/research workflows (search past sessions for "have I solved this before?"). Already-relevant lore: **`jsonl-session-files-investigation.md`** documents what we currently know about the JSONL location/format ÔÇö start there. Open seams the proposal doesn't yet decide: which finalize phase owns the copy (Phase 4 commit alongside the summary, or a new phase 3.5?), whether to also capture related artifacts (workflow scripts run, agent transcripts from spawned subagents), and whether transcript archival is mandatory for finalize or a per-session opt-in. Status: parked, captured as a deferred lifecycle/observability feature; pairs strongly with the call-it-a-day and boot-context-caching entries above (all three are "session as a first-class durable artifact" ideas). See `jsonl-session-files-investigation.md`, `session-summaries-feature.md`, `finalization-process.md`, `framework-improvements-backlog.md` ┬ž Call-It-a-Day, ┬ž Autonomous Agents.

### Agent Boot-Context Caching (parked 2026-06-05)

- **Pre-cache agent initialization context to make boot near-instant.** User-raised 2026-06-05. Today every boot reloads from scratch: `agent-boot.md` + `role.md` + `lore-context.md` + (now) auto-pull + version-check, all paid as input tokens on every session start. With multi-agent teams (`/lr:spawn-teammate`) and frequent re-boots during design iteration, this is one of the largest steady-state costs. Proposal shape: precompute a per-agent **boot-context cache** ÔÇö the assembled prompt + loaded context ÔÇö keyed by the inputs that would invalidate it (framework `VERSION`, agent's `role.md` content hash, `lore-context.md` content hash, possibly `agent-boot.md` + `teammate-conventions.md` content hashes for the universal pieces). Rebuild the cache on every framework update (`/lr:update` succeeds, plugin install) and on every agent edit (reflect/merge that touches `lore-context.md` or `role.md`). At boot, if cache key matches Ôćĺ use the cached context directly, skip the per-file Read calls. ## Knowledge Quality & Curation

### Lore Housekeeping / Consolidation Job ÔÇö the "sleep" pass (2026-06-13)

- **A periodic, agent-run deep-restructuring pass over an agent's own knowledge ÔÇö distinct from per-session reflect/merge.** User-raised 2026-06-13. Per-session merge is *incremental and local* (integrate this session's reflections, preserve-don't-lose) and deliberately does not restructure ÔÇö like waking working memory. Over many sessions the base drifts: `lore-context.md` bloats toward an index, topics over-granulate, the summary-topic mid-tier weakens, stale topics linger, cross-references rot. This job is the **consolidation force** ÔÇö the analogue of what the brain does during sleep: stop encoding and reorganize long-term memory structurally. It is the *active mechanism* for the "no garbage-collector" gap (┬ž Architecture-Review Follow-Ups, subitem 3) and the periodic deep counterpart to the per-session `lore-context.md` shape discipline added to `process-merge.md` Step 4 this session.

  **Operations (one agent's lore at a time):** reshape `lore-context.md` to its intended shape (working-knowledge + summary-topic refs, no index, no history); consolidate over-granular / sibling topics and split oversized ones; summary-topic maintenance (create missing mid-tier summaries, prune redundant ones); staleness pass (delete superseded topics ÔÇö delete-don't-mark); fix dangling cross-references (e.g. the known `contributions-feature.md` dead link); rename / rebalance for clarity.

  **Design seams (for the eventual design doc):**
  - **Distinct from `/lr:check`** ÔÇö check *reports* content inconsistency; this *improves* structural quality (different axis). Check's warnings can seed a groom.
  - **Distinct from merge** ÔÇö merge stays fast and incremental; don't fold this in. Merge may *suggest* "due for a groom"; it shouldn't run one inline. Separate skill.
  - **Execution** ÔÇö booted-as-target subagent (the merge / consult boot-as-lens pattern), scoped per agent.
  - **Trigger / cadence** ÔÇö manual skill first (`/lr:groom` / `/lr:consolidate` / `/lr:housekeeping` ÔÇö name TBD); later signal- or schedule-driven (topic count, `lore-context.md` near budget, time-since-last-groom, accumulated `/lr:check` warnings). Natural fit for a per-agent background "night job" ÔÇö ties to the autonomous-agents direction; the settled Lore Beings design (┬ž Autonomous Agents / Lore Beings, `lore-beings-design.md`) names this as the *night sleep* existential task.
  - **Safety / blast radius** ÔÇö high: it restructures and *deletes* team-shared knowledge. Wants a **preview / dry-run** that reports the proposed restructuring before applying (compose with the dry-run-summary discipline), a **grooming report** (what changed + why, before/after size), user review by default until trusted, and the merge "be conservative with deletions" guard applied more strongly. A big reorg is more push-conflict-prone than an incremental merge ÔÇö consider a "groom when quiet" convention.

  Status: parked vision, captured the day the per-session shape discipline landed ÔÇö they're the incremental / consolidation pair. Promote to its own lore topic + `workdir/` draft when active. See ┬ž Architecture-Review Follow-Ups (subitem 3), `process-merge.md` Step 4, `autonomous-agents-vision.md`, `consistency-checks.md`, `naming-foundational-principles.md`.

### Merge Quality (parked, 2026-07-02)

- **Post-merge diff verification** ÔÇö an outside architecture review (2026-07-02) flagged merge as the framework's highest-risk operation with no automated safety net: a bad `lore-context.md` compaction, or a reflection that silently overwrites/contradicts an existing topic, is currently only caught if a human happens to read the `git diff`.
