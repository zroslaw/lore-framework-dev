# Captured autonomous-agents direction as parked exploration

The session opened with an open-ended brainstorm question rather than a framework task: the user wanted to think through how to manage many simultaneously open macOS terminals, receive notifications from them, and switch focus to whichever one needed attention. The starting framing was generic terminal management — APIs available, what could be wired up — without an explicit tie to lore agents.

The conversation walked through the macOS surface area in sequence: AppleScript and `osascript` as universal scripting glue, iTerm2's two scripting surfaces (the AppleScript dictionary versus the richer Python API exposed as a local WebSocket on a Unix domain socket), Claude Code's own hook system as a structured signal source preferable to regex output triggers, OSC escape sequences for tab colors and titles, the `terminal-notifier` click-handler trick for "click notification → focus that session," tmux as the persistence layer letting sessions survive viewport closure, and finally the architectural sketch of a long-running daemon (Python `iterm2` library plus FastAPI co-running on one asyncio loop) exposing a narrow localhost API in front of iTerm2's broad WebSocket. Security boundaries were noted along the way — never expose iTerm2's raw socket externally, always front it with a token-checked HTTP layer if remote access is needed.

The plot twist landed near the end: when reflection was first attempted, the conversation looked tangential to lore-architect's role and the host reported nothing in scope, declining to write anything. The user then re-framed the session — explicitly stating their vision that lore agents should *become* autonomous background workers, raising for user input only when needed, and that this conversation's findings should be preserved as substrate research for that direction. That single message converted the entire brainstorm from off-topic to directly load-bearing for the framework's future.

Reflection re-ran with the new framing and produced two topics: `autonomous-agents-vision.md` capturing the architectural vision and its implications (process model, state model, agent-to-user communication channels, autonomous reflection, multi-agent surface assumptions that need rethinking), and `autonomous-agents-substrate.md` cataloguing the concrete macOS building blocks discovered. Merge promoted both into `lore/`, added an "Autonomous lore agents" entry at the top of `lore-context.md`'s "Active Design Explorations" section as a sibling to workdir-as-reference-library and vector-DB-search, and added a major-direction section to `framework-improvements-backlog.md`. No `role.md` change — the role still stands; this is a future capability, not a redefinition.

What's deferred: everything actionable. There is no design doc yet, no skill prototype, no decision on whether the runtime is local Claude Code under tmux or hosted Managed Agents. Open threads worth flagging for next time: where does an autonomous agent live (tmux session, `launchd`, cloud runner); how does the framework express "this agent is autonomous" (per-agent flag in `role.md` frontmatter, per-invocation); what does adoption of a running autonomous agent into an interactive session look like (extend `/lr:attach`, new `/lr:resume`); concurrency semantics when two user sessions interact with the same agent; cost/loop-safety guardrails since an autonomous agent can burn budget silently. The framework should not try to act on this until current direction stabilizes — v9 finalization changes are still recent.
