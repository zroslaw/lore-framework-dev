# Ran the first real Codex smoke test against the lore framework, found Cursor blocked on quota

The session opened with the user asking whether enough groundwork existed to actually start porting the lore framework to Codex or Cursor, following up on the prior session's parked port drafts and the lifecycle testing harness. The architect checked `multi-engine-portability-direction.md` and `lifecycle-testing-harness.md` and gave a direct answer: no — the harness had only ever run against Claude Code, so the core open risk (whether the framework's prose-driven procedures execute faithfully on a different model/engine) remained purely theoretical. The recommendation was to point the harness at a real engine before treating either port as unblocked.

The user agreed to proceed and reported that both the `codex` and `cursor-agent` CLIs were already installed locally. After sourcing the shell profile to refresh the PATH cache, the architect confirmed both binaries resolved and were authenticated, then compared their plugin-loading interfaces: `cursor-agent` ships a `--plugin-dir <path>` flag with the same name and per-invocation semantics as Claude Code's own, making it the lower-friction target on paper. `codex` has no equivalent flag — plugins load via a persistent local install (`codex plugin marketplace add <path>` to register a local marketplace, then `codex plugin add <name>@<marketplace>` to install from it). A first attempt to smoke-test cursor failed immediately with an account-level usage-limit error, unrelated to the CLI integration itself, so the user chose to proceed with codex first and revisit cursor once the quota issue clears.

Registering `lore-framework`'s existing `.claude-plugin/marketplace.json` (written for Claude Code) as a local Codex marketplace worked without any adaptation — Codex parsed it natively and installed the `lr` plugin with all 26 skill directories intact, concrete evidence for the framework's "packaging, not redesign" thesis. The first live boot smoke test then hit a scare: a foreground `codex exec` call, bounded by a hard wall-clock timeout, was killed after producing zero bytes of output, and the user reasonably asked whether it should just be considered broken. Rerunning the identical command backgrounded (`nohup ... & disown`) and polling instead showed it had been working correctly the whole time — the earlier kill had simply destroyed a buffered JSONL stream before it flushed. Along the way, the run surfaced a genuine, previously-predicted gap: `${CLAUDE_PLUGIN_ROOT}` resolves to an empty string under Codex, though the model quietly recovered by locating the installed plugin's doc file directly rather than failing outright.

With that lesson applied, the boot scenario ran clean: both the role and lore-context canary codewords printed correctly, auto-pull and the version check both executed and reported accurately, and the fixture repo was left with no new commits and a clean working tree — a full pass on the equivalent of the harness's `test_01_boot_happy_path`. This was a manual probe, not yet a change to `tests/lifecycle/harness.py` itself; wiring an actual `codex` branch into `run_engine()` (including the one-time marketplace/install setup) and running the full six-scenario `test_boot.py` suite against it remains open. Separately, and unrelated to the engine work, the user asked the architect to hold a long wait — a sequence of `lr-wait` sleep calls in ~59-minute chunks — purely to keep the Claude Code prompt cache warm until a fixed clock time, which the architect ran to completion using minimal intervening output.

The session closed with finalization: reflection produced three topics (codex plugin-loading findings, the blocked cursor probe, and the general headless-CLI-backgrounding lesson), and merge folded these into lore — creating three new topics, updating `multi-engine-portability-direction.md` and `lifecycle-testing-harness.md` to reflect the shift from "not yet started" to "first empirical probe complete for one engine," and refreshing `lore-context.md`'s portability bullet (topic count 96 → 99). No role changes were needed. Nothing was committed until this finalize step.

Next steps: add the `codex` branch to `harness.py`'s `run_engine()` and run the full Tier-1 `test_boot.py` suite against it; once Cursor's account usage limit resets or is upgraded, repeat the same single-scenario smoke test for `cursor-agent` before deciding whether it needs the same persistent-install treatment codex does.
