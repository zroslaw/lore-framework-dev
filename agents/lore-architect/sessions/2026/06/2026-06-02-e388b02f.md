# Shipped lore-framework v15: write-aware auto-upgrade gate, spawn-teammate verified, teammate input-direction anchored at boot

The session opened with a spawned-teammate context — the user had launched the architect via Agent Teams as a parallel pane, with three tasks queued by the team-lead activities-domain agent: a boot-time auto-upgrade overfiring on every dirty file, `/lr:spawn-teammate` consistently failing on first attempt, and an iTerm2 keystroke-corruption race during pane focus shifts. A fourth task arrived shortly after: spawned teammates routinely message the lead instead of the user despite the spawn prompt explicitly forbidding it. Each task came with a concrete real-use trace, including the literal `pt acd /Users/...` corruption that motivated task #3.

The substantive work spanned four design decisions and seven multi-lens review rounds. Task 1 reshaped `docs/version-check.md` Step 1 from a blanket dirty-tree gate into a write-aware collision check: each migration declares its `## Write Paths`, the gate intersects against `git status --porcelain`, and only real collisions defer; an addendum case calls out the previously-deferred-stamp pattern with tailored commit-or-revert guidance. Task 2 rewrote `docs/spawn-teammate.md` Step 7 to make the operative call explicit — the current session IS the lead, call `Agent` directly with `team_name`/`name`/`subagent_type`/`prompt`, do not delegate to a subagent that doesn't have the `Agent` tool. Task 3 added Step 7c post-spawn verification reading `~/.claude/teams/<team>/config.json`, with retry-tolerant config-write race handling and four-state outcomes (verified-live / verified-inconclusive / unverified / spawn-errored). Task 4 introduced `docs/teammate-conventions.md` as the single canonical source for the four standing RULES, anchored on every teammate boot via a new Step 5 in `agent-boot.md` that detects spawn context via `ps -o args= -p $PPID`. Adjacent conventions were codified in `docs/conventions.md`: § Migration Write Paths (with Glob token grammar and Empty write-set sentinel forms as canonical sub-specs), § Teammate Discipline as a maintainer's index, and § Known gap: workspace-root paths acknowledging the per-repo gate's blind spot. Migrations 2/5/6 were backfilled with Write Paths sections; `/lr:check` gained check #20 with three substeps; the plugin manifests bumped 1.14.0 → 1.15.0; release-notes/15.md was authored with the cache-clear footer hoisted near the top per the v14 placement fix.

The plot twists were almost entirely review-driven. A first code-review pass surfaced twelve findings; a second medium-effort pass found eight more, including a release-notes self-contradiction (Summary bullet 4 claimed the spawn prompt was "rewritten to a numbered RULE format" while a later paragraph said the RULES were "collapsed to a single canonical site"). A multi-lens parallel review (UX / architectural-consistency / correctness-with-filesystem-verification, three concurrent reviewers each booted as the architect for role-as-perspective) caught real bugs the prose passes missed — most notably that `docs/version-check.md` Step 1b's parser couldn't read its own migrations because the on-disk bodies are wrapped in Markdown code fences while the parser spec required bare-line bodies. The user explicitly asked for "almost perfect, ready to ship," and seven review rounds followed: each round caught real defects clustered around the previous round's edits ("fix-the-fixes surface"), each round shrank the finding count, round 7 finally returned zero. A recurring drift pattern emerged across rounds: "pointer-AND-restatement" — a fix declares one site canonical, but other sites still inline the rule "for clarity," recreating the very drift the canonical declaration was supposed to eliminate. Rounds 5 and 6 specifically tightened version-check.md Step 1b.2 and check.md #20.3 to be pointer-only, with the canonical Glob token grammar and Empty write-set sentinel forms living in conventions.md as the single source of truth.

The session landed with v15 committed (`3749109`) and pushed to `github.com:zroslaw/lore-framework`. The commit covers 13 files: VERSION + both manifests + 6 modified docs + 3 backfilled migrations + 1 new doc (`teammate-conventions.md`) + 1 new release-notes file. The auto-upgrade gate, the spawn flow, and the boot-time teammate detection are all procedural — no migration script needed; existing v14 repos auto-stamp to v15 on the next boot via the routine version-check flow. Round-7 confirmed ship-ready with no TODO/FIXME residue and aligned state names across spawn-teammate.md Step 7c and Step 8.

Open threads carry forward. A `/lr:check` rule for missing Write Paths shipped as #20 within v15 (don't-defer-completable-scope discipline applied), but bringing the write-aware gate to `/lr:update` is deferred — `/lr:update` is interactive-by-design and the user can `git diff` before running, so the asymmetry is intentional rather than a backlog item. The framework's third instance of the inconclusive-state pattern (spawn-teammate Step 7c, after parallel-reviewer-fanout-pattern's graceful-degradation and auto-pull's transport-aware ladder) was promoted to a new lore topic `graduated-verification-confidence.md` during merge, with cross-references to its prior instances. The pointer-not-restatement rule observed across the seven review rounds was promoted to `single-canonical-source-discipline.md` as the negative form of `shared-procedure-doc-pattern.md`. The minimum-blast-radius-refusal framing — the unifying principle behind v13's read-vs-write split and v15's collision check — was folded as a § into `dirty-tree-gates-write-vs-read-distinction.md` rather than promoted as a standalone topic, keeping the v13 and v15 instances co-located. The role's § Lore-Curation Disciplines gained a sixth bullet codifying multi-round multi-lens pre-ship review until convergence as the default for any release that touches procedural docs.
