Core design principles for the lore system:

- **Team-shared knowledge (foundational)** — lore agents are team-shared knowledge containers, not personal notebooks. Sessions, lore topics, and workdir artifacts live in shared git repos; multi-contributor finalization is expected and supported by push-conflict-resolution, merge-in-booted-subagents, and auto-push. Frames every other design choice — when in doubt, ask "what if multiple people use this simultaneously?" and design for that. See `team-shared-knowledge-principle.md`.
- **Directory-driven** — the file system is the source of truth. No external registries. A lore agent repo is identified by the presence of `lore-repo.md` at its root. An agent exists because its directory exists under `agents/`. Command files are the implicit registration registry. Co-location is the rule: descriptors live with what they describe.
- **Plain markdown, minimal frontmatter** — lore *topics* have no frontmatter; git provides all metadata (dates, history, authorship). The exception is *descriptor files* (`lore-repo.md` and `role.md`), which carry YAML frontmatter for `description` and `version` — fields the framework needs to machine-read for discovery and version checking. Frontmatter is appropriate when a file *describes* something the system must identify or validate; inappropriate for knowledge topics where git is the metadata layer.
- **Delete, don't mark** — obsolete topics are removed. Git preserves the full history.
- **Knowledge graph** — topics reference each other by filename. Summary topics act as entry points.
- **Minimal and essential** — topics contain only what matters. No filler, no general knowledge.
- **Git as memory** — all temporal metadata comes from git history.
- **Decoupled framework** — the framework (how agents work) is separate from agent definitions (what agents know). They ship as independent repos.
- **Agent-driven operations** — no shell scripts for agent management. All setup, registration, and management is done by Claude following markdown instructions. (Exception: pure mechanical operations like `workspace-sync` that require no judgment use shell scripts delegated from skills.)
- **Workspace visibility** — agents operate from a workspace root and can access all sibling repos and data, not just their own agent repo.
- **Concise context, detailed docs** — skills are brief; detailed instructions live in `docs/` and are read on demand. This prevents context bloat.
- **Explicit versioning (repo-level only)** — `lore-framework/VERSION` is the single source of truth for the framework version. Each lore agent repo stamps this version in `lore-repo.md` frontmatter at creation, and `/lr:update` advances it after applying migration docs. Agents do not carry their own version stamp — they migrate together with the repo. Version drift is caught by `/lr:check` and reconciled by `/lr:update`. See `update-process.md`.
- **Skill/doc separation** — skills are thin one-line pointers to docs; all operational logic lives in `${CLAUDE_PLUGIN_ROOT}/docs/<name>.md`. Keeps skill files minimal, docs authoritative, and logic in one place. Generated artifacts (e.g., registered boot commands) follow the same pattern — they delegate to a doc rather than inlining logic. See `skill-doc-pattern.md`.
- **Framework-scope vs agent-scope** — the framework owns what is universal across all repos, hosts, and workflows; agents own what is specific to a repo, host, team, or workflow. When designing a new feature, isolate the truly universal core; push everything else into agent lore (host's own or a specialist reached via `/lr:consult` / `/lr:attach`). Diagnostic signals of overreach: command proliferation, schema lock-in, host-mapping docs in the framework, hard gates around repo-specific workflow steps. See `framework-scope-vs-agent-scope.md`.
- **Name foundational principles explicitly (meta-rule for lore curation)** — when a principle is shared by multiple lore topics, name it as its own topic rather than leaving it implicit in the mechanisms it motivates. Otherwise future-self (or a teammate) has to reverse-engineer it under context pressure. The `team-shared-knowledge-principle.md` topic is the canonical example: it was implicit across the multi-author finalization machinery for many versions before being named. See `naming-foundational-principles.md`.
- **Cache-clear-footer convention (v12+)** — release notes / migration docs whose changes touch plugin-cached state (skills, scripts, SKILL.md-referenced docs) must include the **Clear Plugin Cache** footer with concrete commands, hoisted near the top of the document. The cache-affecting axis is orthogonal to the migration-vs-release-notes axis; track it per-version in `versioning-release-types.md`. See `cache-clear-footer-convention.md`.
- **Freshness contracts at session-context boundaries (v13+)** — whenever an agent's working memory is being newly seeded from disk (boot, attach, pre-merge), refresh disk to match the team's latest pushed state. Automatic at boundaries, manual override via `/lr:pull-lore`. The principle that v13's auto-pull mechanism operationalizes; applies to any future framework feature whose correctness depends on reads seeing fresh team state. See `freshness-contracts-at-session-boundaries.md`.
- **Safety-gate discipline: gate writes, not reads (v13+)** — when adding a "skip if working tree is dirty" gate to a framework operation, articulate the collision scenario explicitly. If the operation only advances refs / is naturally idempotent / git handles the conflict cleanly itself, no gate — it becomes friction without a safety benefit. v13's `auto-pull.md` deliberately has no gate (read-y); `version-check.md` does (write-y). See `dirty-tree-gates-write-vs-read-distinction.md`.
- **Shared-procedure-doc pattern (v13+)** — when a non-skill behavior must fire from multiple sites (e.g., auto-pull from boot + attach + pre-merge + `/lr:pull-lore`), factor the procedure into its own `docs/<procedure>.md` with an audience-note banner and have every call site delegate to it. A refinement of skill-doc separation, applied to procedures rather than skills. See `shared-procedure-doc-pattern.md`.
