# Renamed the codex to the DF repo and designed ULA for repeated runs

The session resumed the active lr-dev / SDLC design thread. The user opened by checking whether the prior context — AIQA, the codex repo, ULA, lr-dev — was still in hand; after confirming recall from lore, they switched the session into **follow-me mode** (user drives the thinking, the agent keeps suggestions small and writes no durable artifacts unprompted). The whole session ran as a conceptual design-exploration: no plugin or code changes were made, only design decisions, captured to lore at finalization.

The first and largest decision was a **rename and a reframe**. The per-repo artifact store — introduced as `<repo>-codex` on 2026-06-03 — became **`<repo>-df`**, where DF stands for **Dark Factory**. The user's rationale was that the name should anchor to the mission rather than read as a generic "book of knowledge." More importantly, the repo's *purpose* was elevated: it is no longer framed as a mere per-file artifact mirror of the source but as the **backbone / storage layer the Dark Factory runs on** — all the context, knowledge, and artifacts the DF and its AI coding agents need that do not live in the source repo. ULA/AIQA were positioned as just the first steps, and the per-file folder structure as the fundament a more sophisticated DF accretes onto.

Two structural decisions followed for the DF repo. The per-source-file tree was moved under a top-level **`artifacts/`** directory, deliberately leaving the repo's top level open for future above-file layers (repo-wide context, cross-cutting aspects, eventual agent run/task state). And the per-file aspect nesting was flattened: **`aiqa/ula/` collapsed to just `ula/`**, with AIQA demoted from a directory level to a purely conceptual umbrella (the family of analysis levels — ULA, later ILA, e2e). The user chose `artifacts/` over the agent's lean toward `files/`; the agent flagged the resulting double-meaning of "artifact" (a tracked source file vs a generated YAML output) as a prose-hygiene watch rather than a structural problem.

The substantive design work centered on making **ULA repeatable rather than one-shot**. The user established that in DF mode ULA runs many times as development progresses, and that the artifacts and the pass must be designed for that. Two run kinds were distinguished: incremental re-runs as source evolves (keyed on the file's git blob SHA, since content identity — not the head commit id — is what determines whether re-analysis is needed), and multiple runs with different configs on the *same* file version for model/approach comparison. The decision was to avoid a per-run folder structure entirely: each run's config is embedded *inside* the artifact, and **git history serves as the run store** — a clean fit with the framework's git-as-metadata principle. The agent synthesized the two concerns into a single dedupe key, **(source-content-SHA × config-id)**.

That mechanism crystallized into a named concept the user adopted: the **Provenance Header** — a small self-describing metadata block atop each artifact recording how and from what version it was produced, so each git snapshot stands on its own. The field set was largely worked out: drop `source` (the directory path already encodes it), `source-sha` as the git blob content hash (explicitly not a commit id), an extensible `config` bag whose `config.id` doubles as the dedupe handle, and `ula-version` kept outside `config` to separate process-version from run-tuning. A second open question — artifact **granularity** — landed on a lean rather than a decision: per-file artifacts with `unit` as a field, over noisy per-unit folders, justified by the insight that tracking-grain and storage-grain are separable (run/track per unit, store per file via parent aggregation). The user was explicit that this was "not sure" yet.

The session was deliberately **held mid-thread**. The user asked to preserve all findings, noted the ULA concept is in progress, and triggered `/lr:finalize`. Finalization renamed `codex-per-repo-mirror.md` to `df-per-repo-backbone.md` (git rm + new file, history preserved), created four further topics, swept all codex→df references across the lore, and recorded the in-progress status and a starter DF/ULA terminology glossary the user requested. Open threads carried forward: lock the Provenance Header schema, detail the incremental run algorithm (SHA-compare + per-unit merge), decide the granularity question, address the now-stale plugin-side `dev/aiqa/` + `/lr:dev-aiqa-repo-init` naming, and create the proposed `workdir/draft-df-glossary.md` (proposed but not created, pending the user's go-ahead).
