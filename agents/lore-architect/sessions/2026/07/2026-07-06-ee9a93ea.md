# Delivered v21 (Cursor dual skill tree) with full-suite verification

The session opened with the user asking whether any framework version existed locally but had not yet been delivered. The architect ground-truthed against git rather than lore alone and found that `lore-framework`'s last pushed release was v20, while **v21 — the Cursor dual skill tree — sat entirely as uncommitted working-tree changes**: a `VERSION` bump to 21, manifests already at `1.21.0`, doc edits, and the untracked `skills/cursor/` wrapper tree, `.cursor-plugin/plugin.json`, `scripts/sync-cursor-skills`, and `release-notes/21.md`. The user asked to finish it, align all versions, and deliver v21.

Before committing anything the architect inventoried the full scope and ran an alignment pass, which surfaced two real misalignments beyond the obvious. First, `versioning-release-types.md` had stopped at v19 — **both** the v20 and v21 history entries were missing, meaning the v20 ship had silently slipped the version-history backfill discipline and the next ship inherited the gap. Second, the dev repo carried a **stranded uncommitted 19→20 version stamp** in `lore-repo.md`, which boot had correctly deferred on (dirty `lore-repo.md` intersecting the upgrade write-set) — a live confirmation of the write-aware upgrade gate against a real half-landed stamp. The architect backfilled both history entries, stamped the dev repo to 21, verified cursor-tree parity (27/27, no orphans, `sync-cursor-skills` idempotent), ran `/lr:check` (#19 manifest, #20 migration write-paths, #21 cursor parity all clean), then committed and pushed the plugin (`f7b1c2b`) and the dev-repo alignment (`2f358d0`).

The plot twist came right after the push: the user pushed back — the full test suite should have run **before** the push, not after. The architect had shipped on a proportionate subset (deterministic tests + `/lr:check` + a single boot smoke), reasoning that v21 did not touch the procedures the lifecycle scenarios exercise. That reasoning was a review-time judgment, not a substitute for the gate the user considers part of shipping. The correction was to run the complete lifecycle suite — `LR_LIFECYCLE=1` across all scenarios — which passed **42/42** (19 real-engine scenarios + 23 deterministic) on the claude engine, roughly $9.4 and 27 minutes, entirely green.

The user then set the remaining work to run autonomously end-to-end: deliver the in-progress multi-engine harness changes into the dev repo, then push and finalize everything. The architect committed the harness support (`harness.py` cursor+codex drivers and engine-neutral `memory_file_name()`, plus the `test_repo_workspace.py` init generalization) as `73876d1` and pushed it, then ran finalization. Reflection captured three lessons; merge folded all three into existing topics (no new topics), sharpening `execution-testing-catches-blind-ambiguity.md` and `role.md` on pre-ship = pre-push, adding the "history backfill is not self-healing — scan the whole tail" note to `versioning-release-types.md`, recording check #21 in `consistency-checks.md`, and updating `cursor-dual-skill-tree-one-repo.md` / `lifecycle-testing-harness.md` / `multi-engine-portability-direction.md` and `lore-context.md` to the v21-shipped, full-harness-verified state.

Where it landed: **v21 is delivered and full-harness-verified**, both repos in sync with their remotes, the dev repo aligned at version 21, and the multi-engine harness support committed. The v21 "implemented but not yet full-harness-verified" caveat is retired.
