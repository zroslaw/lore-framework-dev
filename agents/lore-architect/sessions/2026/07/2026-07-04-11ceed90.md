# Validated the `<framework-root>` self-location design on Claude/haiku

The session resumed the multi-engine-portability thread from the prior session, which had left two parked port drafts (Codex, Cursor) and a lifecycle testing harness that so far only ran against Claude Code. The user opened by asking where things stood, then corrected a shared misremembering — believing the harness already had a Codex driver. A direct check of `harness.py` showed otherwise: `run_engine()` still raises `NotImplementedError` for anything but `claude`, and last session's Codex result was a manual smoke test, never wired into the suite. With the state re-grounded, the user steered into the substantive design question: how to make the framework feel native on every engine, starting with killing `${CLAUDE_PLUGIN_ROOT}` and the other Claude-specific couplings.

The architect laid out the engine-adapter approach — neutralize the vocabulary in the shared docs, bind the handful of genuinely engine-specific concepts once per engine in a `docs/engines/<engine>.md` adapter — and, at the user's request, wrote two workdir trial guides (`first-steps-codex.md`, `first-steps-cursor.md`) for booting the framework manually on each engine. A grep sweep then produced the real coupling inventory (`claude-specific-inventory.md`): `${CLAUDE_PLUGIN_ROOT}` at 103 sites but mechanical, subagent spawning at 66 sites across 12 fan-out docs as the one hard, fidelity-sensitive nucleus, and a handful of smaller Tier-A/Tier-C items. The load-bearing insight was that the whole port surface is five adapter bindings, with volume dominated by mechanical framework-root work plus that single subagent nucleus.

The core of the session was empirical. The user asked to actually test the `<framework-root>` idea on Claude, and whether the existing tests could catch this class of misbehaviour — proposing haiku as the probe. They could: the boot test's canary mechanism is a direct detector, and the harness already exposes `LR_FRAMEWORK_DIR` (point at a modified copy) and `LR_TEST_MODEL`. The architect built an isolated converted copy of the framework — swapping the var for `<framework-root>` plus a pure self-location rule ("two levels up from this SKILL.md, the directory with VERSION"), deliberately omitting any env-var mention so a pass would genuinely prove self-location. Boot happy-path and degraded both passed on haiku, and a verbose `stream-json` run settled the confound: haiku read the doc at the copy path and even read the `VERSION` file to anchor the root, never touching a plugin-cache path — clean proof it reasoned out the resolution rather than leaning on the Claude variable.

The plot twist came when the user proposed also applying the subagent-spawning changes and running the whole suite together. The architect pushed back and recommended sequencing instead: the subagent change isn't a ready diff (it needs the `docs/engines/` adapter built first), it's 66 fidelity-sensitive sites where a red test would be hard to localize, and on Claude it could only ever prove no-regression. The user agreed — framework-root-full now, subagent adapter as its own next-session step. A full framework-root conversion (41 files, self-location line into 22 `SKILL.md`) ran the entire 19-scenario suite on haiku against the copy: 18/19 passed, including every subagent fan-out scenario (consult, attach, merge, finalize, recall), clearing the concern that the host wouldn't substitute the resolved path into subagent briefs.

The single failure, `test_06` (upgrade-gate-on-dirty-tree), became the session's best result once the user reframed it: the goal is that even haiku can execute a procedure, so a haiku failure is a signal the doc is ambiguous, not just that the model is weak. A baseline run showed the original framework passing that scenario on haiku — but rather than dismiss it as flakiness, the architect diagnosed the failure mode: haiku got the hard part right (detected the dirty-tree collision, deferred the upgrade) but conflated "upgrade deferred" with "boot failed" and emitted the harness's `BOOT-FAILED` sentinel, because `version-check.md` prints an alarming "cannot auto-upgrade" message inline while the "keep booting" reassurance was buried in the invariants. Hoisting that reassurance to sit adjacent to the scary message (in both defer points and in `agent-boot.md` step 3) fixed it — three consecutive green haiku runs. This crystallized a foundational principle: haiku as an ambiguity detector, and the bar being "explanatory enough that even haiku executes faithfully."

The session landed with everything validated but nothing applied to the live plugin — a deliberate choice to keep the real `lore-framework` stable and land the whole port as one reviewed, tested change set next session. Both change sets (framework-root-full and the defer-clarity fix) are staged only in the scratchpad copy. Finalization then captured the session: four new lore topics, six updated, and a rewritten multi-engine-portability status.
