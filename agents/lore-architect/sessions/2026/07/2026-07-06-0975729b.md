# Shipped v22 install guides and Codex refresh guidance

The session opened with a straightforward operational request: boot `lore-architect` and explain why Codex was reporting an older framework version than the local repo. The architect verified that the source tree had already advanced past the installed Codex plugin cache, refreshed the installed plugin, and then the conversation widened into a framework-design question: how to keep teammates and customers from rediscovering engine-specific update quirks by hand every time the framework ships.

The substantive work turned into a `v22` framework follow-up aimed at that exact pain. The architect added top-level `INSTALL-CODEX.md` and `INSTALL-CURSOR.md` pages next to the README so a fresh engine session can be pointed at one concrete file, added `scripts/codex-refresh-plugin`, updated the Codex and Cursor engine notes with explicit load and refresh behavior, and changed `docs/version-check.md` so a repo-newer-than-framework warning now tells Codex users the real remedy instead of a Claude-flavored generic `/plugin update` line. The framework repo was version-bumped to `22`, the manifests moved to `1.22.0`, and the release shipped publicly as commit `10b41a7`.

The plot twist came in the targeted lifecycle verification the user requested. A new Codex scenario was added to `tests/lifecycle/test_boot.py` to assert the `R > F` warning path on `gpt-5.4-mini`, but the first reruns failed in revealing ways. One version of the test accidentally exercised an older copied framework tree, and the next one proved something more interesting: even with the source repo at `22`, Codex was still loading the installed `1.21.0` cache, so the old generic warning text surfaced from the cached `version-check.md`. The architect corrected the scenario, refreshed the installed plugin to `1.22.0`, reran only the directly affected scenario, and got a clean pass.

The session then closed the loop in the dev repo rather than leaving the release as an uncontextualized public push. Lore was updated to record `v22` in `versioning-release-types.md`, extend `plugin-distribution.md` and `codex-local-plugin-update.md`, and introduce three durable engine hub topics — `claude-engine-capabilities.md`, `codex-engine-capabilities.md`, and `cursor-engine-capabilities.md` — so future engine work starts from a stable per-engine map instead of scattered probe notes. `lore-context.md` was updated to point at those hubs and to describe `v22` as the current shipped framework state.

Where it landed: the framework release is shipped, the directly affected Codex lifecycle path is green on `gpt-5.4-mini`, and the dev repo now captures both the operational fix and the higher-level knowledge-organization lesson the user wanted preserved. Two things were intentionally left out of the finalize commit: unrelated untracked dev work (`draft-lore-quality-benchmark.md` and `tests/quality/`) and a manual stamp of `lore-framework-dev/lore-repo.md` from `21` to `22`, which remains for the normal update path rather than being smuggled in through finalization.
