# Drafted Lore Agents Intro for the Activities team and harvested its framings

The session opened with the user wanting an onboarding guide so colleagues at Agoda — managers and developers — could try Lore Agents in their day-to-day work. The architect was spawned as a parallel teammate via `/lr:spawn-teammate`, then booted from `lore-framework-dev/agents/lore-architect`. After the architect ran the boot procedure, picked up a v9→v10 auto-stamp on `lore-repo.md`, and confirmed identity, the user steered the session away from the team-lead default and asked the architect to converse directly. The architect proposed a structure and the iterative drafting started.

Most of the substantive work was in section 1 — "The idea". Initial drafts framed the framework as a knowledge-and-skills sharing system, which the user rejected three times before the right framing landed: **LRF is the environment, engine, and tooling for running self-improving Lore Agents**, where the knowledge base is a *consequence* of how Lore Agents persist their state, not the framework's identity. From there, four headline beats settled into the prose, each a blockquote followed by an unwrapping paragraph: *structured-hassle-free continuously updated KB+skills*, *team asset not personal*, *positive feedback loop between usage and learning*, *skill set growth made easy and frictionless*. A fifth beat — *the agent is the universal working environment* — was added late at the user's request to position Lore Agents inside the emerging behavior pattern of working-through-an-agent.

Several editorial corrections shaped the doc. The user pushed back on knowledge and skills being treated as a hendiadys and explicitly distinguished them: knowledge is information in markdown; skills are tools plus instructions for specific tasks. The user also flagged "Lore Agents are intended to…" repetition and asked for the third-paragraph headline to focus on the feedback loop dynamic rather than capability claims. A terminology trap surfaced when "domain" appeared in two senses in adjacent paragraphs — the LRF technical concept (top-level workspace dir) and loose English (subject area). The architect ran a `/lr:recall` to retrieve the canonical Domain definition, then swept six occurrences across the doc and replaced the loose-English uses with "area" / "subsystem" / "field of application." The user reacted strongly enough to that mistake that it became its own lore topic afterward.

Plot twists stayed minor. The architect once tried to use `AskUserQuestion` to gather preferences and the user objected — they were here, talking directly, no need for tool-mediated prompts. The architect dropped that pattern and shifted to plain prose questions. Reader-directed "your X" phrasing crept into setup instructions and was swept out in a separate pass — the doc is *for* readers, not *about* them. Late in the session, the user revised the document's structure independently (renaming sections, marking some TBD) and asked the architect to beautify §1 only; the architect added subheaders, converted bold-sentence headlines into blockquote headlines, fixed typos, and inserted the universal-working-environment paragraph as the §1 closer.

Where it landed: `LORE_AGENTS_INTRO.md` exists in `activities-lore-agents/` with §1 (the idea) complete, §4 (features — full command reference grouped by purpose) complete, and §2/§3/§5 marked TBD by the user for later. Two commits and two pushes to `origin/main` along the way (`d0bf725` for the file rename and initial commit, `d1d5bb5` for the Features section + README pointer). The session also produced eight new lore topics for the architect itself — the foundational framings, narrative pattern, and editorial-trap topics that powered the doc, now reusable for future onboarding-doc requests from other teams. The role was extended to cover co-authoring framework onboarding docs, sitting one layer above the existing agent-bootstrapping responsibility. Lore-context was reorganized: the design-principles section now leads with the identity-layer trio (team-shared / engine-not-KB / executor-first) instead of just team-shared, and a new "Onboarding Doc Authoring" cluster groups the eight new topics by purpose.

Next steps belong to the user. The intro's §2 (use cases), §3 (anatomy), and §5 (business domain & swarm) are still TBD; the user has notes inline (supply-manager, manager's pulse, quarter planning, GlobalTix calendar issue, oncall agent, RFC exploration) that point toward §2's eventual shape. The architect's onboarding-doc toolkit is ready; when the next team requests help, the eight topics provide the framing scaffolding from the start instead of having to be re-derived through iteration.
