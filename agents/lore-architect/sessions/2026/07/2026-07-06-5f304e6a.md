# Wired Codex into the harness

The session opened with a straightforward follow-up question after the Cursor and framework release work: should the Codex path finally be wired into the lifecycle harness now that the framework-side port was already shipped. The answer was yes, but only in the narrow sense of automating the already-proven manual Codex path rather than redesigning the harness around Codex.

The substantive work happened in `tests/lifecycle/harness.py`. The harness already supported Claude Code and Cursor, while the Codex path still lived only in lore as a manual methodology. This session added a real `codex` branch to `run_engine()`, gave Codex its own default model (`gpt-5.4-mini`), and introduced a prompt adapter that translates the harness's engine-neutral prompts into the reliable Codex **doc-driven** path. Instead of assuming slash-skill dispatch, the adapter points directly at `LR_FRAMEWORK_DIR/docs/*.md`, which lets the harness exercise Boot Step 0, framework-root resolution, and the procedural docs against the selected framework tree without relying on the user's installed plugin state.

The session's main plot twist was environmental rather than architectural. The first real smoke run of the new Codex branch against `test_boot.py` failed immediately, but the failure was not in the driver logic: Codex could not write its local state database under `~/.codex/state_5.sqlite` because the host process was still running inside an outer sandbox. Rerunning the exact same happy-path boot test unsandboxed passed cleanly in about fifty-three seconds on `gpt-5.4-mini`. That overturned the more suspicious first read and clarified the operational rule: local Codex lifecycle runs from this environment require host-level write access to `~/.codex/`, and state-db write errors should be treated as runtime setup issues rather than as framework regressions.

The lore work at the end of the session updated the framework's current understanding accordingly. `lifecycle-testing-harness.md` no longer describes the Codex driver as unwired, `codex-testing-methodology.md` now sits beside an actual automated branch instead of a purely manual one, and the present-tense context records the new constraint that local Codex harness runs may need an unsandboxed outer launch even when the inner fixture workspace is throwaway. A historical Codex findings topic and the port follow-up topic were also adjusted so the remaining open work is now about broader scenario coverage and rollout-log-grade verification, not about basic harness wiring.

The session closed with only the lore-side updates finalized. The new Codex harness code in `tests/lifecycle/harness.py` remains local and uncommitted because finalize only records the `agents/` tree. There were also unrelated pre-existing local changes in `lore-repo.md`, `tests/lifecycle/test_repo_workspace.py`, and an untracked lore file under `agents/`; those were intentionally kept out of the finalize commit. The next step is a separate manual dev-repo commit for the test harness itself if the team wants the new Codex driver in shared history.
