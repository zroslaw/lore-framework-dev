# Shipped v12 /lr:doctor with ailment-catalog pattern and cache-clear authoring convention

The session opened with the user noticing that `/lr:workspace-sync` was missing from the available-skills list despite the framework being at v11 — a textbook stale-plugin-cache symptom. The user's suspicion was correct, and they proposed a generative response rather than a one-off fix: a new `/lr:doctor` skill that would carry the framework's own runtime-troubleshooting wisdom, with the cache-clearance case as the first ailment, and the ability to grow over time as more failure modes surface from real-world use.

The architect took the proposal directly to implementation. v12 shipped four artifacts in the framework distribution: `skills/doctor/SKILL.md` (a one-line pointer following the established skill-doc pattern), `docs/doctor.md` (the orchestrator with matching procedure, catalog index, and authoring schema), `docs/doctor-stale-plugin-cache.md` (the first ailment topic, with Symptoms / Diagnosis / Remedy / Why It Happens sections), and `release-notes/12.md`. Mid-implementation the user raised a follow-up question — should migration guides always include a cache-clearance note when commands or skills change? — and the architect agreed and codified it as a new authoring convention in `docs/conventions.md` § Migration / Release-Note Authoring, with `release-notes/12.md` itself applying the convention as the worked example.

Before shipping, the user asked for parallel multi-lens review. The architect dispatched three free agents — UX/newcomer/discoverability, framework architectural consistency, and correctness/safety with filesystem verification — using mutually-exclusive lens prompts. The reviews returned roughly twenty-five findings across the three lenses, with minimal noise. The architect triaged these into ~14 substantive accepts (high-impact correctness and clarity defects), 4 deferrals to lore-side capture, and 2 declines as premature. The accepted set included three genuine high-severity defects that would have shipped without the review: the cache wipe was scoped wider than necessary (`~/.claude/plugins/cache/` instead of the targeted `~/.claude/plugins/cache/lore-framework/`, putting ~40MB of unrelated plugin state at risk), a broken cross-reference to a non-existent `versioning-release-types.md` in the framework distribution, and the durable topic itself missing a bootstrap note for the chicken-and-egg case where the very ailment can mask its own diagnostic tool. A focused round-2 single-reviewer pass on the applied fixes confirmed shipping with only one minor nit (a dangling lore-only reference removed).

Plot twists were modest. The cache-clear convention emerged mid-flight rather than being part of the original design — the user's "should we always note cache clearance in migration guides?" prompt converted what was going to be just a doctor-and-ailment ship into a doctor-plus-authoring-convention ship, with the convention dogfooded in v12's own release notes. The round-1 architecture reviewer's strongest finding — that the convention's home should be where authors actually write release notes, not in the general `conventions.md` — was partially accepted: the architect added a § For Framework Authors section in `docs/version-check.md` as a signpost into `conventions.md`, rather than relocating the canonical text.

The session landed with v12 committed as `b735777` and pushed to `github.com/zroslaw/lore-framework`. Two new lore topics were added to the architect's lore (`ailment-catalog-pattern.md` naming the new orchestration variant per the meta-rule, `cache-clear-footer-convention.md` capturing the authoring discipline), `parallel-reviewer-fanout-pattern.md` was extended with the v12 application to doc-only ships, `versioning-release-types.md` was backfilled with v11 + v12 history entries plus a new "cache-affecting?" annotation axis, and `role.md` gained a Lore-Curation Disciplines section codifying four operational rhythms applied at every VERSION-bump finalization. The architect repo's own `lore-repo.md` is still stamped at v11 — the framework's auto-upgrade flow handles that on next boot.

Next steps are mostly hands-off. The next time anyone hits a stale-plugin-cache symptom they should reach for `/lr:doctor`; if they hit a runtime issue the catalog doesn't yet cover, the new ailment topic lands during that session's reflection. The `framework-improvements-backlog.md` accrued an entry for refreshing the README's skills table to include `/lr:doctor`, and a new "Ailment Catalog" section tracking accretion candidates (workspace-sync conflict types, common merge-conflict shapes during finalization, onboarding-doc anti-patterns are the named candidates).
