# Added Lore Beings command UX

The session opened with the user challenging the usability of Lore Beings. The existing quick-start guidance exposed too much raw `lrb` machinery, and the user made the desired shape explicit: `/lr:being` should be the single entry point for Beings-related status, help, init/create flows, validation, logs, Keeper control, engine configuration, workspace registration, and pause/resume operations.

The framework work landed that command surface in `lore-framework`. A new `/lr:being` skill became the user-facing wrapper around the deterministic Being Keeper CLI, `docs/being.md` became the quick command reference, `docs/beings.md` was adjusted to prefer the shortcut, and `scripts/lrb.py` gained zero-cost `validate` and `logs` commands plus richer status output. The framework version was advanced for v29, plugin manifests were updated, release notes were added, and the Cursor skill wrapper was generated. This was committed as `abdcfbe` on `codex/being-skill`.

The dev-side work separated Lore Beings lifecycle tests from the standard lifecycle matrix. The old Keeper files were moved out of `tests/lifecycle/` into `tests/lifecycle_beings/`, a dedicated `tests/lifecycle_beings/run_matrix.py` was added, and the standard runner was narrowed back to ordinary framework lifecycle modules. `tests/test_lrb.py` gained deterministic coverage for the new `lrb validate`, `lrb logs`, status engine metrics, and return-code behavior.

A parallel session had just finalized broader testing documentation and quality reporting improvements. The user asked for this branch to take that update into account before finalization, so the dev branch merged that docs update and reconciled all Beings-specific testing references. The docs now consistently describe standard lifecycle, Lore Beings lifecycle, and quality benchmark evidence as separate tracks with separate gates, result paths, and artifact contracts. The stale-reference sweep removed current references to `LR_LIFECYCLE_KEEPER`, old `tests/lifecycle/test_lrb_lifecycle.py` paths, and the old Keeper sub-suite wording. This reconciliation was committed as `253fd4b` on `codex/being-skill-dev`.

Verification stayed within the no-spend boundary. Syntax checks passed for the modified runners and Beings lifecycle modules. `tests/lifecycle/run_matrix.py --dry-run` resolved only the standard lifecycle modules, `tests/lifecycle_beings/run_matrix.py --dry-run` resolved the separate Lore Beings matrix, the Beings runner refused to execute without `LR_LIFECYCLE_BEINGS=1`, `tests/quality/run_matrix.py --no-local-config --dry-run` still resolved the regular quality matrix, and ungated unittest discovery passed with 185 tests run and 35 real-engine tests skipped.

The unresolved piece is deliberate: the costly real-engine Lore Beings lifecycle matrix was not run in this session. It remains the release-gate command for a Beings-touching release: `LR_LIFECYCLE_BEINGS=1 python3 tests/lifecycle_beings/run_matrix.py --engine-jobs 3 --module-jobs 1`.
