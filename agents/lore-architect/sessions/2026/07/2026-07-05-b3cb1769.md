# Shipped v18 (lr-wait) and v19 (multi-engine Codex port) to customers

The session opened with a simple question — was the Codex version of the lore framework available to
users yet? It was not: the port had been validated end-to-end on real `codex exec` but lived only in
a local, no-remote `lore-framework-codex` sibling build, with landing work still outstanding. The
user then set the goal plainly: make it available for everyone, and guide them through it. A `/goal`
was set to ship the versions to customers, converting the work from advisory to executor mode.

Inspecting disk state reframed the task. The public `lore-framework` `HEAD` sat at v17, which meant
not just the port but the entire v18 lr-wait release was unreleased — sitting uncommitted in the
working tree alongside three uncommitted style skills. Making Codex available therefore required an
ordered untangling, and the user chose a two-ship sequence. Ship one committed and pushed v18
(lr-wait, the first bundled MCP server) as `e7514db` after `test_wait.py` passed 23/23, deliberately
holding back the style skills for the port release. Ship two was the port itself.

The landing technique kept the port ship low-risk. Because the codex build's working tree was a true
superset of committed v18 plus the port, a `diff -rq` between the real v18 tree and the codex build
isolated exactly the port-touched files — everything identical (all of lr-wait) dropped out. After
verifying that a file both v18 and the port had modified (`conventions.md`) contained both change
sets, the port files were copied wholesale rather than hand-merged. Defer-clarity, which was a
separately-staged change set and not present in the codex build, was authored fresh onto
`version-check.md` and `agent-boot.md` step 3. A one-line Engine note neutralized the Claude-specific
timeout prose in `auto-pull.md`. The ceremony followed: VERSION to 19, both manifests to 1.19.0,
`release-notes/19.md` with the cache-clear footer, and consistency checks confirming every SKILL.md
carried its self-location line and only legitimate `${CLAUDE_PLUGIN_ROOT}` references remained.

Verification used the empirical leg the role prescribes. The boot lifecycle suite ran 6/6 on haiku
against the real v19 tree — including test_06, the dirty-tree upgrade gate that had originally
surfaced the defer-clarity ambiguity, now deferring cleanly without emitting a boot-failure sentinel.
With that green, v19 was committed and pushed as `72b1b2a`, and both versions went live on
`github.com/zroslaw/lore-framework`.

A follow-up request closed the loop locally: update the machine's Codex install to the new version.
The marketplace was already registered as a local source pointing at the working copy, so no git
fetch applied; the operational finding was that `codex plugin marketplace upgrade` is Git-only and
that a local-marketplace update needs `codex plugin remove` + `codex plugin add` to re-read the live
`marketplace.json`. The cache moved cleanly from 1.18.0 to 1.19.0, verified to contain
`docs/engines/`, the AGENTS.md memory binding, and the style skills.

Where it landed: both releases are public and the local Codex runs v19. The multi-engine port is
shipped, not exploratory. Deliberately Claude-first and deferred: the lr-wait MCP server (its
`.mcp.json` still uses `${CLAUDE_PLUGIN_ROOT}`), the DF/AIQA module, and `migrations/*`. Still open as
follow-ups: wiring a real `codex` driver into `harness.py` so the Codex path joins the automated
suite, confirming Cursor `--plugin-dir` parity, and deleting the now-superseded `lore-framework-codex`
sibling build.
