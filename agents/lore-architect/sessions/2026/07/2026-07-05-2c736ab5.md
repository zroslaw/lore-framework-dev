# Landed Cursor profile as v20

The session opened with the Cursor migration in an awkward in-between state. Cursor had already been validated locally as a near-landing build, but the canonical `lore-framework` repo still described that work as not yet landed, and the user wanted a clear answer on what remained and then asked to land it.

The work centered on comparing the validated `lore-framework-cursor/` sibling against canonical `lore-framework`, confirming the exact five-file framework diff, and checking the matching lifecycle-harness changes in `lore-framework-dev/tests/`. From there, the canonical plugin was updated with `docs/engines/cursor.md`, the boot-time Cursor detection changes in `docs/agent-boot.md`, the Cursor-specific engine notes in `docs/attach.md`, `docs/init.md`, and `docs/resolve-conflicts.md`, plus `release-notes/20.md` and the routine `VERSION` and manifest bumps to `20` / `1.20.0`.

One useful correction surfaced mid-session: the framework landing had been completed locally, but it had not actually been pushed yet. The git state was checked explicitly, which confirmed that `origin/main` still pointed at the v19 release commit and that the Cursor landing existed only as a local working-tree diff. A second operational nuance also stayed important throughout the session: the validated Cursor harness changes live in `lore-framework-dev/tests/`, outside finalize's `git add agents/` scope, so the framework release push and the dev-repo harness follow-up are separate actions rather than one automatic ship.

The session landed with the canonical plugin released and pushed as **v20** via commit `5cbb967` (`Release v20: land Cursor engine profile`). That changed the architectural record materially: Cursor is no longer merely a validated local build. During finalization, the lore-architect's own knowledge was updated so that `multi-engine-portability-direction.md`, `docs-engines-convention.md`, `cursor-port-validated-end-to-end.md`, `port-landing-next-steps.md`, `lifecycle-testing-harness.md`, and `lore-context.md` now reflect the shipped state rather than the pre-landing one.

The remaining open thread is the dev-repo side of the same work. `lore-framework-dev/tests/lifecycle/harness.py` and `tests/lifecycle/test_repo_workspace.py` still carry local Cursor-support changes, but they sit outside the finalize commit scope and were therefore intentionally left uncommitted here. The next step is to commit and push those harness changes separately if the team wants the validation machinery to match the now-shipped v20 framework state in public history.
