# lr-dev knowledge model reframed to context agents on existing primitives

The session opened with the user returning to the lr-dev design from the previous day — specifically to pressure-test the three-tier knowledge model (framework / repo / agent) that had been the spine of `draft-lr-dev.md`. The user's observation was that the design had accumulated too many new concepts and entities: a new repo kind (`dev-repo-lore.md`), a two-gate write capability, schema bumps to `role.md`, and cross-cutting migration work. The question was whether expressing per-repo artifact knowledge as ordinary tailored agents — using primitives the framework already has — would be simpler and still achieve the same outcomes.

The architect grounded the discussion in the actual drafts before responding, reading `draft-lr-dev.md`, `draft-lr-dev-quality.md`, and `lr-dev-direction.md` in full. The key realization that emerged was that the two-gate write capability was reinventing, for the new tier, the write-isolation agents already have natively: an agent's lore is written only by that agent via reflect/merge, so nobody else scribbles on it. This made the gate not just simpler but unnecessary. The `dev-repo-lore.md` repo kind and the associated migration work dissolved for the same reason — a per-repo agent repo carries a normal `lore-repo.md`, so the existing machinery already walks it. This was `framework-scope-vs-agent-scope.md` doing its job.

The user accepted the reframe and worked through the remaining open questions. The question of how many agents per repo was explored: product and technical knowledge initially seemed like two agents (matching two roles), but the architect argued they are filing categories, not identities — a senior engineer holds both, they belong as `product/` and `technical/` dirs inside one agent's single `lore/`. The user agreed. The question of naming took several rounds: "steward," "curator," "cartographer," "navigator," "twin," "analyst," and "model" were all considered and rejected. The user settled on `<repo>-context` — it harmonizes with existing `lore-context.md` vocabulary, and the `<repo>-` prefix disambiguates from the LLM-context sense. "Context agent" becomes the lr-dev kind-name for this role.

A fourth concern surfaced late: reproducibility. The context-agent role recurs across every repo and must be standardized, but if each instance bakes in the standard, later improvements don't propagate. The architect recognized this as the thin-pointer pattern the framework already uses for skills (`SKILL.md` → `docs/<name>.md`) and for boot commands (`/lr-<name>-agent` → `agent-boot.md`), now applied to a *role* for the first time: a versioned plugin doc (`docs/context-agent.md`) holds the behavior; each instance's `role.md` is a thin pointer to it; updates ride plugin distribution with zero per-instance churn.

The session landed with `draft-lr-dev.md` restructured — §1A added as the leading direction, §§2–4 preserved as the labeled heavier alternative — and `draft-lr-dev-quality.md` annotated with terminology alignment. Two new standalone lore framings were extracted: `framework-defined-role-pattern.md` (the thin-role-pointer pattern generalized) and `agent-split-only-when-forced.md` (one agent per coherent scope; split only when a concrete budget or role pressure reveals the axis). The `lr-dev-direction.md` anchor topic was updated so the reframe is the documented leading direction.

Open threads carried forward: what goes inside `docs/context-agent.md` (the standard operating manual for the context-agent role — hold/provide/learn disciplines, structure standard, reflection discipline, overrides convention); the generator surface (`/lr:create-agent` flag vs dedicated skill); and the question of how the activity-specialist agents (QA, coverage, review) are designed now that they're fully decoupled from the context agent.
