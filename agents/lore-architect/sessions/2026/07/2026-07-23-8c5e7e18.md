# Added quality reporting strategy

The session opened with the user asking Lore Architect to design and implement durable cost, time, execution, and rerun tracking for quality and benchmark tests, analogous to the lifecycle reporting work already on the backlog. The initial implementation added quality matrix artifacts that record per-config JSON, logs, machine summaries, human summaries, release-note blocks, cost metadata, duration metadata, git state, filters, and exact rerun commands.

The user then corrected the interpretation of the first real runs: Codex and Cursor had mostly executed their probe cases, with only a few real engine timeouts, so treating the configs as simply failed erased useful evidence. The reporting model was changed to make `partial` a first-class status and to separate engine technical failures from judge technical failures. `Engine OK`, `Scored`, and `Tech E/J` now make it visually obvious whether a run failed to execute or merely could not finish S3 judging because of a provider/session limit.

The quality matrix was run on the regular cheapest-model set: Claude haiku, Codex `gpt-5.4-mini`, and Cursor `composer-2.5`. The resulting persisted run was partial: Claude completed all engine rows and produced partial S3 evidence before judge quota ran out; Codex completed 34/36 engine rows; Cursor completed 33/36 engine rows. The generated `release-notes.md` showed known cost, unavailable cost fields, LUS and S3 metrics, grouped technical-failure categories, and targeted rerun commands.

The user pushed back again on readability. The report vocabulary was too terse, so `tests/quality/reporting.md` was added as a field guide explaining S1/S2/S3, LUS, S3 uplift, `Engine OK`, `Scored`, `Tech E/J`, cost fields, report artifacts, and links to every probe definition. The reporting generator now links every `summary.md`, `release-notes.md`, and `summary.json` back to that guide.

The work then expanded into documentation strategy. `tests/testing-strategy.md` now explains the whole test pyramid, cost/risk policy, release-gate policy, failure semantics, result artifacts, and when to add each kind of test. `tests/quality/strategy.md` explains the quality benchmark design: treatment/control arms, probe structure, S1/S2/S3 strategy, probe categories, difficulty mix, LUS versus behavior uplift, technical-failure interpretation, matrix tiers, and probe authoring anti-patterns. Finally, a root `README.md` and `lore-repo.md` documentation section were added so external readers can discover the testing and quality docs from the repository front door.

The session landed with implementation checks passing: local Markdown links across the new docs resolve, the touched quality Python files compile, the quality matrix dry-run still resolves the regular cheapest-model matrix, and the saved benchmark summary JSON parses. Lore reflection topics were merged into existing and new lore topics so future Lore Architect sessions retain the reporting contract, the partial technical-failure distinction, the benchmark vocabulary documentation rule, and root documentation discoverability.
