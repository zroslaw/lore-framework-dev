# Shipped v14 — portable auto-pull and cache-correct plugin manifest versioning

The session opened on a boot: the user ran the `lore-architect` boot shortcut and immediately noticed it had "encountered some issues," asking how to fix them and improve the boot process. The boot itself had surfaced the symptom — the auto-pull step's `timeout 60 git …` form had failed with `command not found` (exit 127) and only completed because the host fell back to the Bash tool's own timeout.

Diagnosis traced it to a real, high-impact bug. v13's `docs/auto-pull.md` described the boot pull as running "with a short timeout (60 seconds is reasonable)," which naturally invites wrapping the command in the `timeout` binary. That binary (and `gtimeout`) is absent on a stock macOS/BSD, so on the primary dev platform the pull aborted with exit 127 and boot silently dropped into degraded mode — meaning v13's headline auto-pull feature had never actually run on macOS despite appearing shipped. The user chose a full v14 release. The fix bound the pull via the agent's own tool-level timeout, hardened SSH against prompt-hangs, and named the underlying lesson as a new `§ Tooling: Portable Shell` convention so the whole class of GNU-ism portability bug gets caught at authoring time.

Two threads then deepened the release. First, the user asked whether the fix also covered HTTPS remotes; that produced a transport-symmetric reframe of `auto-pull.md` Step 2 — the Bash-tool timeout stated as the transport-agnostic backstop, with `GIT_TERMINAL_PROMPT=0` (HTTPS), `GIT_SSH_COMMAND` BatchMode/ConnectTimeout (SSH), and optional `http.lowSpeed*` as per-transport fast-fail niceties on top. Second, a `/code-review high` pass surfaced the root cause of years of recurring cache pain: `.claude-plugin/plugin.json` and `marketplace.json` had been frozen at `1.0.0` since v1, so Claude Code never detected a new plugin release and never refreshed its own cache — the manual Clear Plugin Cache ritual was a workaround for a self-inflicted gap. The user recognized it ("ah, that's why…") and asked to fix it permanently. The remedy: a mechanical `1.<VERSION>.0` manifest mapping (so v14 → `1.14.0`) that can't be forgotten because it derives from `VERSION`, documented as `§ Plugin Manifest Versioning`, and enforced by a new `/lr:check` check #19.

Two corrections shaped the back half. When offered the check-#19 enforcement rather than simply shipped it, the user pushed back — "yes, you should have done it above" — sharpening the don't-defer discipline into "don't merely offer a clearly-correct, bounded completion; just do it." And while wiring check #19, the host mis-attributed two parallel `Read` results and nearly "fixed" a healthy `skills/check/SKILL.md` pointer into a broken one; an `ls`/`grep` verification before editing caught it (the confusion: `consistency-checks.md` is a lore-topic name, while the plugin catalog doc is `check.md`). Both lessons were captured as lore.

It landed as a clean ship. The plugin v14 release — VERSION, both manifests, `auto-pull.md`, `check.md`, `conventions.md`, `scripts/workspace-sync` (which also gained the SSH hardening), and `release-notes/{13,14}.md` — was committed to `lore-framework` main as `5acaff3` and pushed. Two low-severity `/code-review` findings shipped as-is per the user's call (check #19 should skip a missing `marketplace.json` gracefully; `workspace-sync`'s new BatchMode warrants a behavior-change note) and were routed to the backlog. Finalization then recorded the knowledge: three new lore topics (`plugin-manifest-versioning`, `portable-shell-in-framework-docs`, `verify-before-acting-on-suspected-bugs`), five updated topics, a new role-level manifest-versioning discipline, and a refreshed `lore-context.md` at v14.
