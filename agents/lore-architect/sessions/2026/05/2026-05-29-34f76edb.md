# Shipped v11 — `/lr:workspace-sync`, `repos:` field, workspace vocabulary

The session opened with a brainstorm on what's still missing from the framework's domain (now: workspace) operations. The user's framing converged quickly: rather than introduce a separate domain manifest, *associate the domain with an agent repo itself* — each `lore-repo.md` declares its dependencies, the parent directory becomes a workspace, and a single command bootstraps and syncs everything.

The design landed in iterations. An early sketch over-elaborated with subject-repos vs companion-repos and `/lr:sync` separate from `/lr:workspace-sync`; the user pushed back with "too complex, simpler, wording and ideas," and the design collapsed cleanly to one flat `repos:` list and one renamed skill (`/lr:pull-domain` → `/lr:workspace-sync`). Bootstrap and ongoing sync became the same operation — first run happens to clone everything because nothing exists yet.

The implementation phase produced `scripts/workspace-sync` (Bash 3.2+ compatible, parallel clone + pull, conflict reporting), `skills/workspace-sync/SKILL.md`, `docs/workspace-sync.md`, and a hard rename of the old skill (no compat alias). Three reviewer subagents were fanned out in parallel against the draft — correctness/bash, security/parser robustness, UX/docs. Each returned a focused report with severities; findings spanned BOM-prefixed descriptors silently dropping all repos, code-fenced YAML examples being parsed as live frontmatter, path-traversal via crafted `..git` URLs, parallel-write tmpfile collisions on similar dir names, locale-sensitive output heuristics, Ctrl-C cleanup gaps, and ANSI-escape injection via hostile descriptor URLs. All BLOCKER, HIGH, and MEDIUM findings were applied; the script was hardened against every concrete attack the security reviewer surfaced.

A second-round fan-out — terminology coherence, newcomer experience, release readiness — was added after the doc sweep. The newcomer reviewer's headline finding reshaped the README: the Quick Start was mis-routing team-joiners to `/lr:create-repo` when they should run `/lr:workspace-sync`. The README was restructured with a new Concepts section (workspace, agent repo, agent, lore, boot, finalize) and a two-path Quick Start (joining a team / starting fresh). The terminology reviewer caught a leftover BLOCKER in `version-check.md:78` ("in the domain") and HIGH ambiguities in `agent-boot.md` and `consult.md`. Release-readiness verdict was ship-as-is for the framework plugin; remaining cleanup (lore-architect's own lore) became post-ship work.

The plot twist mid-session was the user's pushback on a deferral. The architect had punted "full doc sweep of remaining `<domain>` refs in 12 files" to v11.1 follow-up, calling it tedious-but-mechanical. The user asked plainly: *"Sorry, why didn't you do this and postponed?"* The sweep took five minutes and the framework was half-migrated until done. The reviewer fan-out then independently caught the half-migration as a BLOCKER, validating the user's instinct. This crystallized into a feedback rule — don't defer bounded mechanical scope — that earned its own lore topic.

The release shipped at commit `919df1c` on `github.com/zroslaw/lore-framework`: 21 files changed, 949 insertions, 146 deletions. The `<domain>` placeholder is retired in favor of `<workspace>`; "domain" survives as prose for the conceptual scope of an agent repo. The `/lr:init` canonical payload was edited from "Lore Framework Domain" to "Lore Framework Workspace" — existing workspaces will see show-diff-and-confirm on their next `/lr:init`. Auto-stamp v10→v11 happens at boot via release-notes-only flow.

Deferred items captured in the backlog: `repos:` validators in `/lr:check`, undeclared-top-level-repo nudge after sync, per-entry overrides (branch pinning, custom dir name), inline-flow YAML form. The architect's role.md gained a "Three-Layer Design Model (v11+)" section formalizing plugin/domain/workspace as discrete decision layers — every future framework change should now triage against this trichotomy.
