# Shipped v23 to hide Cursor wrappers from Codex

The session opened with a small but user-visible defect in the freshly shipped multi-engine Lore plugin: Codex was showing a redundant second set of Cursor wrapper skills such as `lr:lr-boot`, `lr:lr-check`, and `lr:lr-finalize`, cluttering the skill picker and wasting context on duplicate capabilities. The architect first verified the symptom directly instead of trusting the repo layout, confirming that the live Codex plugin still exposed 54 Lore skills even after the wrapper tree had been moved out of `skills/cursor/`.

What followed was a packaging investigation rather than a framework redesign. The first fix moved the Cursor wrappers into a top-level `cursor-skills/` directory and repointed `.cursor-plugin/plugin.json`, but a real `codex exec` run still showed the duplicates. That forced a deeper look at the installed cache, which revealed two separate truths: Codex was still reading a stale cached plugin snapshot, and the repo itself still carried an old generated `cursor-skills/` directory that kept the duplicate tree alive even after the new hidden path was added. The correction was to move the wrapper side all the way into a hidden plugin root, `.cursor-skills/`, delete the old `skills/cursor/` tree, remove the stale top-level `cursor-skills/` directory, refresh the installed Codex plugin, and only then judge the result.

The turning point was empirical verification on the real engines rather than more reasoning about the filesystem. After the refresh, Codex reported exactly 27 `lr:*` skills with `has_double_prefixed: false`, proving the duplicate-skill problem was actually gone. Cursor was then checked from its own CLI against `--plugin-dir /Users/yaroslav/Documents/git-repos/lore-framework`: it reported 27 visible `/lr` slash skills, and `/lr-boot lore-architect` still worked. One negative Cursor probe (`/lr-lr-boot`) stayed ambiguous because the model inferred intent instead of cleanly rejecting the string, but the engine-positive checks were sufficient for the user’s ship decision.

With the packaging fix proven, the user explicitly chose to ship and to keep verification bounded to Codex and Cursor rather than spending remaining budget on Claude Code. The architect turned the fix into **v23** of `lore-framework`: bumped `VERSION` and all plugin manifests to `1.23.0`, added `release-notes/23.md`, updated the wrapper generator and runtime docs, refreshed the installed local `lr@lore-framework` plugin, and pushed the framework repo as commit `3c7eeba` (`v23: hide cursor wrappers from codex`). The durable lesson carried into lore was that Codex’s plugin crawl can surface wrapper trees outside Cursor’s intended scope, and that packaging changes are not actually validated until the installed plugin cache is refreshed and a real CLI skill-count check is rerun.

The session ended with Lore finalization for the architect’s own repo. Reflection distilled the hidden-wrapper-tree lesson, merge folded it into the existing Cursor/Codex portability topics and the versioning history, `lore-context.md` was updated to the `v23` state, and `lore-framework-dev/lore-repo.md` was stamped to 23 so the next boot would not defer on a stale repo version. The unrelated untracked artifacts under `agents/lore-architect/workdir/` and `tests/quality/` were intentionally left out of the finalize commit. The remaining next step is straightforward: push the dev-repo lore commit so the framework-development knowledge about `v23` becomes team-shared, without bundling those unrelated scratch files.
