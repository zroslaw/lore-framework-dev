# Full architecture review yields prioritized improvement list

The user, running at max reasoning effort, asked the lore-architect for a full review of
the Lore Framework's architecture and implementation: what to improve, what gaps or
inconsistencies exist, and what new features would make Lore Agents "more sexy and
better." The ask was explicit about form — come back with a prioritized WHAT TO IMPROVE
list, not an exhaustive essay.

The session opened by loading prior review dispositions (`architecture-review-
dispositions.md`, `framework-improvements-backlog.md`) so settled questions — DF living
inside `lr`, team-shared/multi-author lore as deliberate design — wouldn't be
re-litigated. From there the review ran a mechanical consistency sweep across the real
`lore-framework/` tree: version stamps and all four plugin manifests (clean at v27),
Cursor skill-tree parity (30/30, in sync), and a from-scratch Python script checking every
`docs/*.md` reference in skills and docs, plus every topic-style reference inside the
lore-architect's own 147-topic lore graph, against files that actually exist on disk. The
plugin-tree doc references came back clean; the lore-graph sweep did not — it surfaced 14
unresolved references, at least 8 of them genuine rot from renames and deletions
(`contributions-feature.md`, `workspace-sync.md`, DF-rename leftovers, and others). That
finding mattered beyond being a bug: `/lr:check` #9–10 promise exactly this coverage via
LLM read-through, so the gap was empirical evidence that a mechanical check degrades at
this scale in a way a deterministic script doesn't. The review closed with a tiered list —
verified inconsistencies fixable now, known backlog gaps promoted with fresh evidence, and
new feature directions (ambient recall, a `/lr:note` micro-capture skill, a lore MCP
server, scheduled autonomous maintenance) — plus a recommended shipping sequence.

The user's follow-up reframed the deliverable: rather than treating the list as a one-off
review output, they asked that this *type* of list always exist going forward, as the
framework's continuous-improvement instrument. That turned the review into
`workdir/what-to-improve.md` — a standing, ranked action list distinct from the unranked
`framework-improvements-backlog.md`, carrying its own refresh protocol (reread at every
framework session, full re-rank at each architecture review, delete-don't-mark on ship),
linked from `lore-context.md` so it survives across boots.

The user then switched models to Sonnet 5 and triggered `/lr:finalize`. Reflection ran
inline against the session and produced four topics: a role-update instructing that the
standing-list practice be folded into `role.md`'s named curation disciplines, the practice
itself as a new foundational-principle topic, the deterministic-sweep-vs-LLM-check lesson,
and a live environmental discovery — on case-insensitive macOS APFS, Claude Code's
`CLAUDE.md` memory auto-injection collided with the lowercase `docs/engines/claude.md`
engine-profile file, silently duplicating context. Merge ran as a single subagent
(host-only session, no guests attached) booted fresh as lore-architect: it created three
lore topics, folded the sweep lesson into the existing `consistency-checks.md` and
`claude-engine-capabilities.md` topics rather than over-atomizing, added a reciprocal
cross-reference on the sibling `consistency-sweep-read-not-just-grep.md` topic, added the
new curation-discipline bullet to `role.md`, and restructured `lore-context.md`'s backlog
section to point at the new practice topic instead of restating it inline.

Everything from reflect and merge landed uncommitted on disk, per the finalize contract —
phase 4 (this same invocation) covers the commit and push. One pre-existing wrinkle
surfaced during merge: `lore-repo.md` at the repo root was already dirty with an
uncommitted version stamp (26→27) and `lore-context.md`'s backlog heading had already been
touched, both leftover from an earlier session rather than this one — left untouched, and
notable because that root-level file sits outside the `git add agents/` scope the finalize
commit uses, so it will not ride along in this session's commit despite being real,
already-completed work waiting to be pushed. Merge also flagged one unrelated pre-existing
drift, left alone as out of scope: `lore-context.md`'s "Versioning & Migration" section
still said "v26 is shipped" in one place while "Current State" correctly said v27
elsewhere.

Next steps sit in `workdir/what-to-improve.md`'s recommended sequencing: an A-tier
"hygiene ship" first (script-back the mechanical subset of `/lr:check`, clean the rotten
lore references, fix `sync-cursor-skills`' fake `--check` mode, close the v27 partial
test-gate follow-up), then a B-tier quality ship (post-merge diff verification, staleness
surfacing at recall), then a C-tier UX ship (ambient recall, `/lr:note`). The orphaned
`lore-repo.md` version stamp is also worth a deliberate follow-up commit outside this
finalize run.
