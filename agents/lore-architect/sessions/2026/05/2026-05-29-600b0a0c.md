# Shipped v13: auto-pull at boot/attach/pre-merge with /lr:pull-lore skill

The session opened with the user identifying a freshness gap in the team-shared agent repo model: a teammate (or a parallel session) could push lore changes between the host's last boot and a finalize-merge in the current session, and the merge would integrate fresh reflections into stale lore — re-introducing decisions the team had already revised. The user asked for an automatic refresh at boot, a manual force-refresh for mid-session use, and a dedicated refresh right before merging reflections into lore. They also asked for three-perspective parallel review with iterative cycles, prioritizing quality, with summary-and-confirm before push.

The architect designed the three integration points around a single shared procedure to avoid copy-paste drift. A new `docs/auto-pull.md` became the canonical per-repo refresh: skip if not a git repo or no `origin`, run `GIT_TERMINAL_PROMPT=0 git -C <repo> pull --ff-only` with a 60-second timeout, report per-call-site verbosity (boot/attach/merge quiet on no-op; user-invoked verbose). Boot got a new step 2 between discovery and version-check, with the rationale articulated explicitly (pull first so version-check sees the freshest `lore-repo.md` stamp). Attach got a new step 2 before version reconcile. Process-merge got a new step 0 — defense-in-depth atop the boot pull, making the freshness contract visible at the moment when stale lore is most damaging. The new `/lr:pull` skill was added at `skills/pull/` plus `docs/pull.md`, iterating auto-pull across active agents and re-reading their `role.md`/`lore-context.md` so the host's working context picks up the pulled changes (the discoverable surprise — git pull alone doesn't refresh memory).

The first plot twist was an early dirty-tree safety gate copied reflexively from `version-check.md`. The architect caught it during the first design pass: `version-check.md` writes to `lore-repo.md`, so a dirty gate is correct there; `git pull --ff-only` is non-writing in spirit (advances `HEAD`, refuses cleanly if it would clobber edits, otherwise leaves the working tree untouched). Gating on dirty would defeat the most important invocation site — pre-merge auto-pull, where `reflections/` from phase 1 is intentionally uncommitted. The gate was removed; the doc explicitly explains the write-vs-read distinction. Round 2 review verified this live by constructing two scenarios in `/tmp` and confirming `git pull --ff-only` preserves edits in both the conflict and no-conflict cases.

Three parallel reviewers ran round 1 with mutually-exclusive lenses: correctness with filesystem verification, framework-architectural-consistency reading the architect's `lore-context.md` as baseline, and UX/discoverability/newcomer. The UX reviewer returned a complete 19-finding report. Two of three (correctness and architecture) timed out after 600s with stream-watchdog stalls, but each surfaced a partial finding before stalling — the correctness reviewer caught a stale "skip to Step 3" reference in `attach.md` (left over from the renumbering when auto-pull became step 2), and the architecture reviewer caught the boot-ordering rationale gap. Round 2 ran a single focused reviewer with full diff in scope, verified all 14 round-1 fixes landed correctly, ran live `git pull --ff-only` tests against constructed scenarios, and surfaced two additional must-fix issues: a `pull.md` self-contradiction with `auto-pull.md` over a non-existent "skipped — uncommitted changes" outcome, and a stale `currently 12` line in `lore-context.md` § Versioning while another section already said `VERSION=13`. Both fixed. The session converged on ship-ready in two review rounds.

The second plot twist came after the architect declared the design done: the user said `/lr:pull` was a vague name that could be confused with a working-repo pull. The architect offered three rename options (`/lr:lore-pull`, `/lr:pull-lore`, `/lr:refresh-lore`) and the user chose `/lr:pull-lore`. The rename was a clean diff: directory `skills/pull/` → `skills/pull-lore/`, doc `docs/pull.md` → `docs/pull-lore.md`, all references updated across 8 framework files plus 3 lore files, with grep verification confirming no stale references except the legitimate historical `/lr:pull-domain` (retired in v11) and `/lr:auto-pull` (the explicit "no such skill exists" disclaimer in `auto-pull.md`'s audience banner).

The ship landed as one v13 commit to `lore-framework` (commit `428f5d9`, pushed to `github.com/zroslaw/lore-framework`): VERSION 12→13, three new files (`auto-pull.md`, `pull-lore.md`, `release-notes/13.md`, `skills/pull-lore/`), and six modified files (`agent-boot.md`, `attach.md`, `process-merge.md`, `recall.md`, `workspace-sync.md`, plus the existing version cross-references). Release notes carry the Clear Plugin Cache footer hoisted near top per `cache-clear-footer-convention.md`, with the disambiguation phrasing ("Repo migration required: no — but a cache-clear is required") and the targeted-before-broader cache wipe ordering. The lore-architect's lore was extended with three new generalizable topics distilled from this session — `freshness-contracts-at-session-boundaries.md` names a foundational principle that crosses session-context boundaries (boot/attach/pre-merge), `shared-procedure-doc-pattern.md` names the auto-pull.md/version-check.md factoring as a reusable framework principle, and `dirty-tree-gates-write-vs-read-distinction.md` names the gate-writes-not-reads safety discipline. `parallel-reviewer-fanout-pattern.md` was updated with the v13 graceful-degradation refinement (treat partial returns from stalled reviewers as additive evidence; round 2 with single focused reviewer + filesystem access is a distinct job from round 1's per-lens fan-out).

Open threads are limited. Two follow-up items were flagged but legitimately out of v13 scope: `conventions.md` § Migration / Release-Note Authoring still says "near the end (just before See Also)" for the Clear Plugin Cache footer placement, but v12 and v13's actual practice is "near the top (after Summary)" — a future session should reconcile the convention text to match practice. A "pull keeps failing" ailment in the `/lr:doctor` catalog is correctly deferred per the catalog's accrete-from-real-cases rule. The framework's worktree convention interaction is now explicitly documented in both `auto-pull.md` § Worktrees and the v13 release-notes Impact section, closing what was a discoverability gap surfaced by the UX reviewer.
