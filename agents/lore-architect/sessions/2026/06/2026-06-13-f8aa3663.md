# Reviewed framework architecture and shipped the lore-context shape discipline (v17)

The session opened with the user asking for a thorough, unhurried architecture review of the Lore Framework — was it optimal, what could improve, which decisions might be revised — with depth explicitly licensed. The architect grounded the review in the real implementation rather than the compressed lore-context: reading the plugin tree, the core design-rationale topics, the README, conventions, the finalize/merge lifecycle, and the DF module. It produced six ranked findings, leading with the observation that the framework's accreted machinery had outgrown its knowledge-persistence core, and that `lore-context.md` was the structure most likely to strain at scale.

The user triaged the findings executive-style in a single pass: one was actioned immediately, three were deferred to the backlog as a combined "simplification / subtraction" item (migration machinery built past its three real migrations, the cross-agent mechanism count, and the absence of a garbage-collector for named principles), and two — keeping the DF module inside the `lr` plugin, and the team-shared / multi-author framing — were closed as deliberate current decisions rather than defects, with a DF-to-sibling-plugin split noted as a future option.

The actioned finding became the session's main deliverable. The architect had initially framed `lore-context.md` as a "hand-maintained denormalized index," but the user corrected that framing: lore-context is maintained by the merge subagent at finalization, not by hand, and its intended design (compacted working knowledge plus references to summary topics) has no denormalization problem. The real issue was drift — the actual file had accreted a full enumeration of every topic and per-version "vN did X" narrative, against its own design intent. That reframing turned the fix from de-duplication into a shape discipline.

The architect added that shape discipline to the framework: merge Step 4 (`process-merge.md`) and the `conventions.md` Lore Context section now direct `lore-context.md` to be a compacted, present-tense working-knowledge entry point that references summary topics rather than enumerating detail topics, with shape — not the 50K size ceiling — as the governing constraint. As the first application, the architect reshaped its own `lore-context.md` from roughly 6,400 to 1,400 words, dropping the topic index and the version history while preserving navigability (topic discovery has always been the on-demand `lore/` scan, so nothing was orphaned). Two backlog items were captured alongside: the deferred simplification pass, and a new "lore housekeeping / consolidation sleep-pass" — a periodic deep-restructuring job that is the active mechanism for the no-garbage-collector gap and the consolidation sibling of the per-session shape discipline.

The change was then shipped as framework **v17**: a release-notes-only, cache-affecting version. The cascade ran in full — VERSION to 17, `release-notes/17.md` with the Clear Plugin Cache footer hoisted near the top, both plugin manifests to 1.17.0, the `versioning-release-types.md` history backfilled, the repo stamped, and a new `lore-context-shape-discipline.md` topic promoted. After validating version/manifest agreement and showing the changeset, the user approved pushing the framework repo only; commit `ff023bd` landed on `zroslaw/lore-framework`. The architect's own lore repo (`lore-framework-dev`) was finalized separately at the end of the session through this reflect / merge / summarize / commit flow.

One reconciliation worth recording: the user surfaced a half-memory that boot used to "read everything." Boot has only ever loaded `role.md` plus `lore-context.md` — the weight came from lore-context having grown comprehensive, which is exactly what the shape discipline now resists. Next steps are deferred but scoped: the simplification / subtraction pass, and the design of the housekeeping sleep-pass when it becomes active. The DF-module-as-sibling-plugin split remains a deliberate future option, not a problem to fix.
