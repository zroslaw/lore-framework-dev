# Designed and shipped AIQA/ULA — first in-plugin lr-dev BETA module

The session opened with two threads from the user. First, a working-style request: adopt a "soft skill" — follow the user's thinking direction without racing ahead or overtaking it, offering only small suggestions — and, separately, treat "soft skills" (guidance carried with the agent, easily updated) versus "hard skills" (shipped in deliberately-installed repos) as a concept worth making an explicit framework entity later. The soft skill was named **follow-me mode** (opt-in, off by default), captured as a lore topic, surfaced in `lore-context.md` for future boots, and backlogged as a framework concept. Follow-me mode stayed active for the rest of the session.

The main work was designing **AIQA** (AI-based Quality Assurance), the first concrete materialization of the lr-dev SDLC direction beyond workdir drafts. AIQA is organized by testing *level*; the level built was **ULA — Unit-Level Analysis**, specifically the **single-file pass**: split a file into units, then per unit run one agent through three ordered steps — find bugs (A), generate clean-room test scenarios excluding bug behavior (B), and two-directional gap analysis against existing tests (C). The design was built up collaboratively and incrementally: artifact set, YAML schemas, the closed `coverage-intent` kind enum, the matching rule, the mismatch-as-quality-signal framing, and a deliberate single-source split (structure in `schemas/`, semantics in `prompts/`, an index doc pointing at both). Decisions to drop SHAs/line ranges and the file-level manifest (the directory tree is the record) kept it lean.

The pieces were packaged as two BETA skills — `/lr:dev-aiqa-repo-init` and `/lr:dev-ula-file` — under a new `dev/` module subtree in the plugin (`dev/aiqa/`), using the `dev-` prefix (required, since `lr:init` already exists) and a skill-doc-pattern variant where thin `SKILL.md` pointers reference docs inside the module subtree rather than the global `docs/`.

A plot twist clarified the runtime story: the dynamic Workflow primitive that would *run* `ula-file-pass.js` is not present in every session — verified by checking the deferred-tool list and `ToolSearch`. So the end-to-end test was run by **simulating** the workflow with Sonnet subagents (one splitter, then a unit-pass agent), which the prompts/schemas were designed to support. The test on `My-Turbo-Boost-Switcher`'s `SystemCommands.m::readCurrentCpuFreq` produced four bugs (three verified real against the source) and nine scenarios, with bug/scenario separation holding.

Two rounds of `/lr:code-review` (run inline to respect the cost ceiling) found nine then four findings. The first round caught real correctness bugs in the freshly-authored workflow — index-based result/unit zipping, silent unit drops, unspecified prompt→key mapping embedding `undefined`, over-strict `coverage-intent` failing whole units. All were fixed (match-by-artifact-slug, named dropped units, fail-loud key guard, relaxed schema, agent self-reads file, honest clean-room preamble), and the re-review confirmed the fixes held, leaving only lower-severity residuals (notably unit-id uniqueness assumed but unenforced).

Where it landed: the AIQA module was committed locally to `lore-framework` as `2f1e788` (18 files) — **not pushed**, per the user's instruction, with the VERSION→16 / manifest→`1.16.0` ship deliberately deferred while the module is BETA and iterating. This finalization captured the session into lore. Next steps for future sessions: fix the unit-id uniqueness gap, decide whether to split A/B/C into separate agents, the broader `/lr:dev-aiqa-analyze` (repo understanding + prioritized file list), the formal v16 ship, and elaborating "soft skills" into an explicit framework entity.
